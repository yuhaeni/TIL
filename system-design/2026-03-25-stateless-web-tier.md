# 무상태(Stateless) 웹 계층

> 날짜: 2026-03-25

## 내용

웹 계층을 수평적으로 확장하려면 상태 정보(사용자 세션 등)를 웹 계층에서 제거해야 한다.
상태 정보는 관계형 DB나 NoSQL 같은 지속성 저장소에 보관하고, 필요할 때만 가져오도록 한다.

### 1. 상태 정보 의존적인 아키텍처

클라이언트 정보, 즉 상태를 유지하여 요청들 사이에 공유되도록 하는 방식이다.

예를 들어, 서버 1, 서버 2, 서버 3이 있고 서버 1에만 사용자 A의 세션 정보가 저장된 경우,
사용자 A를 인증하기 위한 HTTP 요청은 항상 서버 1로 전송되어야 한다.
만약 사용자 A의 HTTP 요청이 서버 2로 전송되면, 서버 2에는 사용자 A에 대한 세션 정보가 없기 때문에 인증에 실패한다.

→ 같은 클라이언트로부터의 요청은 항상 같은 서버로 전송되어야 한다.

이를 위해 로드밸런서는 **고정 세션(Sticky Session)** 이라는 기능을 제공한다.
로드밸런서가 "사용자 A → 서버 1", "사용자 B → 서버 3"처럼
사용자 정보가 어떤 서버에 저장되어 있는지 매핑 정보를 계속 기억하고 관리하는 것을 의미한다.

고정 세션 없이 그냥 골고루 나눠주는 상황에 비해,
고정 세션 기능이 켜진 경우 사용자가 수천, 수만 명으로 늘어나면 로드밸런서 입장에서 부담을 받는다.

### 2. 무상태 아키텍처

사용자 정보는 관계형 DB나 NoSQL, 또는 Redis 같은 캐시 시스템에 저장되어 있기 때문에,
사용자 A의 HTTP 요청이 서버 1로 전송되든 서버 2로 전송되든 상관이 없다.

무상태 웹 계층으로 만들면 로드밸런서는 고정 세션 없이 그냥 골고루 나눠주기만 하면 되고,
세션 정보는 별도 저장소가 담당하니까 훨씬 단순해진다.

**상태 의존 → 고정 세션 필요 → 로드밸런서 부담 → 그래서 무상태로!**
