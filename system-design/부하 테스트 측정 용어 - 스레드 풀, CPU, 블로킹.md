# 부하 테스트 측정 용어 — Tomcat busy·current·max, pcpu·scpu, 블로킹 vs 논블로킹

> 날짜: 2026-07-06

## 내용

```
14:44:20 busy=200.0 current=200.0 max=200.0 pcpu=0.3225 scpu=0.9963
```

### busy / current / max — Tomcat 스레드 풀 지표 3종

톰캣 스레드 풀의 상태를 나타내는 세 숫자다. 식당 웨이터에 비유하면 이해가 쉽다.

| 지표 | 의미 | 웨이터 비유 |
|---|---|---|
| `max` | 스레드 풀 최대치 (이 서버는 200) | 고용 가능한 웨이터 최대 인원 |
| `current` | 지금 살아있는(생성된) 스레드 수 | 지금 출근해 있는 웨이터 (바쁜 + 대기) |
| `busy` | 지금 요청에 배정되어 붙잡힌 스레드 수 | 손님 테이블을 맡아 일하는 웨이터 |

- `current - busy` = **놀고 있는 여유 스레드** (새 요청을 기다리는, 아직 투입 가능한 인력).
  예: `busy=126 current=129` → 대기 중(여유) 스레드 3개.
- **헷갈린 포인트:** `current - busy`(여유 스레드)는 "요청이 밀려서 쌓인 것"이 **아니다.** 오히려 반대로 아직 아무 요청도 안 맡은 **여유분**이다.
- `busy=200 current=200` = 여유 스레드 0개. 이 순간부터 새 요청은 스레드를 못 얻어 **접속 대기 큐(accept queue)에 줄을 선다** → 응답시간(p95)이 늘어난다.

> **면접 예상 질문:** 톰캣 스레드 풀에서 `busy`, `current`, `max`는 각각 무엇을 세는 값이며, `current - busy`가 커진 상태와 `busy == max`가 된 상태는 서버 부하 관점에서 어떻게 다르게 해석해야 하나요?

### busy에 포함되는 스레드 — "두 종류의 대기" 구분

`busy`는 CPU를 실제로 쓰는 스레드뿐 아니라, **요청을 맡은 채 I/O 응답을 기다리며 블로킹된 스레드**까지 합친 값이다.

- **여유 스레드 (busy에 미포함):** 새 요청이 오길 기다리며 놀고 있는 스레드 → `current - busy`
- **I/O 대기 스레드 (busy에 포함):** 요청은 맡았지만 DB 응답·외부 API 왕복을 기다리며 붙잡혀 멈춘 스레드

즉 로그의 `busy=200`은 대부분 후자(요청 맡고 I/O 기다리는 중) 상태라, CPU를 안 쓰면서도 스레드 풀을 다 먹은 것이다.

> **면접 예상 질문:** 스레드 풀의 `busy` 카운트에는 CPU 연산 중인 스레드와 I/O를 기다리며 블로킹된 스레드가 모두 포함됩니다. 이 둘을 구분하지 못하면 어떤 잘못된 튜닝 판단(예: 스레드 수 증설 vs 다른 처방)을 내릴 수 있나요?

### pcpu / scpu — 프로세스 CPU vs 시스템 CPU

CPU 사용률을 두 범위로 나눠 찍은 값이다.

| 지표 | 뜻 | 의미 |
|---|---|---|
| `pcpu` | **p**rocess CPU | 우리 서버(JVM) **프로세스 하나**가 실제로 쓴 CPU 비율 |
| `scpu` | **s**ystem CPU | **머신 전체**가 쓴 CPU 비율 |

- 이번 측정은 서버·부하도구(k6)·mock을 **한 컴퓨터**에서 같이 돌려서 `scpu`가 ~99%로 합산·왜곡됨 → **판단은 `pcpu` 기준으로만** 한다.
- `pcpu`가 낮다 = JVM이 계산을 안 하고 그냥 **기다리는 중**이라는 신호.

> **면접 예상 질문:** 부하 테스트에서 시스템 CPU(scpu)는 100%에 가까운데 애플리케이션 프로세스 CPU(pcpu)는 낮게 나옵니다. 어떤 원인들을 의심할 수 있으며, 왜 병목 판단을 시스템 CPU가 아닌 프로세스 CPU 기준으로 해야 하나요?

### busy=200 & pcpu=30% — 스레드 풀 고갈의 정체

로그의 핵심 현상: 스레드는 `busy=200`으로 꽉 찼는데 `pcpu`는 0.3(=30%)에 불과하다.

- CPU가 **부족해서** 스레드가 꽉 찬 게 아니다 (CPU는 30%, 여유 만빵).
- 요청당 "DB 저장 2번 + mock 왕복 + 콜백"이 있어, 스레드가 **외부 I/O 응답을 기다리며 블로킹된 채** 붙잡혀 있어서 스레드 풀이 고갈된 것이다.
- 한 줄 요약: **"요청을 처리 중이지만 외부 I/O 응답을 기다리느라 블로킹된 스레드가 200개라, CPU는 30%인데도 스레드는 꽉 찼다."**

> **면접 예상 질문:** "스레드 풀은 포화(busy=max)인데 CPU 사용률은 낮다"는 상황을 보면, 병목이 CPU가 아니라 무엇이라고 진단할 수 있나요? 이때 스레드 수를 늘리는 처방과 I/O 방식을 바꾸는 처방 중 무엇이 더 근본적인가요?

### 블로킹 / 논블로킹 vs 동기 / 비동기 — 헷갈리기 쉬운 두 축

가장 헷갈렸던 지점. 서로 **다른 축**이다.

| 축 | 기준 질문 | 구분 |
|---|---|---|
| **블로킹 / 논블로킹** | 기다리는 동안 **스레드가 붙잡혀 있나?** | 블로킹: 스레드가 묶여서 멈춤 / 논블로킹: 스레드가 풀려나 다른 일 함 |
| **동기 / 비동기** | 결과를 **누가 챙기나?** | 동기: 내가 직접 확인해서 받음 / 비동기: 완료되면 통보(콜백)받음 |

- **헷갈린 포인트:** "직접 확인하나 vs 누가 알려주나"는 **동기/비동기** 축이다. 블로킹/논블로킹과 섞으면 안 된다.
- 로그의 `busy=200 / pcpu=30%`는 "스레드가 I/O 기다리며 **붙잡혀** 있다" → **블로킹/논블로킹** 축의 이야기.
- 웨이터 비유:
  - **블로킹:** 주문 받고 주방 창구 앞에 **가만히 서서** 음식 나올 때까지 기다림 → 테이블 하나당 웨이터 하나가 묶임.
  - **논블로킹:** 주문서를 주방에 걸어두고 **바로 돌아서서** 다른 테이블을 받음. 음식이 되면 "딩동!" 알림에 손 빈 웨이터가 내줌 → 소수의 웨이터로 많은 테이블 감당.
- 그래서 외부(Nebula)가 느려질수록 블로킹(RestTemplate)은 스레드가 더 빨리 고갈되고, 논블로킹(WebClient)은 기다리는 시간에 스레드를 붙잡지 않아 덜 부족하다.

> **면접 예상 질문:** 블로킹/논블로킹과 동기/비동기는 서로 다른 축입니다. 두 축의 기준을 각각 한 문장으로 설명하고, "논블로킹이지만 동기" 또는 "블로킹이지만 비동기" 같은 조합이 성립하는지 예를 들어 설명해 보세요.

## 학습 정리

- `max`(최대치) / `current`(출근한 스레드) / `busy`(요청 맡아 일하는 스레드), 그리고 `current - busy` = 여유 스레드(밀린 게 아니라 노는 것).
- `busy`에는 CPU 연산 중 스레드 + I/O 기다리며 블로킹된 스레드가 **둘 다** 포함된다.
- `pcpu`=프로세스(JVM) CPU, `scpu`=시스템(머신 전체) CPU. 한 컴퓨터에서 부하도구를 같이 돌리면 scpu는 왜곡되니 pcpu로 판단.
- `busy=max`인데 `pcpu`가 낮으면 = CPU 부족이 아니라 **블로킹 I/O 대기로 인한 스레드 풀 고갈**. → RestTemplate(블로킹) vs WebClient(논블로킹) 비교의 근거.
- 블로킹/논블로킹(스레드가 붙잡히나) vs 동기/비동기(결과를 직접 챙기나 통보받나)는 **다른 축**이다.
