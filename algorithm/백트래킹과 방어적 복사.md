# 백트래킹과 방어적 복사

> 날짜: 2026-04-28

## 내용

### 백트래킹(Backtracking) — DFS + 상태 복원

**백트래킹 = DFS(깊이 우선 탐색) + 상태 복원(undo).** 한 길로 깊이 들어간 뒤, **원래 상태로 되돌려놓고** 다른 길을 탐색하는 패턴이다. 카카오 2018 "불량 사용자" 문제처럼 **모든 조합/순열 완전탐색** 류에 자주 등장한다.

**언제 쓰는가:**
- 모든 조합/순열을 탐색
- 유효한 해를 찾을 때까지 시도하고 안 되면 되돌아가야 할 때
- N-Queen, 부분집합, 순열, 조합 등 전형적인 완전탐색

**핵심 코드 패턴 — "선택 → 재귀 → 복원" 3박자:**
```java
void dfs(int depth, State state) {
    if (depth == 목표) {
        results.add(snapshot(state));   // ⚠️ 반드시 복사!
        return;
    }
    for (Choice c : 후보들) {
        if (불가능) continue;
        state.add(c);            // ① 선택 (전진)
        dfs(depth + 1, state);   // ② 재귀
        state.remove(c);         // ③ 복원 (백트래킹) ⭐
    }
}
```

**시간 복잡도:** 가지치기 없을 때 최악 **O(분기수^깊이)**. 입력 제약(`user_id` ≤ 8)을 먼저 보고 **복잡도 예산**을 정하는 습관이 실무에서도 중요.

### 자바 객체 참조 vs 방어적 복사

자바는 객체를 **참조로 다룬다**. 상태가 변하는 객체를 컬렉션에 저장하면 **나중에 그 객체가 변경될 때 컬렉션 안 데이터도 같이 바뀐다**.

```java
Set<String> picked = new HashSet<>();
picked.add("a");
results.add(picked);    // picked의 "참조"가 들어감
picked.clear();         // → results 안의 그것도 비워짐 ❌
```

**방어적 복사(Defensive Copy) — 그 순간의 스냅샷 저장:**
```java
results.add(new HashSet<>(picked));   // ✅ 복사본 (스냅샷)
results.add(picked);                  // ❌ 참조 (이후 변경에 오염)
```

**실무 예시 — 불변 객체 설계:**
```java
public class Order {
    private final List<Item> items;

    public Order(List<Item> items) {
        this.items = new ArrayList<>(items);              // ✅ 입력 복사
    }
    public List<Item> getItems() {
        return Collections.unmodifiableList(items);       // ✅ 출력 보호
    }
}
```

**기억할 도구:**
- `List.copyOf()`, `Set.copyOf()` (Java 10+) — 불변 복사본
- `Collections.unmodifiableList()` — 읽기 전용 래퍼
- **Record는 컬렉션 필드 참조를 그대로 공유** → 방어적 복사 필요

### Set vs List — 자료구조 선택

| 특성 | List | Set (HashSet) |
|---|---|---|
| 순서 | 유지 | 무관 |
| 중복 | 허용 | 비허용 |
| `contains` | O(N) | **O(1)** |
| 동등성 | 순서 영향 | 순서 무관 |

**불량 사용자 문제 자료구조 선택:**
```java
List<List<String>> matching;   // index 접근 + 작은 후보 목록
Set<String> picked;            // contains O(1) + 한 user 한 번만 사용
Set<Set<String>> results;      // 집합끼리 중복 제거 (순서 무관)
```

**`Set<Set<String>>`이 자동으로 중복 제거되는 이유:**
- `HashSet`은 `equals()` + `hashCode()` 로 중복 판단
- `Set.equals()`는 **원소가 같으면 true** (순서 무관)
- → `{a,b}`와 `{b,a}`가 자동으로 같은 것으로 처리됨

**확장:**
- `LinkedHashSet` — 삽입 순서 보존
- `TreeSet` — 키 정렬 (Red-Black Tree)
- `equals/hashCode` 계약 — 재정의 시 **둘 다** 함께

> **면접 예상 질문:** `Set<Set<String>>`이 어떻게 순서 무관 중복 제거를 보장하는가? `equals`와 `hashCode`를 함께 재정의해야 하는 이유는?

### 재귀와 호출 스택 — "다음 줄로 돌아온다"

```java
picked.add(candidate);           // ①
dfs(depth + 1, picked);          // ② 자식 호출이 끝날 때까지 대기
picked.remove(candidate);        // ③ ②가 return 한 뒤 실행
```

함수 호출은 **콜 스택**에 쌓이고, 호출된 함수가 `return`하면 **호출한 곳의 다음 줄부터 재개**한다. 재귀도 동일.

```
dfs(0) → add → dfs(1) → add → dfs(2) → 저장 → return
                                                     ↓
dfs(0) ← remove ← dfs(1) ← remove ← (재귀 호출 다음 줄로 복귀)
```

**실무 연결:**
- **`StackOverflowError`** — 재귀 깊이 한계 (보통 1만~10만)
- 자바는 **TCO(Tail Call Optimization) 미지원**
- 깊은 재귀 → 반복문 + 명시적 `Stack` 자료구조로 변환
- 안전선: 재귀 깊이 **~10,000 이내**

> **면접 예상 질문:** 재귀가 끝나면 어떤 코드부터 다시 실행되는가? `StackOverflowError`를 피하려면 어떤 패턴을 쓰는가?

### 시간 복잡도 예산 (실무 감각)

"이게 빠를까?" 보다 **"이 데이터 양에서 충분히 빠른가?"** 가 실전 사고법.

| N의 크기 | 허용 복잡도 |
|---|---|
| N ≤ 10 | O(N!) ≈ O(2^N) — 백트래킹/완전탐색 |
| N ≤ 20 | O(2^N) — 비트마스크 DP |
| N ≤ 100 | O(N³) — 플로이드, 작은 DP |
| N ≤ 1,000 | O(N²) — 일반적 DP |
| N ≤ 100,000 | O(N log N) — 정렬, 이분탐색 |
| N ≤ 10,000,000 | O(N) — 선형 |

**원칙:**
- **조기 최적화 금지** (Premature Optimization Is the Root of All Evil — Knuth)
- 측정 → 병목 식별 → 최적화 (프로파일러 기반)

### 백트래킹 흔한 실수 정리

| 실수 | 증상 | 해결 |
|---|---|---|
| `remove` 누락 | 다음 가지 오염 | 재귀 직후 반드시 `remove` |
| `remove`를 재귀 전에 둠 | 빈 상태로 재귀 → 빈 결과만 | `add → 재귀 → remove` 순서 |
| `results.add(picked)` 참조 저장 | 모든 결과가 같은 객체로 수렴 | `new HashSet<>(picked)` |
| `Set<List>` 사용 | 순서 다른 같은 조합이 다른 것으로 처리 | `Set<Set>` 사용 |
| `int[]` Comparator 정렬 | 객체 배열 아님 | `Integer[]` 변환 또는 박싱 |
| `char == "*"` | 컴파일 에러 | 작은따옴표 `'*'` |

**대안 풀이 (꼬리 질문 대비):**
- **비트마스크 DP** — `user_id` ≤ 8이라 비트로 표현 가능, 빠르지만 코드 난해
- **`Set<String>` (정렬 후 join)** — `"a,b,c"` 문자열 키로 메모리 효율 ↑, 가독성 ↓
- **입력이 100배 커지면** — 백트래킹은 폭발 → 비트마스크 DP, 실무라면 분산/캐싱/근사 검토

> **면접 예상 질문:** 백트래킹에서 `remove` 위치를 잘못 두면 어떤 결과가 나오는가? `Set<List>` 대신 `Set<Set>`을 쓰는 이유는?

## 학습 정리

- **백트래킹 = DFS + 상태 복원**, 핵심은 "선택 → 재귀 → **복원**" 3박자
- 자바는 객체를 **참조**로 다루므로, 변하는 상태를 컬렉션에 저장할 땐 **방어적 복사**(`new HashSet<>(picked)`) 필수
- 불변 객체 설계 시 **입력 복사 + 출력 보호**(`unmodifiableList`) 둘 다 필요 — Effective Java Item 50
- `Set.equals()`는 **원소 동일성**(순서 무관)으로 동작 → `Set<Set<String>>`이 자동 중복 제거
- 재귀는 콜 스택에 쌓이고 **호출 다음 줄로 복귀** — 깊이 한계는 보통 1만~10만, 자바는 TCO 미지원
- 입력 크기로 **복잡도 예산**부터 정하는 것이 실전 사고법 (조기 최적화 금지)
- 백트래킹 5대 실수: `remove` 누락 / 순서 뒤바뀜 / 참조 저장 / `Set<List>` 오용 / 박싱 누락
