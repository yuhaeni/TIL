# 자연 정렬과 상태 머신 문자열 파싱

> 날짜: 2026-04-24

## 내용

### 자연 정렬(Natural Sort)

단순 문자열 정렬은 `img10 < img2`로 판정한다. `'1' < '2'`가 먼저 결정되기 때문. 하지만 사람의 직관은 `img2 < img10`이다.

**자연 정렬**은 문자열 내부의 숫자를 **숫자 값**으로 비교하는 정렬 방식이다.

| 정렬 방식 | `img2` vs `img10` | `F-15 9`.txt vs `F-5 16`.txt |
|---|---|---|
| 사전순(렉시코그래픽) | `img10 < img2` ❌ | F-15 ... 먼저 |
| **자연 정렬** | `img2 < img10` ✅ | 직관에 맞음 |

카카오 2018 블라인드 3차 "파일명 정렬"이 대표 문제. 파일명을 `HEAD + NUMBER + TAIL` 구조로 파싱하고, HEAD는 사전순, NUMBER는 숫자값으로 비교한다.

### 상태 머신(State Machine) 기반 파싱

`HEAD → NUMBER → TAIL` 3단계 파싱은 **상태 머신 패턴**으로 구현한다. 한 문자씩 순회하며 "지금 어느 단계인가"를 `state` 변수로 추적한다.

```
state = 0  → HEAD 수집 중 (숫자 아닌 문자)
state = 1  → NUMBER 수집 중 (숫자)
state = 2  → TAIL 수집 중 (나머지 전부)
```

**왜 상태 머신인가?**
- "숫자 만나면 HEAD 종료" — 단순 if 분기로 가능
- "NUMBER 뒤에 다시 문자 → TAIL 시작, 그 후 숫자가 또 나와도 NUMBER에 섞이면 안 됨" — **이전 상태를 기억**해야 구분 가능

**올바른 조건:**
```java
if (state == 0 && !isDigit(c))      // HEAD에 추가
    head.append(c);
else if (state != 2 && isDigit(c)) { // NUMBER 수집 (0→1 전환 포함)
    state = 1;
    number.append(c);
} else if (state == 1 && !isDigit(c)) { // NUMBER 종료, TAIL 시작
    state = 2;
    tail.append(c);
} else if (state == 2)                // TAIL에 계속 추가
    tail.append(c);
```

**핵심 조건: `state != 2`**
- state==0에서 첫 숫자 → 전환 + 추가
- state==1에서 연속 숫자 → 계속 추가
- state==2에서는 숫자가 나와도 NUMBER에 섞지 않음

**자주 하는 실수:**

| 실수 | 증상 |
|---|---|
| HEAD 조건을 `isLetter()`로만 체크 | `F-15`의 `-`, 공백이 HEAD에 안 들어감 |
| NUMBER 조건에 state 체크 누락 | `foo010bar020`의 TAIL `020`이 NUMBER에 섞임 |
| state==1에서 연속 digit 조건 누락 | `img12`의 NUMBER가 `1`만 됨 (2 누락) |

### Comparator 체이닝 — 다중 기준 정렬

정렬 조건이 여러 개일 때 `Arrays.sort()`를 **여러 번 호출하면 안 된다**. 뒤의 sort가 앞의 결과를 덮어쓴다.

반드시 **하나의 Comparator 안에서 순차적으로** 비교해야 한다.

```java
Arrays.sort(arr, (a, b) -> {
    // 1순위: HEAD (대소문자 무시)
    int r1 = a[0].compareToIgnoreCase(b[0]);
    if (r1 != 0) return r1;

    // 2순위: NUMBER (숫자값)
    int r2 = Integer.compare(Integer.parseInt(a[1]), Integer.parseInt(b[1]));
    if (r2 != 0) return r2;

    // 3순위: 원본 입력 순서
    return Integer.compare(Integer.parseInt(a[3]), Integer.parseInt(b[3]));
});
```

Java 8+에서는 `Comparator.comparing(...).thenComparing(...)` 체이닝도 가능하지만, 코테에서는 람다 내 순차 비교가 더 직관적이다.

> **면접 예상 질문:** 여러 기준으로 정렬할 때 왜 Comparator를 하나로 통합해야 하는가?

### 대소문자 무시 비교와 Integer.parseInt의 앞자리 0 처리

**대소문자 무시:**
```java
"MUZI".compareToIgnoreCase("muzi");  // 0 (같음)
"MuZi".compareToIgnoreCase("MUZI");  // 0
```

**앞자리 0 자동 처리:**
```java
Integer.parseInt("0011");  // 11
Integer.parseInt("012");   // 12
Integer.parseInt("12");    // 12
```

→ `"0011"`, `"012"`, `"12"`를 "같거나 값 기준으로 비교"할 때 그냥 `Integer.parseInt`만 써도 앞자리 0은 자동으로 무시된다.

비교는 **`Integer.compare(x, y)`** 사용 (오버플로우 안전, `x - y`는 음수 오버플로우 위험).

> **면접 예상 질문:** `a - b` 대신 `Integer.compare(a, b)`를 쓰는 이유는?

### 안정 정렬과 원본 인덱스 Tiebreaker

문제 조건: "HEAD와 NUMBER가 같으면 **입력 순서 유지**"

**방법 1: `return 0` (권장 X)**
Java `Arrays.sort(Object[])`는 TimSort(안정 정렬)라 이론상 `return 0`만 해도 순서 유지. **하지만** 전처리 단계에서 배열을 재구성하는 경우 원본 인덱스가 유실될 수 있어 안전하지 않다.

**방법 2: 원본 인덱스 저장 (권장)**
파싱 시 원본 인덱스를 함께 저장하고, 동점일 때 인덱스로 tiebreaker 처리.

```java
String[][] arr = new String[n][4];
for (int i = 0; i < n; i++) {
    // ...파싱 결과...
    arr[i][0] = head;
    arr[i][1] = number;
    arr[i][2] = tail;
    arr[i][3] = String.valueOf(i);  // 원본 인덱스
}

// Comparator 마지막 단계
return Integer.parseInt(a[3]) - Integer.parseInt(b[3]);
```

→ 명시적이고 안전. 코테에서 이 패턴을 반사적으로 쓸 수 있게 훈련.

> **면접 예상 질문:** 안정 정렬이란 무엇인가? `return 0`에 의존하지 않고 원본 인덱스를 저장하는 이유는?

### 2차원 배열 정렬 패턴

객체 클래스를 만들기 번거로운 코테에서는 **`String[][]` 또는 `int[][]`** 로 묶는 게 관용구.

```java
String[][] arr = new String[n][k];
// arr[i] = {필드1, 필드2, 필드3, ...}

Arrays.sort(arr, (a, b) -> a[0].compareTo(b[0]));

// int[][] 예시
int[][] arr2 = new int[n][2];
Arrays.sort(arr2, (a, b) -> a[0] - b[0]);
// 또는 안전하게: Arrays.sort(arr2, Comparator.comparingInt(x -> x[0]));
```

이 패턴은 카카오 문제뿐 아니라 회의실 배정, 구간 스케줄링, 좌표 정렬 등에서 반복적으로 쓰인다.

### 복잡도 감각과 디버깅 흐름

**복잡도 감각:**
- 파일 최대 1,000개 × 파일명 100자 → 이중 반복 100,000회 → 충분
- 정렬 O(N log N) ≈ 10,000회 비교 → 여유
- 코테 기준 **10^6 ~ 10^7 이하면 시간 걱정 불필요**

**디버깅 흐름 (실전 효과적):**
1. **파싱 결과를 먼저 출력**
   ```java
   System.out.println(Arrays.deepToString(arr));
   ```
   → NUMBER가 한 자리만 들어가는 버그를 즉시 포착

2. **엣지 케이스를 손으로 따라가기**
   - `F-15` → HEAD에 `-` 들어가는가?
   - `foo010bar020.zip` → TAIL의 `020`이 NUMBER로 새지 않는가?
   - `img12` → NUMBER에 `12` 전체가 들어가는가?

3. **실패 패턴 읽기**
   - 실패 출력이 `img1`, `img0`처럼 **앞 한 글자만** → 숫자 수집 로직 문제
   - 실패 출력이 뒤섞임 → Comparator 문제

> **면접 예상 질문:** 문자열 파싱 코딩 테스트에서 디버깅을 어떻게 접근하는가?

### 다시 풀 때 체크리스트

- HEAD 조건이 `!isDigit`인가? (`isLetter`는 `-`, 공백, `.` 누락)
- NUMBER 수집 조건에 `state != 2` (또는 `state <= 1`)이 있는가?
- Comparator를 하나로 통합했는가? (여러 번 sort ❌)
- 대소문자 무시 비교는 `compareToIgnoreCase`?
- 앞자리 0은 `Integer.parseInt`가 처리하게 맡겼는가?
- 동점 처리용 원본 인덱스를 저장했는가?
- 숫자 비교는 `Integer.compare` 또는 `Long.compare` (오버플로우 안전)?

## 학습 정리

- **자연 정렬**은 문자열 내부 숫자를 **숫자값**으로 비교하는 정렬 — 사전순과 결과가 다름
- **상태 머신 파싱**은 `state` 변수로 단계 전환 추적 — `state != 2` 조건이 TAIL의 숫자 섞임 방지의 핵심
- HEAD 조건은 `!isDigit`이어야 함 (`isLetter`는 `-`, 공백 누락)
- **Comparator 체이닝**으로 다중 기준 처리 — `sort`를 여러 번 호출하면 안 됨
- `Integer.parseInt`가 **앞자리 0을 자동 처리**, 비교는 오버플로우 안전한 `Integer.compare`
- **원본 인덱스를 tiebreaker로 저장**하는 게 `return 0`보다 안전하고 명시적
- 코테에서는 **`String[][]` / `int[][]`** 로 묶어서 정렬하는 패턴이 국룰
- **10^6 ~ 10^7 이하면 시간 걱정 불필요** — 감 잡기
- 디버깅은 **파싱 결과 출력 → 엣지 케이스 손 추적 → 실패 패턴 분석** 순서
