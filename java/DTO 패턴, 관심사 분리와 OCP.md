# DTO 패턴, 관심사 분리와 OCP

> 날짜: 2026-04-14

## 내용

### DTO(Data Transfer Object) 패턴

DTO는 계층 간 데이터 전달을 담당하는 객체로, Entity와 분리하여 사용한다.

**Entity를 직접 노출할 때의 문제 3가지:**

1. **불필요한 필드 노출**: 외부에 노출하면 안 되는 컬럼까지 함께 노출됨
2. **변경 영향 범위 확대**: Entity 하나를 바꾸면 그것을 사용하는 API 전부가 영향받음
3. **양방향 의존성**: A가 B에 의존하고 B도 A에 의존하면, 어느 한쪽을 수정할 때 다른 쪽도 바꿔야 함. DTO가 Entity에만 의존하면 Entity 변경만으로 충분하지만, 서로 의존하면 둘 다 변경해야 함

### 관심사 분리(SoC, Separation of Concerns)

각 클래스는 하나의 책임만 가져야 한다.

**DTO의 역할:** 데이터를 담는 것  
**파싱/변환 로직의 역할:** 별도 클래스가 담당

예를 들어 `@FixedLengthField` 어노테이션을 사용하는 경우:
- 어노테이션을 **정의(선언)** 하는 것 → DTO의 역할
- 어노테이션을 **읽어서 파싱하는 것** → 별도 파싱 클래스의 역할

DTO 안에 파싱 로직까지 있으면 파싱 방식이 바뀔 때마다 DTO를 직접 수정해야 한다.

**DTO 내부 변환 로직의 실용적 기준:**

| 상황 | 판단 |
|---|---|
| 단순 변환 (필드 복사 수준) | DTO 내부 메서드도 OK |
| 외부 의존성 필요 (Entity + 외부 서비스 + DB 조회 등) | 별도 클래스로 분리 |

### OCP(Open-Closed Principle)

> "확장에는 열려 있고, 수정에는 닫혀 있어야 한다."

새로운 전문파일 타입이 추가될 때:
- 새로운 DTO 파일만 추가하면 됨 (확장에 열려 있음)
- 기존 파싱 클래스 수정 불필요 (수정에 닫혀 있음)

**OCP 위반 예시:** 새 타입 추가 시 기존 분기 코드를 수정해야 하는 경우

```java
// OCP 위반 - 새 타입이 추가될 때마다 여기를 수정해야 함
if (dto instanceof SellerDataBody) { ... }
else if (dto instanceof BuyerDataBody) { ... }
```

**어노테이션 + 리플렉션 기반 파싱이 OCP를 지킬 수 있는 이유:**  
리플렉션으로 `@FixedLengthField`를 읽으면 어떤 DTO가 들어오든 클래스 타입을 몰라도 어노테이션 값만 보고 동적 처리가 가능하기 때문이다.

### EUC-KR 바이트 계산

전문 파일 파싱에서 고정 길이 필드를 다룰 때 인코딩에 따른 바이트 계산이 중요하다.

| 인코딩 | 한글 1글자 | 영문/숫자 | length=20일 때 한글 최대 |
|---|---|---|---|
| EUC-KR | 2바이트 | 1바이트 | 10글자 |
| UTF-8 | 3바이트 | 1바이트 | 6글자 (2바이트 남음) |

금융권 레거시 시스템에서 EUC-KR을 많이 사용하므로, 고정 길이 파싱 시 인코딩을 반드시 확인해야 한다.

> **면접 예상 질문:** EUC-KR과 UTF-8에서 한글의 바이트 수 차이는? 고정 길이 필드 파싱 시 왜 중요한가?

## 학습 정리

- Entity 직접 노출 시 불필요한 필드 노출, 변경 영향 범위 확대, 양방향 의존성 문제 발생
- DTO는 데이터를 담는 역할만, 파싱/변환 로직은 별도 클래스가 담당 (SoC)
- 단순 변환은 DTO 내부 메서드도 허용, 외부 의존성이 필요하면 별도 클래스로 분리
- OCP: instanceof 분기 대신 어노테이션 + 리플렉션으로 동적 처리하면 기존 코드 수정 없이 확장 가능
- EUC-KR은 한글 1글자 = 2바이트, UTF-8은 3바이트
