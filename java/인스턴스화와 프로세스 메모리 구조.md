### 프로세스 메모리 3영역
<img width="700" height="763" alt="image" src="https://github.com/user-attachments/assets/1dc7936b-f254-4797-a7c5-01d7456d68ab" />
<img width="700" height="601" alt="image" src="https://github.com/user-attachments/assets/0f4c9a9c-261c-4ec2-a3ad-563b447c0feb" />


Java 프로그램이 실행될 때 JVM은 메모리를 크게 3영역으로 나눠 관리한다.

```
[ Method Area ] → 클래스 정보 (설계도)
[ Stack ]       → 지역변수, 참조값 (스레드별로 독립)
[ Heap ]        → 실제 객체 (프로세스 전체 공유)
```

- **Method Area**: 클래스 파일을 읽어 클래스 정보(설계도)를 저장하는 영역. 프로그램 실행 전 클래스 로딩 단계에서 채워진다
- **Stack**: 메서드 호출 시 지역변수와 참조값이 저장되는 영역. 스레드마다 독립적으로 존재
- **Heap**: `new` 키워드로 생성된 실제 객체가 저장되는 영역. 프로세스 전체가 공유

> **면접 예상 질문:** `User user = new User();` 실행 시 Stack과 Heap에 각각 무엇이 저장되는가?

### 인스턴스화 전체 흐름

`User user = new User();` 한 줄이 실행될 때 내부적으로 일어나는 과정은 다음과 같다.

| 단계 | 영역 | 설명 |
|---|---|---|
| ① 클래스 로딩 | Method Area | 클래스 파일에서 설계도를 읽어 Method Area에 올림 |
| ② 객체 생성 | Heap | `new User()` 실행 시 실제 객체가 Heap에 생성됨 |
| ③ 참조 저장 | Stack | 참조 변수 `user`에 Heap 객체의 주소가 저장됨 |
| ④ 생성자 호출 | - | `User()` 생성자 실행으로 필드값 초기화 (할당값 없으면 `null`, `0` 등 기본값) |

> **면접 예상 질문:** `User user = new User();` 실행 시 JVM 내부에서 일어나는 전체 흐름을 설명하시오.

### GC(Garbage Collector)의 삭제 기준

GC는 **"최근에 안 쓴 객체"가 아니라 "참조가 없는 객체"** 를 삭제 대상으로 판단한다.

- Stack의 참조 변수가 더 이상 가리키지 않는 Heap 객체 → GC 수거 대상
- 참조가 하나라도 남아 있으면 GC는 해당 객체를 살아있다고 판단하여 수거하지 않음

> **면접 예상 질문:** GC는 어떤 객체를 삭제해도 되는 객체로 판단하는가?

### String 리터럴 vs `new String()`
<img width="700" height="401" alt="image" src="https://github.com/user-attachments/assets/fe5609b5-97d3-4dcf-82e8-97273c338b67" />

Java에서 문자열을 생성하는 방법에 따라 메모리 동작 방식이 달라진다.

- `"홍길동"` (리터럴) → **String Pool**에 저장. 같은 값이면 기존 객체를 공유 (재사용)
- `new String("홍길동")` → **매번 Heap에 새 객체** 생성

이로 인해 `==` 비교 결과가 달라진다.

```java
"홍길동" == "홍길동"                         // true  (같은 String Pool 주소)
new String("홍길동") == new String("홍길동") // false (다른 Heap 주소)
```

`==`는 주소값을 비교하고, `.equals()`는 실제 값을 비교한다. 문자열 내용 비교 시 반드시 `.equals()`를 사용해야 한다.

> **면접 예상 질문:** `"홍길동" == "홍길동"`과 `new String("홍길동") == new String("홍길동")`의 결과가 다른 이유는?

### 멀티스레드 환경에서의 Heap 공유 문제

Heap 영역은 프로세스 전체가 공유하므로, 두 스레드가 같은 Heap 객체를 동시에 수정하면 **데이터 정합성 문제**가 발생한다.

해결 방법: `synchronized` 키워드로 스레드 간 잠금(뮤텍스 락)을 건다. 한 스레드가 객체를 사용 중이면 다른 스레드는 잠금이 해제될 때까지 대기한다.

```java
public synchronized void update(String name) {
    this.name = name;
}
```
