# OOM 실측 리포트 면접 분석 — ExitOnOOM 2-패스, BigDecimal 힙 점유, 누수 vs volume, k6 지표 함정, GC 튜닝

> 날짜: 2026-06-21

256MB 힙(G1)에서 대용량 트래픽을 시뮬레이션해 `OutOfMemoryError: Java heap space`를 **재현**하고,
그 실측 리포트(실행 구성 → 결과 → 힙덤프 → 누수 vs volume → 산출물 → 튜닝 계획)를 면접 수준으로
한 항목씩 분석한 기록이다. 핵심 관통 철학은 **"추측이 아니라 측정으로 판단한다"**.

## 내용

### 고정 힙(`-Xms` = `-Xmx`) — 측정 변수 통제

- `-Xmx`(ma**x**imum)가 **최대 힙**, `-Xms`(**s**tart)가 **초기 힙**이다. (헷갈리기 쉬운 포인트 — 거꾸로 외우기 쉬움)
- 둘을 다르게 주면(`-Xms64m -Xmx256m`) JVM이 작게 시작했다가 부족할 때 힙을 **resize**한다.
  이번 실험은 둘을 `256m`으로 **동일하게 고정**했다.
- 이유 = **변수 통제(control)**. 힙이 동적으로 변하면 OOM 시점·GC 동작에 노이즈가 끼므로,
  천장을 고정해 "정확히 256m에서 터진다"는 재현 가능한 기준선을 만든다. → before/after 튜닝 비교의 공정성.

```
| 힙 | -Xms256m -Xmx256m |
```

### JVM 진단 옵션 — `-Xlog:gc*`, `HeapDumpOnOutOfMemoryError`, `ExitOnOutOfMemoryError`

- JVM 옵션 접두사: `-X`는 안정적 확장 옵션(`-Xms/-Xmx/-Xlog`), `-XX`는 고급 옵션(`+`=켜기, `-`=끄기).
- **`-Xlog:gc*`**: `gc`로 시작하는 모든 GC 로그 출력 → "GC가 언제·몇 번·얼마나 오래 돌았나" 기록(정황 증거).
- **`-XX:+HeapDumpOnOutOfMemoryError`**: OOM이 터지는 **그 순간**의 힙을 `.hprof`로 떠둠.
  OOM은 재현이 어려운 일회성 장애라, 이 사진이 없으면 "범인은 사라지고 현장만" 남는다(사후 부검 불가).
  - **트레이드오프**: 힙 16GB 서버면 덤프도 수십 GB → 디스크 full·다운타임. `-XX:HeapDumpPath`로 경로 지정 필요.
- **`-XX:+ExitOnOutOfMemoryError`**: OOM 첫 발생 시 JVM **즉시 종료**. 좀비로 limping하지 않음.
  - 운영 관점: 죽으면 → 헬스체크 실패 → LB 차단 → K8s가 새 인스턴스 재시작(self-healing). **Fail fast** 철학.

> **면접 예상 질문:** `HeapDumpOnOutOfMemoryError`를 운영 서버에 항상 켜두면 되지 않나요? 큰 힙에서의 부작용은 무엇인가요?

### 측정 패스 2개 — `ExitOnOOM`이 강제하는 트레이드오프

한 번의 실행으로는 "깨끗한 지표"와 "다중 스레드 발생 위치 증거"를 **둘 다 얻을 수 없어서** 패스를 나눴다.

| 패스 | `ExitOnOOM` | 얻는 것 | 잃는 것 |
|---|---|---|---|
| **본 패스** | ON | 깨끗한 GC/heap/k6 지표 | 다중 스레드 OOM 스택 (첫 OOM에 즉시 종료) |
| **보조 패스** | OFF | 스레드별 발생 위치(37건 전부 `*-8080-exec-N`) | 깨끗한 지표(좀비 limping으로 오염) |

- 근본 원인: **`ExitOnOOM` 하나가 ON/OFF로 정반대 동작을 강제**한다. 깨끗한 지표는 즉시 죽어야(ON),
  다중 스택은 버텨야(OFF) 얻을 수 있어 양립 불가 → 목적별로 실험을 분리한 것.
- 이건 단순 결과가 아니라 **측정 방법론(methodology)에 대한 이해**를 보여주는 디테일.

> **면접 예상 질문:** 동일한 OOM을 두 번 측정했다고 했는데, 왜 한 번의 실행으로 모든 데이터를 못 얻나요?

### OOM 종류와 확인 위치 — 종류는 로그, 정황은 GC 로그, 범인은 힙덤프

- **발생 위치(스레드)가 곧 수사 방향**: `http-nio-...exec`(요청 처리)면 트래픽이 범인,
  `scheduling`/`batch`면 누수 의심. → 위치별로 조치가 달라진다.
- OOM은 위치보다 **종류(메시지)** 로 외우는 게 강하다:

  | 종류 | 의미 |
  |---|---|
  | `Java heap space` (이번) | 힙 부족 — 객체를 너무 많이/크게 |
  | `GC overhead limit exceeded` | GC는 도는데 회수가 거의 안 됨(누수 전조) |
  | `Metaspace` | 클래스 메타정보 영역 부족 |
  | `unable to create native thread` | 스레드 과다 생성 |

- 확인 3단 콤보: **종류 = 앱 로그**, **정황 = GC 로그(`-Xlog:gc*`)**, **범인 = 힙덤프 + MAT**.
- 스택 트레이스는 **맨 아래 `Caused by`부터 거꾸로**(양파 껍질) 읽는다.

```
jakarta.servlet.ServletException: Handler dispatch failed: java.lang.OutOfMemoryError: Java heap space
    at org.springframework.web.servlet.DispatcherServlet.doDispatch(...)   ← 요청 디스패치 경로
Caused by: java.lang.OutOfMemoryError: Java heap space   ← 종류는 여기서 확인
```

> **면접 예상 질문:** OOM이 발생했을 때 어떤 종류인지, 어디서 터졌는지를 각각 어떤 자료로 확인하나요?

### GC 데스스파이럴 & 좀비 서버 — Full GC를 돌려도 안 비워질 때

- OOM은 힙이 차자마자 바로 나는 게 아니다. JVM은 마지막에 **Full GC를 풀파워로** 돌리고,
  그래도 공간을 거의 못 비우면 그제서야 OOM을 던진다.
- GC 로그 집계에서 `Pause Full`이 비정상적으로 반복(예: Full 70회 중 회수 ≤2M 47회)되면 **데스스파이럴**:
  1. 힙 거의 참 → Full GC → 2. in-flight 객체가 살아있어 거의 못 비움 → 3. STW로 요청 밀림 → 4. 다시 참 → 반복.
- 이때 서버는 **"살아있지만 죽은(좀비)"** 상태: CPU 100%(GC), 처리량 0. 프로세스가 안 죽어 **헬스체크는 통과** →
  LB가 계속 트래픽을 보냄 → 장애가 조용히 지속. → 감지는 "프로세스 살아있나"가 아니라 **응답시간/GC 시간 비율**로.

> **면접 예상 질문:** Full GC가 짧은 시간에 수십~수백 번 반복되는 로그를 봤다면, 서버는 어떤 상태이며 왜 위험한가요?

### 힙덤프 히스토그램 & 왜 돈 계산에 `BigDecimal`인가

- 힙덤프(`.hprof`)는 OOM 순간의 힙 **스냅샷** + 분석용 메타정보(클래스 구조·참조 관계·GC root)까지 담아,
  **덤프 파일이 힙보다 크다**(예: 256m 힙 → 덤프 380MB). 원본은 날것이라 `hprof_histo.py`/MAT로 **파싱**해서 본다.
- 히스토그램 1등이 `BigDecimal`(예: 2,400,532개 = 73.3MB, 힙의 41%)이었다. **왜 무거운 BigDecimal을 쓰나?**
  - `double`은 **2진수 부동소수점**(가수 × 2^지수): 범위는 넓지만 10진 소수에 오차. `0.1`은 2진수로 무한 반복 →
    반올림 절단 → `0.1 + 0.2 = 0.30000000000000004`. 돈 계산에선 치명적.
  - `BigDecimal`은 값을 **정수(`unscaledValue` = `BigInteger`) + 소수점 위치(`scale`)** 로 분리 저장 →
    2진 반올림 오차 원천 제거. (히스토그램에 `BigDecimal`과 `BigInteger`가 함께 보이는 이유)
- **트레이드오프**: `double` 8바이트 고정 vs `BigDecimal` 객체 그래프(헤더+BigInteger+int[]). 약 **4배** 메모리.
  단 돈은 정확성 타협 불가 → `double` 전환이 아니라 **힙 right-sizing·동시성 제한**으로 푼다.

> **면접 예상 질문:** 돈 계산에 `double` 대신 `BigDecimal`을 쓰는 이유는? 그 선택이 메모리에 치르는 대가는 무엇인가요?

### 누수(leak) vs volume·동시성형 — 욕조 비유

힙덤프의 `SalesRow` 수를 요청당 행으로 나누면(예: 1,210,500 ÷ 200,000 ≈ **6**) "동시 ~6개 요청분이 live"가 나온다.
메모리가 **단일 캐시가 아니라 in-flight 요청들에 분산** → **누수 아님, volume·동시성형 확정**.

**욕조에 물이 넘친(OOM)** 원인 두 가지:

| | volume·동시성형 (이번) | 메모리 누수 |
|---|---|---|
| 비유 | 수도꼭지를 한꺼번에 많이 틀었다 | 배수구가 막혔다 |
| 본질 | 회수 **가능**한 working-set이 순간 초과 | 회수 **불가능**(참조 살아있음)이 계속 쌓임 |
| 힙 그래프 | **톱니**(오르락내리락) | **단조 우상향** |
| 부하 멈추면 | 회수됨(정상) | 안 빠짐 |
| 힙덤프 범인 | 여러 요청에 **분산** | 한 곳(Map/캐시)에 **집중** |
| 해결 | 힙↑, 동시성 제한 | 버그 수정 |

- GC는 **참조 없는 객체만** 회수한다. 누수는 "안 쓰는데 참조가 살아있는" 객체 → MAT의 **Path to GC Root**로 추적.

> **면접 예상 질문:** 트래픽이 거의 없는 새벽에도 힙이 계속 우상향하다 OOM이 났다면, 누수와 volume형 중 어느 쪽이며 근거는?

### GC 로그로 누수 반박하기 — heap before/after 톱니의 3막

GC 로그에서 추출한 `heap_before_mb → heap_after_mb`(GC 전 → 후, 차이가 회수량)가 결정적 증거다.

```
# 1막 — 건강한 톱니 (회수 정상 = 누수 아님)
19.846  Young  175 -> 123   (52MB 회수)
22.044  Full   256 -> 111   (145MB 회수)
# 2막 — 회수 점점 감소
28.061  Full   252 -> 247   (5MB)
# 3막 — 힙 고착 & 데스스파이럴
30.624  Full   254 -> 254   (회수 0)
31.719  Full   254 -> 254   (직후 OOM)
```

- **면접관 반박**: "마지막에 `254->254`로 회수가 0인데, 이거 누수 아니에요?"
- **재반박**: 누수라면 **처음부터** 우상향이어야 한다. 1막에서 `175->123`처럼 잘 회수됐으니 객체는 회수 **가능**했다.
  마지막 고착은 garbage라서가 아니라 그 순간 ~6개 요청이 **실제로** 250MB를 정당하게 점유한 **동시 live working-set 초과**.
  → 누수 = "안 쓰는데 못 비움", 이번 = "**실제로 쓰는 중이라 안 비움**". 본질이 다르다.

> **면접 예상 질문:** 힙이 마지막에 `254->254`로 회수가 0이 됐는데도 메모리 누수가 아니라고 판단한 근거는 무엇인가요?

### k6 부하 테스트 지표 함정 — `http_req_duration` p95 = 0의 정체

대량 실패 상황에서 평소 쓰던 `http_req_duration`을 그대로 믿으면 안 된다.

- `http_req_duration`은 **성공/실패를 안 가리고** 모든 요청을 포함한다. OOM으로 99.7%가 connection refused로
  **즉시(≈0ms) 실패**하면, 빠른 순서로 줄 세웠을 때 앞쪽 99.7%가 전부 0 → **p95 = 0**(med·p90도 0).
  avg만 13.76ms인 건 max 7272ms짜리 소수가 끌어올린 것.
- 그래서 지표를 분리해서 봐야 한다:
  - **`ok_req_duration`**(= `http_req_duration{expected_response:true}`): 성공한 요청만 → avg 2.03s / **p95 2.86s**
  - **`http_req_failed`**: 실패율 → **0.9974 (99.74% 실패)**
- 둘 중 하나만 보면 함정:
  - `ok_req_duration`만 → "2초네, 괜찮은데?" = **생존자 편향(survivorship bias)** (죽기 전 운 좋은 20건만 본 것)
  - `http_req_failed`만 → "99.7% 실패" (왜 느렸는지 모름)
- **상태별 지표 전환**: before(OOM, 대부분 실패)에선 `http_req_failed`로 장애를 측정하고,
  after(튜닝, OOM 해소, 대부분 성공)에선 `http_req_duration` p95가 **부활**해 GC별 성능 비교에 쓴다.
  → 지표 신뢰성은 **"요청이 성공하느냐"** 에 달려 있다.
- 보너스: `vus_max`=100인데 실제 `vus` max=14에서 멈춤 = **14 VU 시점에 앱 사망** = breaking point가 숫자로 박힘.

> **면접 예상 질문:** 부하 테스트에서 `http_req_duration`의 p95가 0으로 나왔습니다. 왜 그럴 수 있고, 무엇을 대신 봐야 하나요?

### 증거 기반 분석 — 주장 ↔ 산출물 1:1 매칭

"추측이 아니라 측정"을 증명하려면 모든 주장에 증거 파일을 짝지을 수 있어야 한다.

| 주장 | 증거 파일 |
|---|---|
| OOM = 요청 처리 경로 | `oom-stacktrace-*.txt` (`DispatcherServlet.doDispatch` 스택) |
| 누수 아님(톱니 회수) | `heap-*.csv` (`heap_before -> heap_after`) |
| 성공도 2초 느림 + 99.7% 실패 | `k6-*.txt` / `.json` |
| `BigDecimal`·`SalesRow`가 힙 점유 | `histo-hprof-*.txt` (원본 `.hprof`를 파싱한 산출물) |

- 분석 워크플로우: **원본 덤프(.hprof) → 도구로 파싱 → 사람이 읽는 산출물(.txt)**.

### JVM 튜닝 3후보 — 측정으로 조합을 고른다

| 후보 | 방법 | 트레이드오프 |
|---|---|---|
| ① 힙 right-sizing | `256m → 1g` | 물리 메모리 한계로 무한 증설 불가, 트래픽↑면 또 터짐(근본 해결 X). **힙↑ → GC 청소 면적↑ → STW↑** |
| ② GC 교체 | ZGC vs G1 vs ParallelGC | ParallelGC=처리량/STW 길다, G1=균형(기본), **ZGC=Zero pause(1ms 이하)**. ZGC도 CPU·메모리 더 쓰고 처리량은 G1보다↓ |
| ③ 동시성 제한 | `MaxRAMPercentage` / Tomcat 스레드 상한 | 동시 6→3 제한 시 ~240MB→~120MB로 버팀(원인 정면 겨냥). 대신 **처리량↓, 대기 요청 응답 지연↑** |

- 핵심: 힙을 키우면 STW가 길어지므로 ②(ZGC)로 상쇄하고, 메모리 압박이 크면 ③을 더한다.
- **"셋 중 무엇이 정답?"은 함정**. 하나로 단정하지 말고 **before/after를 측정해 우리 서비스 특성(처리량 vs 지연)에 맞는 조합**을 고른다.

> **면접 예상 질문:** OOM 해결책으로 힙 증설·GC 교체·동시성 제한이 있는데, 무엇을 어떤 기준으로 선택하겠어요?

## 학습 정리

- **재현 + 측정**이 분석의 전제다. 고정 힙으로 변수를 통제하고, `HeapDumpOnOOM`·`-Xlog:gc*`로 증거를 남기고,
  `ExitOnOOM` ON/OFF의 트레이드오프 때문에 측정 패스를 2개로 나눴다.
- OOM 진단 3단: **종류(앱 로그) → 정황(GC 로그) → 범인(힙덤프/MAT)**. 스택은 `Caused by`부터 거꾸로 읽는다.
- **누수 vs volume·동시성형**의 결정적 차이는 "부하를 멈추면 회수되는가"(톱니 vs 단조 우상향)와 "범인이 한 곳에
  집중인가 분산인가"다. 마지막 `254->254` 회수 0은 garbage가 아니라 동시 live working-set 초과였다.
- k6 `http_req_duration`은 실패 요청이 섞여 대량 실패 상황에서 p95가 0으로 오염된다. `ok_req_duration`(성공)과
  `http_req_failed`(실패율)를 함께 봐야 하며(생존자 편향 주의), 지표 신뢰성은 상태(성공률)에 따라 전환된다.
- 돈은 정확성 타협 불가라 `BigDecimal`(double의 ~4배)을 쓰며, OOM은 코드가 아니라 **힙 증설·GC 교체·동시성 제한을
  측정으로 비교**해 해결한다. 정답을 미리 정하지 않고 before/after로 고른다.
