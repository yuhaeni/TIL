# Strategy vs Template Method와 결합도

> 날짜: 2026-05-06

## 내용

### 추상 클래스의 결합도 — "내부 구현까지 알게 된다"

**결합도가 높다는 것 = 자식 클래스가 부모의 내부 구현(데이터 저장 방식)까지 의존한다는 것.**

추상 클래스(Template Method)는 `protected` 인스턴스 필드를 둘 수 있어 자식이 `this.baseRate` 처럼 **직접 참조**한다. 그 결과:

- 부모 필드 타입을 `BigDecimal → long`으로 바꾸면 → **자식 클래스 전부 컴파일 에러/로직 깨짐**
- 변경의 파급 범위가 크다 = **결합도 높음**

반면 인터페이스 + `default` 메서드는 **내부 상태 없이 행동의 계약**만 공유 → 자식이 부모의 데이터 저장 방식에 의존하지 않음 → 결합도 낮음.

> **면접 예상 질문:** "결합도가 높다"는 말이 구체적으로 무슨 뜻인가? 추상 클래스에서 부모 필드 타입이 바뀌면 어떤 일이 벌어지는가?

---

### 인터페이스가 인스턴스 필드를 못 가지는 이유 — 다이아몬드 문제

**인터페이스에 둘 수 있는 것:**
- ✅ `public static final` 상수 (모든 구현체 공유)
- ✅ `default` 메서드 (행동 공유)
- ❌ 인스턴스 필드 (상태 충돌 위험)

**Diamond Problem:** 인터페이스 `A`, `B`가 둘 다 `int count` 인스턴스 필드를 가질 수 있다고 가정하고, 한 클래스가 두 인터페이스를 동시 구현하면:

```
class C implements A, B { ... }
obj.count → A의 count? B의 count? ❌ 모호
```

→ 자바는 인터페이스에 **인스턴스 상태 자체를 못 두게 막아** 다이아몬드 문제를 원천 차단.

**대비 — C++의 다중 상속:** 가상 상속(`virtual inheritance`)으로 해결 시도하지만 복잡도 폭증. 자바는 **단일 상속 + 다중 구현(인터페이스)** 으로 단순화.

> **면접 예상 질문:** 자바 인터페이스가 인스턴스 필드를 갖지 못하는 이유는? 다이아몬드 문제와 어떻게 연결되는가?

### `default` 메서드에서 자식 상태를 쓰는 법 — 추상 getter 계약

인터페이스에 **추상 getter**를 선언 → 자식이 자기 필드 노출을 강제 → `default` 메서드는 getter 호출로 우회 접근.

```java
interface LimitStrategy {
    BigDecimal getBaseRate();  // 추상 — 자식이 구현 강제

    default BigDecimal calculate() {
        return getBaseRate().multiply(...);  // 자식 상태에 간접 접근
    }
}
```

**핵심:** 인터페이스는 **상태를 직접 갖지 않고**, 자식의 상태를 **getter 계약을 통해 빌려 쓴다**. 이 우회 구조 자체가 결합도를 낮춘다.

> **면접 예상 질문:** 인터페이스 `default` 메서드에서 자식의 상태를 어떻게 활용할 수 있는가? 이 방식이 왜 결합도를 낮추는가?

---

### 인터페이스 + default 방식의 트레이드오프 — 필드 중복

**공통 필드가 많아질수록 코드 중복 ↑.**

```java
// 추상 클래스 — 부모에 한 번 선언
abstract class LimitStrategy {
    protected BigDecimal baseRate;  // 한 곳에서만 정의
}

// 인터페이스 — 자식마다 중복
class RateStrategy implements LimitStrategy {
    private BigDecimal baseRate;
    public BigDecimal getBaseRate() { return baseRate; }
}
class AmountStrategy implements LimitStrategy {
    private BigDecimal baseRate;                                // 같은 필드 또 선언
    public BigDecimal getBaseRate() { return baseRate; }        // 같은 Getter 또 구현
}
```

**감수 가능한 경우:** 공유할 상태가 거의 없고 **계산 로직(행동)** 위주일 때.

**감수 불가능한 경우:** 공유 필드가 많아지면 **추상 클래스로 마이그레이션** 또는 **컴포지션**(공통 상태를 별도 클래스로 추출해 Strategy가 보유)으로 전환 검토.

> **면접 예상 질문:** 인터페이스 + default 방식의 단점은? 공통 필드가 많아지면 어떻게 대응할 것인가?

### 선택 기준 — Strategy(인터페이스) vs Template Method(추상 클래스)

| 기준 | 추상 클래스 (Template Method) | 인터페이스 + default (Strategy) |
|---|---|---|
| 공통 **상태** 공유 | ✅ 쉬움 (필드 상속) | ❌ 자식마다 중복 |
| 공통 **행동** 공유 | ✅ | ✅ (default 메서드) |
| 상속/구현 개수 | 단일 상속 (`extends`) | 다중 구현 (`implements`) |
| 결합도 | 높음 (내부 구현 노출) | 낮음 (계약만 노출) |
| 확장성 | 다른 클래스 추가 상속 불가 | 다른 인터페이스 추가 구현 가능 |

**선택 가이드라인:**
- 공유할 게 "**상태 + 행동**" → 추상 클래스
- 공유할 게 "**행동만**" → 인터페이스 + default
- **변경이 잦은 도메인** (기획 자주 바뀜) → 결합도 낮은 인터페이스 우선

> **면접 예상 질문:** Strategy 패턴과 Template Method 패턴 중 어떻게 선택하는가? 변경이 잦은 도메인에서 어느 쪽이 유리한가?

### 실제 케이스 — 대출 한도 계산 리팩토링

**상황:**
- 한도 **요율** 계산과 한도 **금액** 계산이 **계산 흐름은 동일, 세부 로직만 다름**
- 기획이 자주 변경되는 도메인 → 결합도 최소화 필요

**선택:** 인터페이스 + `default` 메서드 (Strategy)

- 공통 흐름은 `default` 메서드로 두고
- 세부 계산은 각 전략 클래스에서 오버라이드
- 다중 구현 가능 → 다른 정책 인터페이스도 함께 적용 가능

**감수한 트레이드오프:** 공통 필드 중복 + Getter 강제. 그러나 한도 계산은 **상태 공유보다 계산 로직 중심**이라 비용이 작음.

**면접 답변 템플릿:**
> "Template Method는 추상 클래스라 단일 상속만 가능하고, 자식이 부모 내부 구현까지 알아 결합도가 높습니다. 한도 계산은 기획이 자주 변경되는 로직이라 다중 구현이 가능하고 결합도가 낮은 인터페이스가 적합하다 판단했습니다. 계산 흐름은 같고 세부만 다른 구조라 공통 흐름은 `default`로 두고, 세부는 각 전략에서 오버라이드했습니다. 인터페이스는 공통 필드 사용 시 자식마다 필드+Getter 중복이 생기지만, 이 도메인은 상태 공유보다 계산 로직 중심이라 그 비용이 작아 인터페이스 방식이 적합하다 판단했습니다."

> **면접 예상 질문:** 변경이 잦은 도메인에서 Strategy 패턴을 인터페이스로 구현한 사례를 설명해보라.

## 학습 정리

- 추상 클래스의 **결합도가 높다**는 것 = 자식이 부모의 **내부 구현(필드 저장 방식)** 까지 의존
- 인터페이스가 인스턴스 필드를 못 갖는 건 **다이아몬드 문제** 원천 차단 — 자바는 단일 상속 + 다중 구현 모델
- `default` 메서드에서 자식 상태가 필요하면 **추상 getter 계약**으로 우회 접근 → 결합도 낮춤
- 인터페이스 방식의 단점은 **공통 필드 중복** — 상태 공유 ↑면 추상 클래스/컴포지션 마이그레이션 검토
- 선택 기준: 공유할 게 **상태+행동이면 추상 클래스**, **행동만이면 인터페이스 + default**
- 변경 잦은 도메인은 **결합도 낮은 인터페이스** 우선 — 다중 구현 + 파급 범위 최소화
