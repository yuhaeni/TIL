# GC 컬렉터 비교 — Parallel, G1, ZGC, Allocation Stall, live 객체

> 날짜: 2026-06-26

## 내용

자바(JVM)의 주요 GC 컬렉터 **Parallel · G1 · ZGC**가 서로 어떻게 다른지를 정리한다.
세 컬렉터를 가르는 축은 두 가지뿐이다: **(1) 무엇을 우선하는가 — 처리량 vs 지연**, **(2) 언제 멈추는가 — STW vs 동시(concurrent) 수거**.
그리고 모든 컬렉터에 공통인 대전제 하나 — **GC는 live(reachable) 객체는 절대 못 치운다**를 먼저 짚는다.

### GC는 garbage만 치운다 — live 객체엔 무력

GC가 회수할 수 있는 객체는 **참조 관계가 끊긴(unreachable) 객체**뿐이다.
어떤 컬렉터를 쓰든 이 원칙은 동일하다 — GC는 "쓰레기(garbage)"를 치우는 것이지 "살아있는(live)" 객체는 건드리지 못한다.

따라서 데이터가 작업이 끝날 때까지 **계속 reachable한 상태로 누적**되면(예: 전체 결과를 메모리에 모았다가 한 번에 내보내는 구조), 어떤 컬렉터로 바꿔도 회수할 garbage 자체가 없어 메모리 부족(OOM)을 못 막는다.
즉 컬렉터 선택은 **"같은 압력 아래의 거동(지연·처리량)"** 을 바꿀 뿐, **회수 가능한 양 자체**를 바꾸지는 못한다. 그건 코드(참조를 끊는 설계)나 힙 용량의 영역이다.

> **면접 예상 질문:** GC 컬렉터를 바꿔도 OOM이 안 사라지는 경우가 있습니다. GC는 메모리를 회수하는 역할인데 왜 컬렉터 교체로 해결이 안 되나요?

---

### Parallel — 처리량 우선, 멈춤은 신경 안 씀

- **동작**: Young도 Old도 **전부 Stop-The-World**. 앱을 완전히 멈추고 여러 GC 스레드가 동시에 청소 + 압축(compaction)을 한 번에 끝낸다.
- **장점**: GC에 쓰는 총 CPU 시간이 가장 적음 = **처리량(throughput) 최강**. 멈춘 김에 압축까지 해서 단편화가 없다.
- **단점**: Old가 가득 차면 **Full GC = 힙 전체를 멈추고 통째로 압축** → 힙이 크면 멈춤(STW)이 수백 ms~초 단위로 길어진다. 멈춤 시간 목표치 같은 개념이 없다(그냥 빨리 끝내기).
- **힙 구조**: Young/Old가 **고정된 통짜 연속 영역**.
- **언제**: 배치처럼 잠깐 멈춰도 괜찮고 전체를 빨리 끝내는 게 중요한 워크로드.

> **면접 예상 질문:** Parallel GC는 어떤 워크로드에 적합한가요? 처리량이 가장 좋은데도 요즘 서버의 기본값이 아닌 이유는 무엇인가요?

---

### G1 — region 조각내서 garbage 많은 것부터 (JDK9+ 기본)

- **동작**: 힙을 **수천 개 동일 크기 region**으로 쪼갠다(보통 1~32MB). 각 region은 Eden/Survivor/Old 역할이 **동적으로** 바뀐다. (Parallel은 Young/Old 칸막이가 고정인 것과 대조)
- **수거 전략**: garbage가 가장 많은 region부터 골라 청소한다(이름이 **Garbage-First**). 목표 멈춤시간(`MaxGCPauseMillis`, 기본 200ms)에 맞춰 "이번엔 이 region들만" 하고 양을 조절한다.
- **Old 처리**: **Concurrent Marking**(앱이 도는 중에 live 객체를 표시) 단계가 있어 보통 Full GC를 회피한다. 단, 회수 속도가 할당 속도를 못 따라가면 **Evacuation Failure(대피 실패)** → 최후엔 Full GC로 추락.
- **Parallel과의 핵심 차이**: 처리량 최우선(Parallel) vs **멈춤 시간 균형**(G1). 통짜 힙을 통째로 멈추는 대신, 조각내서 일부만 골라 치우므로 한 번의 멈춤이 짧고 예측 가능하다. → 그래서 대부분 서버의 **기본값**.

> **면접 예상 질문:** Parallel과 G1의 구체적인 차이는 무엇인가요? G1이 요즘 서버의 기본 GC가 된 이유를 두 가지 축(처리량/지연, STW/동시 수거) 관점에서 설명해보세요.

### ZGC — 동시 수거로 거의 안 멈춤, 대신 헤드룸 필요

- **동작**: 청소의 **거의 모든 단계를 앱과 동시에(concurrent)** 수행한다. STW는 마킹 시작/끝 같은 찰나의 순간만 → 멈춤이 **1ms 미만**(보통 sub-ms). 힙이 수십~수백 GB로 커도 멈춤은 일정하다.
- **어떻게 앱을 멈추지 않고 객체를 옮기나**: **Colored Pointers(컬러 포인터) + Load Barrier(읽기 장벽)** — 포인터에 표식(메타데이터)을 박아두고, 앱이 객체를 읽을 때마다 "이거 옮겨졌나?"를 검사해 즉석에서 주소를 고친다. 그래서 앱을 멈추지 않고도 객체 이동(relocation)이 가능하다.
- **대가**: 동시에 돌리려면 **여유 힙(headroom)이 항상 넉넉**해야 한다. 앱의 객체 할당 **속도**를 GC의 동시 회수 속도가 못 따라가면 → **Allocation Stall**(아래 항목)이 발생한다. 여유 힙이 부족한(힙이 작은) 환경에선 오히려 불리하다.
- **언제**: 응답 지연(pause)에 민감하고 힙이 큰 서비스. **저지연이 목표지만 힙 여유가 빠듯한 환경엔 부적합.**
- 세대형 ZGC(`-XX:+ZGenerational`)는 **JDK 21+** 부터.

> **면접 예상 질문:** ZGC는 STW가 sub-ms로 극도로 짧은데도 항상 최선은 아닙니다. ZGC가 불리해지는 환경은 어떤 경우이고, 그 근본 원인(설계상의 트레이드오프)은 무엇인가요?

### Allocation Stall — STW가 아니라 "할당 대기"로 멈추는 것

**Allocation Stall = 객체를 할당(allocation)하려는데 빈 메모리가 없어 앱 스레드가 막혀(stall) 대기하는 것.**

주차장 비유: 앱 스레드 = 차 대려는 손님(할당=주차), GC = 떠난 차 자리를 비우는 직원, ZGC = 영업 중에 동시에 자리를 비우는 방식.
평소엔 손님 오는 속도 ≈ 직원이 비우는 속도라 항상 빈자리가 있다.
그런데 손님(할당)은 폭주하는데 빈자리(회수)가 안 생기면(치울 garbage가 없거나 회수가 할당을 못 따라감) → 새 손님은 **빈자리가 생길 때까지 입구에서 대기** = Allocation Stall.

| | STW (Stop-The-World) | Allocation Stall |
|---|---|---|
| 누가 멈추나 | **모든** 앱 스레드 일제히 | 할당하려는 스레드만 개별적으로 |
| 왜 멈추나 | GC가 "잠깐 다 멈춰" 하고 일함 | 빈 메모리가 없어서 할당 대기 |
| ZGC에서 | sub-ms (약속대로 극도로 짧음) | 헤드룸 부족 시 **여기서 폭발** |

핵심: ZGC는 **"멈춤 짧게(STW)"라는 약속은 완벽히 지켜도, 앱은 할당 대기(stall)로 굶어 죽을 수 있다.**
"멈춤이 짧다(pause)"와 "앱이 안 멈춘다(throughput)"는 다른 얘기다. STW를 아무리 줄여도 **메모리 빈 공간 자체가 없으면 OOM은 못 막는다.**
즉 문제의 병목이 **멈춤 시간**인지 **메모리 용량**인지를 구분해야 하고, 메모리 용량이 부족한 문제에 저지연 컬렉터(ZGC)를 들이대는 것은 도구를 잘못 고른 것이다.

> **면접 예상 질문:** Allocation Stall과 STW(Stop-The-World)의 차이는 무엇인가요? ZGC에서 STW는 sub-ms인데도 Allocation Stall로 앱이 사실상 멈추는 상황을 설명해보세요.

### 세 컬렉터 비교 — OOM 한계가 아니라 "죽는 방식"이 다르다

세 컬렉터 모두 **live 객체는 못 치운다**는 점이 동일하므로, 같은 메모리 압력에서 **OOM이 나는 한계 자체는 비슷**하다.
다른 것은 **"같은 압력에서 어떻게 버티고 어떤 방식으로 붕괴하느냐"** 다.

| | Parallel | G1 | ZGC |
|---|---|---|---|
| 우선 가치 | **처리량** | 균형(지연) | **초저지연** |
| 멈춤(STW) | 길다 | 짧다(목표치 맞춤) | sub-ms |
| 청소 시점 | 전부 STW | 일부 동시(마킹) | 거의 전부 동시 |
| 힙 구조 | 통짜 Young/Old(고정) | region 조각(동적 역할) | region + 컬러 포인터 |
| 헤드룸 요구 | 적음 | 중간 | **많이 필요** |
| 압력 시 붕괴 모드 | Full GC 폭주 | Evacuation Failure → Full GC | **Allocation Stall** |
| 기본값 여부 | 과거(JDK8) | **현재(JDK9+) 기본** | 저지연·대용량 힙에 선택 적용 |

정리하면: **컬렉터 선택은 "지연이냐 처리량이냐"를 고르는 것**이지, 회수 가능한 메모리 양을 늘리는 게 아니다.
"최신·저지연이라서 ZGC"가 아니라 **"멈춤 시간이 문제인지, 메모리 용량이 문제인지에 맞는 도구"**를 골라야 한다.

> **면접 예상 질문:** Parallel·G1·ZGC를 하나의 표로 비교한다면 어떤 축으로 나누겠습니까? "최신 컬렉터가 항상 최선은 아니다"를 어떤 근거로 설명하겠습니까?

## 학습 정리

- 모든 GC는 **참조가 끊긴(unreachable) garbage만** 회수한다. live 객체가 누적되면 어떤 컬렉터로도 못 치우므로, **컬렉터 교체는 회수량이 아니라 거동(지연·처리량)만 바꾼다.**
- **Parallel**: 처리량 최강, 통짜 힙 전부 STW, Old 차면 Full GC로 길게 멈춤. 배치형에 적합.
- **G1**: 힙을 region 조각으로 나눠 garbage 많은 것부터(Garbage-First), 목표 멈춤시간을 맞추는 균형형. JDK9+ 기본값.
- **ZGC**: Colored Pointer + Load Barrier로 거의 멈추지 않지만(STW sub-ms), 동시 수거를 위해 **여유 힙(헤드룸)이 필수** → 힙이 작은 환경에선 Allocation Stall로 오히려 불리.
- **Allocation Stall**은 STW가 아니라 "빈 메모리가 없어 할당이 막히는 것". 멈춤 시간을 줄여도 메모리 용량이 부족하면 못 막는다 → **멈춤 시간이 문제인지, 메모리 용량이 문제인지를 구분**해 컬렉터를 골라야 한다.
