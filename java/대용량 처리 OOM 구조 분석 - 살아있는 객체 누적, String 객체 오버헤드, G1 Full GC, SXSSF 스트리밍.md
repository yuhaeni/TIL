# 대용량 처리 OOM 구조 분석 — 살아있는 객체 누적, String 객체 오버헤드, G1 Full GC, SXSSF 스트리밍

> 날짜: 2026-06-25

대용량 데이터를 메모리에 적재해 처리하는 경로에서 OOM이 발생하는 **구조적 원인**과, GC 튜닝 vs 구조 개선의 차이를 개념 위주로 정리한다. 핵심 질문은 "GC가 있는데 왜 OOM이 나는가", "힙을 키우면 해결되는가"이다.

## 내용

### OOM의 본질 — 입력 크기에 선형 종속하는 힙 점유

전 행을 한꺼번에 메모리에 적재해 결과물을 만드는 전형적 안티패턴은, 동일 데이터를 **여러 겹으로 힙에 동시 적재**한다.

```
파일 전체 byte[]                    // ① 파일 전체 (행수 × 레코드폭)
→ List<RowData>                     // ② 전 행 객체화 (행당 N개 String 필드)
→ List<Body>                        // ②' map/stream으로 변환하며 또 적재
→ XSSFWorkbook                      // ③ 전 행을 메모리에 올린 워크북
→ ByteArrayOutputStream → byte[]    // ④ 직렬화 버퍼 + 응답 byte[]
```

핵심은 **①~④가 응답이 끝날 때까지 모두 살아있다(= GC가 회수할 수 없다)** 는 점이다. GC는 "아무도 참조하지 않는 쓰레기"만 회수할 수 있는데, 이들은 전부 참조가 살아있어 **회수가 불가능**하다. 결과적으로 **힙 점유가 입력 행수에 선형 종속**한다.

코드 레벨에서 객체를 계속 살려두는 흔한 함정:
- **지역 변수**는 메서드가 끝날 때까지 스택에서 강한 참조로 살아있다. 예를 들어 메서드 끝의 `log.info("bytes={}", data.length)` 한 줄이 `data`(①)를 메서드 끝까지 회수하지 못하게 붙잡는다.
- **중간 산출물 미해제**: `bodies`를 만들었으면 원본 `rows`는 버려도 되는데, `rows`도 지역 변수라 메서드 끝까지 안 놓아주어 `rows`와 `bodies`가 **동시에** 힙에 뜬다.

> **면접 예상 질문:** GC가 있는데 왜 전 행 적재 방식은 OOM이 나나요? 어떤 객체가 왜 회수되지 못하는지 코드 레벨에서 설명해보세요.

### String 객체 오버헤드 — 데이터 양이 아니라 객체화가 문제

같은 데이터라도 `byte[]` 한 덩어리로 들 때와 작은 String 여러 개로 쪼갤 때 메모리 차이가 크다. 범인은 **데이터 양이 아니라, 잘게 객체화하면서 붙는 객체 오버헤드**다.

비유: byte[]는 **큰 상자 1개**에 물건을 꽉 채운 것(포장재 1번). String 여러 개는 **작은 상자 여러 개**에 나눠 담은 것(빈 상자·운송장·손잡이가 상자 수만큼). "고정 오버헤드" = 내용물이 1개든 10개든 **상자마다 무조건 붙는 빈 상자 무게**.

Java String 객체 1개의 구성 요소(한 글자여도 ~40B):

| 구성 요소 | 설명 | 대략 |
|---|---|---|
| 객체 헤더 | 모든 자바 객체의 꼬리표(타입/GC 정보) | 12~16B |
| 내부 byte[] 참조 | 글자를 담은 배열을 **가리키는 주소** | 4~8B |
| 내부 byte[] 자체 | 그 배열도 **별개 객체** → 자기 헤더+길이 | 16B + 글자수 |
| hash 필드 | 캐싱된 해시값 int | 4B |

즉 String 1개당 **객체가 최소 2개(String + 내부 byte[])** 생기고 헤더도 2번 붙는다. 행당 필드가 많을수록(예: 91컬럼) 행 하나가 객체 수백 개로 부푼다. byte[] 한 덩어리(객체 1개)와 비교 불가.

→ 그래서 **파일 byte[] 할당이 아니라 파싱 적재(전 행 객체화)가 진짜 메모리 병목**이 된다. 큰 파일에서 byte[] 읽기는 통과해도 파싱 단계에서 객체가 폭증해 OOM이 나는 패턴이 이 때문이다.

> **면접 예상 질문:** 같은 데이터를 byte[]로 들 때와 String 객체로 쪼갤 때 메모리 차이가 왜 나나요? String 객체 하나가 힙에서 어떻게 구성되죠?

### G1 GC와 Full GC — 포화의 비명

- **G1(Garbage First)**: Java 9+ 기본 GC. 힙을 수백 개 region으로 쪼개 쓰레기 많은 곳부터 우선 청소, STW(Stop-The-World)를 짧고 예측 가능하게 유지한다.
- **Full GC**: 힙 전체(Young+Old)를 멈추고 청소하는 가장 무거운 GC. **G1은 평소 Full GC를 피하도록** 설계됨 → Full GC 등장 = "region 단위 조금씩 청소로는 감당이 안 됨"의 비명. 힙이 한계에 가까울수록 Full GC 횟수가 급증한다.
- **Evac Fail(Evacuation Failure)**: G1이 살아있는 객체를 빈 region으로 대피시키려는데 **옮길 빈 공간이 없어 실패** = 힙 포화 신호.

**GC 후 회수 여부의 차이는 살아있는 데이터 양**이다. 입력이 작으면 GC가 쓰레기를 치워 힙이 확 내려가지만(예: peak 1,173M → 326M), 입력이 커지면 **살아있는(회수 불가능한) 데이터 자체가 선형으로 커져** 천장에 포화된다. 이때 GC가 쓰레기를 다 치워도 **줄일 쓰레기가 없어** 천장에 붙어버린다.

OOM은 즉사가 아니라 **Full GC 폭주를 거쳐 "느려지다 죽는다"**. 운영에선 "다운로드 응답 없음 → 스레드 점유 → 결국 500"으로 나타나고, 힙이 공유 자원이라 **한 요청의 OOM이 서버 전체 장애로 전파**된다.

> **면접 예상 질문:** GC가 도는데도 힙 사용량이 안 줄고 천장에 붙는 이유는? G1에서 Full GC가 발생한다는 건 어떤 신호인가요?

### 힙 증설 vs 구조적 해결 — 임계만 미루기 vs 종속 끊기

OOM 임계 행수(이하 `R_oom`, 첫 OOM이 나는 입력 크기)는 힙을 키우면 같이 올라간다. 하지만 **힙 점유가 입력에 선형 종속**하는 구조 자체는 그대로라, 입력이 더 커지면 또 터진다. 즉 **힙 증설(1.5G→3G→6G)은 OOM 지점을 오른쪽으로 미룰 뿐 근본 해결이 아니다.**

근본 해결 = **입력 크기와 메모리를 독립(상수)시키기**.

- **SXSSF + 파일 스트리밍 파싱**: 전 행을 동시 적재하지 않고 **한 행(또는 묶음)씩 읽어 → 쓰고 → 비우는(flush)** 흐름. 메모리에는 최근 N행만 유지하고 나머지는 임시 디스크 파일로 흘려보낸다.
- 비유: 양동이 100개를 다 채워 한 번에 옮기는 게 아니라(다 들고 있어야 함), **컨베이어 벨트처럼 한 개 채우면 바로 흘려보내기**(손엔 항상 1개). 100만 개든 1개든 손에 든 무게는 일정.
- 결과: **힙이 입력 크기와 독립(상수)** → R_oom 소멸. 트레이드오프는 디스크 I/O 증가와 임시파일 관리.

> **면접 예상 질문:** 힙을 키우면 OOM이 해결되지 않나요? 왜 그것이 근본 해결이 아닌지, SXSSF는 어떻게 메모리를 입력 크기와 독립시키는지 설명해보세요.

### 부하 테스트 설계 — 측정 신뢰성과 대조군

OOM 같은 현상을 "측정으로" 규명할 때 신뢰성을 확보하는 개념들.

- **매 측정마다 새 JVM**: 같은 JVM에서 연속 측정하면 **이전 실행의 잔여 힙**이 다음 측정 시작점에 깔려 "그 입력 단독의 순수 힙 사용량"을 못 잰다. 매번 깨끗한 0에서 출발해야 변수가 격리된다.
- **JIT 워밍업과 지표 신뢰성**: JIT(Just-In-Time) 컴파일러는 자주 실행되는 코드를 기계어로 즉석 컴파일·캐싱한다. JVM을 갓 띄운 상태에서는 **소요 시간에 워밍업 비용이 섞이므로** 시간은 참고치로만 본다. 반면 **힙 사용량·OOM 임계는 워밍업과 무관**(같은 입력은 JIT 상태와 상관없이 같은 양의 객체를 쌓음)하여 핵심 지표는 JIT의 영향을 받지 않는다.
- **증상-원인 교차 검증**: HTTP 500은 NPE·타임아웃 등 온갖 원인의 증상일 뿐이다. **GC 로그의 OutOfMemoryError(원인) + HTTP 500(증상)** 두 신호가 동시에 잡혀야 OOM으로 확정한다.
- **통제 변수 고정**: 두 해결책을 비교할 땐 힙 크기처럼 바뀌면 안 되는 값을 **상수로 고정**하고 변수 하나(예: GC 알고리즘)만 바꿔야 인과가 깨끗하다. 힙을 같이 키우면 "무엇이 효과를 냈는지" 구분이 안 된다.
- **대조군의 가치**: GC 튜닝처럼 "값싼 대안"이 **안 통한다는 것을 데이터로 입증**해야 비싼 구조 변경(SXSSF)이 정당화된다. 살아있는 객체가 병목이면 ZGC로 STW를 밀리초로 줄여도 **살아있는 객체는 회수 불가**라 임계는 거의 안 오른다 — 이 실패를 측정으로 보여주는 것이 "JVM 튜닝 한계 → 구조적 해결" 서사의 근거가 된다.
- **수치의 도메인 번역**: "16,000행에서 터진다"는 비개발자가 위험도를 판단할 수 없다. "**정상 규모의 약 5배**"처럼 도메인 언어로 번역해야 우선순위(지금 핫픽스인가, 대비책인가)를 합의할 수 있다. OOM은 상시 위험이 아니라 **조건부 위험**이라는 점도 이렇게 드러난다.

> **면접 예상 질문:** OOM을 측정으로 규명할 때 신뢰성을 어떻게 확보하나요? 어차피 실패가 예상되는 GC 튜닝을 왜 굳이 측정(대조군)하나요?

## 학습 정리

- **OOM 본질 = 입력 크기에 선형 종속하는 힙 점유**. 전 행 동시 적재(byte[] + List + 워크북)가 전부 살아있어 GC가 무력. 힙 증설은 임계만 미룰 뿐.
- **String 객체 오버헤드**: 같은 데이터라도 작은 객체로 쪼개면 헤더·내부 배열·참조가 객체 수만큼 붙는다. 그래서 파일 byte[] 할당이 아니라 **파싱 적재가 진짜 병목**.
- **G1 Full GC = 포화의 비명**, Evac Fail = 대피할 빈 공간 없음. GC 후 회수 여부는 살아있는 데이터 양이 좌우한다. "느려지다 죽는다".
- **근본 해결 = 입력-메모리 독립화**. SXSSF 스트리밍은 한 행씩 읽고 쓰고 비워 힙을 상수로 유지(트레이드오프: 디스크 I/O).
- **측정 신뢰성 개념**: 매 측정마다 새 JVM(잔여 힙 격리), JIT 워밍업의 영향을 받지 않는 핵심 지표, 증상-원인 교차 검증, 통제 변수 고정, 대조군으로 값싼 대안의 실패 입증, 수치의 도메인 번역.
