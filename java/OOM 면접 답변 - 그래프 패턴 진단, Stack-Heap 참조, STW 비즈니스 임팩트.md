## 내용

### OOM 3가지 원인과 처방 매칭 — "무작정 `-Xmx` 늘리기"는 왜 안 되는가

동료가 "OOM 났으니 `-Xmx`를 2GB → 4GB로 늘리자"고 제안하면, *원인부터 분석*하지 않고 메모리만 늘리는 건 임시방편이다. OOM의 원인은 3가지이고, 각각 처방이 완전히 다르다.

| 원인 | 본질 | 처방 |
|---|---|---|
| ① 메모리 누수 | 코드 버그로 참조가 안 끊김 → GC 회수 불가 | 코드 수정 (참조 제거, Caffeine TTL 도입) |
| ② 워크로드 과다 | 한 번에 처리하는 양이 비정상적으로 큼 | chunk size↓, 페이징, 스트리밍 |
| ③ 힙 부족 | 정상 워크로드인데 양동이(`-Xmx`)가 너무 작음 | `-Xmx` 증설 (이때만 정답) |

**핵심 통찰**:
- ②와 ③은 헷갈리기 쉽다 — *데이터 양이 비정상*이냐, *양동이 크기가 비정상*이냐
- ①은 코드 문제이므로 `-Xmx`를 늘려도 며칠 후 또 터진다 (시한폭탄)

**비유** 🪣:
```
② 워크로드 과다: 양동이는 적정한데 한 번에 너무 많이 들이부음 💦💦💦
③ 힙 부족:      물의 양은 정상인데 양동이가 너무 작음 🪣💧
```

> **면접 예상 질문:** OOM의 원인 3가지는 각각 무엇이고, 처방은 어떻게 다른가? `-Xmx` 증설이 정답인 경우와 아닌 경우를 어떻게 구분하는가?

### 그래프 패턴 분석으로 원인 진단 — 패턴 A vs 패턴 B

Grafana의 Heap 사용량 그래프를 보면 *어떤 원인의 OOM인지* 판단할 수 있다.

**🟢 정상 패턴 (저점이 일정)**
```
1GB ┤  /\      /\      /\      /\
800 ┤ /  \    /  \    /  \    /  \
500 ┤/    \  /    \  /    \  /    \
200 ┤      \/      \/      \/      \    ← GC 후 매번 200MB로 회수
    └────────────────────────────────→ 시간
```
- GC가 매번 회수 잘 함 → 톱니의 *저점이 일정*

**🔴 패턴 A: 메모리 누수 (저점이 우상향)**
```
1GB ┤              /\      /\
800 ┤      /\     /  \____/  \___   ← GC 돌아도 못 비워지는 메모리가 누적
500 ┤  /\_/  \___/                       (저점이 계속 ⬆️)
200 ┤_/
    └────────────────────────────────→ 시간
```
- GC가 돌아도 회수 못 한 객체가 누적 → 저점 점점 ⬆️
- = **메모리 누수의 결정적 시그니처**

**🔴 패턴 B: GC 폭주 (톱니 매우 촘촘)**
```
500 ┤/\/\/\/\/\/\/\/\/\/\/\/\/\/\
200 ┤        ← GC가 너무 자주, 저점은 회복
    └────────────────────────────────→ 시간
```
- 톱니가 매우 촘촘 = GC가 너무 자주 발생
- 누수는 아니지만 **워크로드 과다** 또는 **힙 부족** 의 신호

> **면접 예상 질문:** Grafana 메모리 그래프에서 메모리 누수를 어떻게 시각적으로 판단하는가? 톱니파의 어느 부분을 봐야 하는가?

### 워크로드 과다 vs 힙 부족 구분 — 시점성 vs 만성

패턴 B (촘촘)는 ②와 ③ 둘 다 가능하다. *시점성*과 *만성* 으로 구분한다.

**🔵 케이스 A: 워크로드 과다 (시점성)**
```
┌─────────────────────────────────
│         /\/\           ← 배치 시점에 메모리 폭발
│        /    \              (저점도 고점도 둘 다 치솟음)
│ ___/\__      \___/\___    ← 작업 끝나면 다시 정상 복귀
└─────────────────────────────────
   평소 정상  ↑ 새벽 2시   평소 정상
              배치 시간
```
- 평소 트래픽엔 멀쩡, *특정 시점*(매일 새벽 2시 배치 등)에만 폭발
- 예: 정산 배치 100만 건 `SELECT *`, 페이징 없이 전체 로딩

**🔴 케이스 B: 힙 부족 (만성)**
```
┌─────────────────────────────────
│ ████████████████████████ ← 항상 -Xmx 상한선에 닿음
│/\/\/\/\/\/\/\/\/\/\/\/\/  ← 톱니 매우 촘촘 (GC 폭주)
│       항상 빠듯한 상태
└─────────────────────────────────
```
- 평소부터 메모리가 항상 빠듯, 정상 트래픽도 GC 빈도 높음
- 특정 시점 없음 → 만성적

**판단 기준**:
- *시점성* → 워크로드 과다 → chunk size 조절, 페이징
- *만성* → 힙 부족 → `-Xmx` 증설

> **면접 예상 질문:** 데이터 양이 문제인지, 힙 크기가 부족한 게 문제인지 어떻게 판단하는가? 그래프 패턴에서 어떤 차이가 보이는가?

### 참조와 GC Roots 메커니즘 — Stack 지역변수도 GC Root

OOM 누수를 이해하려면 *참조*와 *GC Roots*의 관계를 알아야 한다.

**Java 변수의 두 종류**:
- **원시 타입(Primitive)**: 값 자체를 변수에 저장 (`int age = 25;`)
- **참조 타입(Reference)**: 객체의 *주소*를 저장 (`Template t = new Template();`)

```java
Template t = new Template();
```

```
🆕 Stack (process() frame)        🆕 Heap
┌─────────────────┐               ┌──────────────────┐
│ Template t      │               │ Template 객체    │
│ = 0x1234     ───┼──────────────→│ {field1, ...}    │
└─────────────────┘               └──────────────────┘
   ↑ 주소만 저장 (= "참조")        ↑ 객체 실체
```

**GC Roots의 대표 종류**:
1. **클래스의 static 필드** (Method Area) — 거의 영원 ♾️
2. **Stack frame의 지역 변수** — *메서드 실행 중*에만 GC Root ⏳
3. JNI 참조
4. 활성 스레드

**비유** 🏛️:
```
static = 회사 정문에 박힌 명패 (영원)
지역변수 = 회의실 화이트보드에 적은 이름 (회의 끝나면 지움)
```

**메서드 종료 → GC 회수 흐름**:
```java
public void process() {
    Template t = new Template();
    doSomething(t);
}  // ← 여기서 메서드 끝남!
```
1. Stack frame 사라짐 → 지역변수 `t` 소멸 → 객체 주소를 아무도 모름
2. Heap의 `Template 객체`는 *즉시* 회수되지 않음
3. *다음 GC* 발생 시 "참조 없음" 판정 → 회수 ✅

> 💡 **흔한 오해**: "참조가 끊긴다 = 메모리 누수". 반대다! 참조가 *끊겨야* GC가 회수 가능 → 정상. 참조가 *안 끊겨야* 누수.

> **면접 예상 질문:** Java 변수가 객체를 참조한다는 것의 의미는? 메서드가 끝나는 순간 객체는 즉시 회수되는가? 지역변수와 static 필드가 GC Root로서 어떻게 다른가?

### 진짜 캐시 vs 가짜 캐시 — evict 메커니즘이 핵심

`static HashMap`을 "캐시"라고 부르며 `put`만 하고 `remove` 안 하는 코드는 *캐시가 아니라 영구 저장소*다.

```java
public class Cache {
    public static HashMap<Long, Template> CACHE = new HashMap<>();  // ❌
}
// CACHE.put(id, template) 만 호출, remove 없음
// → static 필드(GC Root)가 영원히 객체 참조 → 누수
```

**진짜 캐시의 본질**: 들어왔다 *나갔다* 해야 함 (휘발성). 그래서 evict 정책이 필수.

| 정책 | 의미 |
|---|---|
| 크기 기반 (`maximumSize`) | 최대 N개까지만 보관 |
| 시간 기반 (`expireAfterWrite`) | 쓴 후 N분 지나면 자동 삭제 |
| 접근 기반 (`expireAfterAccess`) | N분 동안 안 쓰면 삭제 |

**Caffeine — 3가지 evict 정책을 모두 지원**:
```java
public static Cache<Long, Template> CACHE = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(10))
    .expireAfterAccess(Duration.ofMinutes(5))
    .build();
```

**핵심 통찰**: Caffeine도 `static`으로 들고 있지만 누수가 안 된다. 왜?

```
🏛️ static 필드 (Method Area)
    → 📦 Caffeine 컨테이너 (Heap) ← 살아있어도 OK (구조 자체는 가벼움)
        → 📋 entry들 (Template 객체) ← 여기를 evict로 자동 정리!
```

**Caffeine과 GC의 협력**:
1. TTL 만료 → Caffeine이 entry를 자기 자료구조에서 *제거* → 참조 끊김
2. 다음 GC → "참조 없음" 판정 → Heap에서 회수

= **Caffeine은 "쥐고 있던 참조를 놓아주는" 역할, 실제 회수는 GC가 함**.

> **면접 예상 질문:** `static HashMap`을 캐시처럼 쓰는 게 왜 누수가 되는가? Caffeine도 `static`인데 왜 누수가 안 되는가? Caffeine과 GC의 역할 분담은?

### STW(Stop The World)와 운영 임팩트 — Major GC가 길어지면 무슨 일이?

GC가 동작할 때 JVM은 *모든 애플리케이션 스레드를 일시 정지*시킨다. 이 정지 시간을 **STW (Stop The World)** 라고 한다.

**왜 멈춰야 하나?** GC가 객체의 살아있음을 판정하는 동안 다른 스레드가 참조를 바꾸면 상태가 꼬임 → 잠깐 모두 멈추고 청소.

**GC별 STW 길이**:
| GC 종류 | STW 길이 | 사용자 체감 |
|---|---|---|
| Minor GC (Young 영역) | 수 ms ~ 수십 ms | 거의 못 느낌 |
| Major GC / Full GC (Old 영역) | 수백 ms ~ 수 초 | 응답 지연 체감 |
| OOM 직전 | STW 연쇄 발생 | 사실상 서비스 다운 |

**비유** 🏢:
```
🧹 사무실 청소 시간 (STW)
- 모든 직원: 일 중단, 손님: 대기실에서 기다림
- 청소 1초 → 손님 "좀 늦네?"
- 청소 10초 → 손님 "왜 이렇게 오래?"
- 청소 60초 → 손님 "이 가게 망했나?" 떠남
```

**실무 임팩트**:
```
😵 STW 1초 = 그 시간 들어온 모든 HTTP 요청 대기
😵 STW 5초 = 사용자 타임아웃, 5xx 에러
😵 STW 10초 = 헬스체크 실패 → ALB가 인스턴스 제외
😵 OOM 직전 = STW 연쇄 = 서비스 다운 = 장애
```

> **면접 예상 질문:** STW는 무엇이고 왜 발생하는가? Minor GC와 Major GC의 STW 차이는? STW가 길어지면 운영 환경에서 어떤 연쇄 효과가 발생하는가?

### OOM 비즈니스 임팩트 답변 — KCB 전문 파일 배치 사례

면접관이 "기술적 임팩트 말고 *비즈니스* 임팩트로 설명해주세요" 라고 묻는 경우, 본인이 경험한 도메인을 끌고 와서 *돈의 흐름*과 *고객 영향*으로 풀어야 한다.

**사례 — 정산 전문 파일 생성 배치**:
```
🏦 도메인 특성:
- 매일 새벽 2시 실행, 어제 거래 데이터 가공
- KCB(신용평가사)로 파일 전송, KCB가 은행에 전달
- 셀러들의 *대출 심사*에 사용
- 카드사 마감 시간: 오전 9시
```

**OOM 발생 시 영향 사슬**:
```
배치 실패 (OOM)
  ↓
KCB로 전문 파일 미전송
  ↓
KCB → 은행 정산 데이터 누락
  ↓
셀러 대출 심사 불가
  ↓
셀러 자금 흐름 직접 차단 (재고 매입/직원 급여 등 차질)
  ↓
셀러 사업 운영 차질 → 플랫폼 신뢰 추락, 클레임/이탈
```

**임시방편(`-Xmx`만 증설)의 위험**:
- 거래량은 매일 *증가하는 특성* → 메모리만 늘려도 또 터질 시한폭탄
- 재발 시 셀러 신뢰와 플랫폼 평판 동시 추락
- 새벽 호출, 운영 부담 누적

**5년차 답변 톤**:
> "예를 들어, 제가 운영했던 정산 전문 파일 생성 배치는 매일 KCB로 데이터를 보내고 KCB가 은행에 전달해서 셀러 대출 심사에 사용됩니다. 만약 OOM이 재발해 STW가 길어지거나 배치가 실패하면, 셀러들이 그날 대출을 못 받아 *자금 흐름이 직접 막히는* 치명적 영향이 생깁니다. 그래서 `-Xmx`만 늘리는 건 *언제 또 터질지 모르는 시한폭탄*이고, 근본 원인 분석이 반드시 선행돼야 합니다."

> **면접 예상 질문:** OOM 장애의 비즈니스 임팩트를 본인 도메인 예시로 설명하라. 임시방편의 위험을 *고객 관점* 으로 어떻게 설명할 것인가?

## 학습 정리

- **OOM의 3가지 원인**(누수/워크로드 과다/힙 부족)은 처방이 완전히 다르므로 *원인부터 진단* 후 처방. `-Xmx` 증설은 ③번에만 정답.
- **그래프 패턴 매칭**으로 원인을 진단한다. 패턴 A(저점 우상향)는 누수, 패턴 B(촘촘)는 워크로드 과다(시점성) 또는 힙 부족(만성)으로 세분화.
- **참조의 본질은 "객체 주소"**. Stack(주소) ↔ Heap(객체 실체). 일반 지역변수도 *메서드 실행 중*에는 GC Root지만, 메서드 종료 → Stack frame 소멸 → *다음 GC* 때 회수.
- **진짜 캐시 = evict 메커니즘**. `static HashMap`은 캐시가 아니라 영구 저장소. Caffeine은 `static`이어도 evict로 entry 참조를 놓아주고 GC가 회수하는 협력 구조.
- **STW**는 GC 시 모든 스레드 정지. Major GC가 길어지면 응답 지연 → 타임아웃 → 헬스체크 실패 → ALB 인스턴스 제외 → 연쇄 장애.
- **5년차 답변 구조 5영역**: 흐름의 연결성 / 비즈니스 가치(고객·매출) / 팀 관점(데이터 기반 합의·문서화) / 측정과 검증(도구·지표) / Trade-off 이해.
