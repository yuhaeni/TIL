# 동기화 기법

> 날짜: 2026-04-01

## 내용

### 뮤텍스 락

하나의 공유 자원에 대해 lock이 걸려있으면 다른 프로세스가 접근하지 못하게 하는 동기화 도구이다.

- 자물쇠 역할: 임계 구역 진입 시 lock, 빠져나올 때 unlock
- 공유 자원이 **1개**일 때 사용

---

### 세마포

여러 공유 자원에 대해 wait과 signal로 접근을 제어하는 동기화 도구이다.

- 공유 자원이 **여러 개**일 때 사용
- S(세마포 변수): 사용 가능한 자원의 개수
- wait(): S를 1 감소시키고, S가 0 이하이면 대기
- signal(): S를 1 증가시키고, 대기 중인 프로세스를 깨움

**세마포 오용 시 문제**: signal을 먼저 호출하고 wait을 나중에 호출하면 S 값이 실제보다 부풀려져서 임계 구역에 들어가면 안 되는 프로세스가 들어갈 수 있다. 즉, **상호 배제가 깨진다.**

---

### 바쁜 대기 vs 대기 큐

| 방식 | 동작 | 문제점 |
|---|---|---|
| **바쁜 대기** | 사용 가능한 자원이 있는지 반복적으로 확인 | CPU 주기를 낭비 |
| **대기 큐** | 자원이 없으면 큐에 넣어 대기, 자원이 생기면 깨움 | - |

실무에서는 **대기 큐 방식**을 사용한다. CPU가 다른 작업에 할당되면 더 효율적인데, 바쁜 대기는 확인하는 데에 CPU를 계속 써버리기 때문이다.

---

### 데드락 (교착 상태)

두 프로세스가 서로 필요한 자원에 락이 걸려있어서 사용하지 못하고 무한정 기다리는 상태이다.

- 예: 서비스 A는 자원 1의 락을 잡고 자원 2를 기다리고, 서비스 B는 자원 2의 락을 잡고 자원 1을 기다림
- 예방법: 자원을 잡는 순서를 동일하게 맞추면 방지할 수 있다
- (데드락은 추후 학습 예정)

---

### 모니터

세마포는 사용자가 직접 wait()과 signal()을 구현해야 하지만, 모니터는 **상호 배제를 자동으로 보장**하는 동기화 도구이다.

- **상호 배제를 위한 큐**: 하나의 프로세스만 공유 자원에 접근할 수 있게 자동 보장
- **조건 변수에 대한 대기 큐**: 실행 조건이 만족되지 않은 프로세스는 대기시키고, 조건이 충족되면 signal로 깨워서 실행 순서를 제어
- 인터페이스를 통해서만 공유 자원에 접근하게 하여 **개발자의 실수를 방지**

---

## 면접 예상 질문

- **Q1.** DB 커넥션 풀의 동작 방식은 뮤텍스 락과 세마포 중 어느 쪽에 더 가까울까?
  - DB 커넥션 풀이 최대 10개 연결을 허용하는 건 공유 자원이 여러 개인 상황이므로 **세마포**에 가깝다. S의 초기값은 10이고, 커넥션을 하나 가져갈 때마다 S가 1씩 감소한다.

- **Q2.** 커넥션이 전부 사용 중일 때, 바쁜 대기 방식과 대기 큐 방식 중 실무에서는 어느 쪽을 쓸까?
  - **대기 큐 방식**을 사용한다. 바쁜 대기는 CPU 주기를 낭비하기 때문이다.

- **Q3.** 서비스 A는 자원 1의 락을 잡고 자원 2를 기다리고, 서비스 B는 자원 2의 락을 잡고 자원 1을 기다리면 어떤 문제가 발생할까?
  - **데드락(교착 상태)**이 발생한다. 예방법은 자원을 잡는 순서를 동일하게 맞추는 것이다.

- **Q4.** 세마포에서 signal을 먼저 호출하고 wait을 나중에 호출하면 어떤 문제가 생길까?
  - S 값이 실제보다 부풀려져서 **상호 배제가 깨진다.**

- **Q5.** 모니터가 상호 배제를 자동으로 보장하는 방식은 세마포와 어떤 점이 다를까?
  - 세마포는 직접 wait/signal을 구현해야 하지만, 모니터는 상호 배제를 위한 큐와 조건 변수 대기 큐를 통해 자동으로 보장한다. 인터페이스를 통해서만 접근하게 하여 개발자의 실수를 방지한다.
