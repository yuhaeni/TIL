# 프로세스 동기화

> 날짜: 2026-03-31

## 내용

### 레이스 컨디션 (Race Condition)

멀티스레드 환경에서 여러 스레드가 공유 자원에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 현상이다.

- 예: 재고가 1개 남았는데 사용자 A와 B가 동시에 구매하면, 재고가 0개가 되기 전에 각각 주문이 완료되어 **오버셀링** 발생
- `sum++` 같은 한 줄 코드도 저급 언어로 변환하면 여러 단계(읽기 → 증가 → 쓰기)로 나뉘기 때문에, 중간에 컨텍스트 스위칭이 끼어들면 레이스 컨디션이 발생할 수 있다

> **면접 예상 질문:** 멀티스레드 환경에서 레이스 컨디션이 발생하면 어떤 비즈니스 문제가 생길 수 있는가?

---

### 원자적 연산 (Atomic Operation)

더 이상 쪼갤 수 없는 하나의 연산으로, 중간에 컨텍스트 스위칭이 끼어들 수 없다.

- atom = "더 이상 쪼갤 수 없는"이라는 뜻
- `sum++`을 원자적 연산으로 만들면 읽기 → 증가 → 쓰기가 하나의 단위로 실행됨
- 임계 구역에 동시 접근하는 문제를 방지할 수 있다

> **면접 예상 질문:** `sum++`은 한 줄인데 왜 스레드 안전하지 않은가? 이를 해결하려면 어떻게 해야 하는가?

### 동기화의 두 가지 종류

프로세스 동기화에는 **실행 순서 제어**와 **상호 배제**, 두 가지가 있다.

| 동기화 종류 | 적용 상황 | 예시 |
|---|---|---|
| **실행 순서 제어** | 프로세스 간 실행 순서가 중요한 경우 | 큐가 비어있으면 소비자는 생산자가 넣을 때까지 대기해야 함 |
| **상호 배제** | 공유 자원에 동시 접근을 방지해야 하는 경우 | 두 스레드가 `sum`을 동시에 읽고 쓰는 상황 |

**생산자-소비자 문제**에서는 둘 다 필요하다:
- 생산자가 먼저 넣어야 소비자가 뺄 수 있으므로 **순서 제어**가 필요
- 공유 큐에 동시에 접근하면 안 되므로 **상호 배제**도 필요

> **면접 예상 질문:** 생산자-소비자 문제에서 실행 순서 제어와 상호 배제 중 어떤 동기화가 필요한가?

### 임계 구역의 세 가지 원칙

**임계 구역(Critical Section)**: 공유 자원에 접근하는 코드 영역. 동시에 실행되면 문제가 생기는 구간이다.

임계 구역 문제를 해결하려면 세 가지 원칙을 **모두** 지켜야 한다:

1. **상호 배제(Mutual Exclusion):** 한 프로세스가 임계 구역에 있으면 다른 프로세스는 못 들어감
2. **진행(Progress):** 임계 구역을 사용하는 프로세스가 없으면 다른 프로세스가 들어갈 수 있어야 함
3. **유한 대기(Bounded Waiting):** 임계 구역에 들어가려는 프로세스가 무한정 기다리면 안 됨

**셋 중 하나라도 빠지면 문제가 생긴다:**

- 상호 배제만 있으면 → 아무도 못 들어가는 상황 발생 가능
- 진행만 있으면 → 동시에 들어가버릴 수 있음
- 유한 대기 없으면 → 특정 프로세스가 **기아(Starvation)** 상태에 빠짐

> **면접 예상 질문:** 임계 구역의 세 가지 원칙이 모두 필요한 이유는? 하나라도 빠지면 어떤 문제가 생기는가?
