# 교착 상태

> 날짜: 2026-04-06

## 내용

### 교착 상태 발생 조건

교착 상태는 다음 4가지 조건이 **모두** 만족할 때 발생한다.

- **상호 배제**: 한 프로세스가 자원을 사용 중이면 다른 프로세스는 접근 불가
- **점유와 대기**: 자원을 이미 점유한 채로 다른 자원을 기다림
- **비선점**: 프로세스가 자원을 스스로 놓기 전까지 빼앗을 수 없음
- **원형 대기**: 프로세스들이 원형으로 서로의 자원을 대기

실무에서 깨뜨리기 가장 쉬운 조건은 **원형 대기**이다. 자원에 번호를 매기고 오름차순으로만 요청하면 원형 고리가 끊어진다. 식사하는 철학자 문제에서 "작은 번호 포크부터 집기" 규칙이 이에 해당한다.

나머지 조건을 깨뜨리기 어려운 이유:

| 조건 | 깨뜨릴 때의 부작용 |
|---|---|
| 상호 배제 | 임계 구역에 여러 프로세스 접근 → 데이터 일관성 깨짐 |
| 점유와 대기 | 자원을 한꺼번에 확보하거나 아예 안 받아야 함 → 기아 현상 |
| 비선점 | 작업 중 자원을 빼앗기면 미완성 작업/데이터 손상 |

> **면접 예상 질문:** 교착 상태 발생 조건 4가지를 설명하고, 실무에서 깨뜨리기 가장 쉬운 조건은?

### 자원 할당 그래프와 교착 상태

- 자원 할당 그래프에서 **cycle이 있으면** 교착 상태가 발생할 **수** 있다
- 같은 종류의 자원 인스턴스가 **여러 개**이면 다른 인스턴스를 할당하여 빠져나올 수 있으므로, cycle이 있어도 교착 상태가 아닐 수 있다

> **면접 예상 질문:** 자원 할당 그래프에서 cycle이 있어도 교착 상태가 아닌 경우는?

### 교착 상태 해결 방법

#### 1. 예방 (Prevention)

4가지 발생 조건 중 하나를 원천적으로 제거한다. 하지만 각 조건을 깨뜨리면 부작용이 크다.

- 점유와 대기를 없애면 자원을 많이 쓰는 프로세스가 동시에 모든 자원을 확보하기 어려워 **기아 현상** 발생
- 원형 대기를 깨뜨리는 것이 가장 현실적

> **면접 예상 질문:** 점유와 대기를 없애면 왜 기아 현상이 발생할 수 있는가?

#### 2. 회피 (Avoidance) — 은행원 알고리즘

- **안전 상태**: 모든 프로세스가 무사히 끝날 수 있는 안전 순서가 존재하는 상태
- **불안전 상태**: 안전 순서가 존재하지 않는 상태
- 자원 요청 시 안전 상태가 유지되면 허용, 불안전 상태가 되면 거부
- 안전 순서열의 핵심: 자원을 줬을 때 **작업을 완료하고 반환할 수 있는지** 여부
- **한계**: 매번 안전 상태 시뮬레이션 → 오버헤드, 프로세스의 최대 자원 요구량을 미리 알아야 함

> **면접 예상 질문:** 안전 상태와 불안전 상태의 차이는? 은행원 알고리즘이 실무에서 잘 쓰이지 않는 이유는?

#### 3. 검출 후 회복 (Detection & Recovery)

- **검출**: 대기 그래프(Wait-for Graph)에서 cycle 감지
- **회복 방법**:
  - 프로세스 강제 종료: 한꺼번에 종료(작업 내역 손실 큼) vs 하나씩 종료(매번 해소 확인 → 오버헤드)
  - 자원 선점: 비용이 적은 프로세스의 자원을 강제로 빼앗아 다른 프로세스에 할당 (= 비선점 조건 깨뜨리기)
  - 희생양(victim) 선택: 비용이 적은 트랜잭션을 롤백

> **면접 예상 질문:** DB에서 교착 상태를 검출하고 해결하는 방법은? 프로세스를 한꺼번에 종료하는 방식과 하나씩 종료하는 방식의 트레이드오프는?

#### 4. 무시 — 타조 알고리즘

- 교착 상태 발생 빈도나 심각성이 낮으면, 예방/회피/검출 비용이 더 클 수 있음
- OS가 아무것도 하지 않고 사용자가 직접 재시작/리부팅으로 해결
- 검출 후 회복과의 차이: 검출은 OS가 자동 감시/회복, 타조는 감시 오버헤드 자체가 없음

> **면접 예상 질문:** 타조 알고리즘과 검출 후 회복의 차이는? 교착 상태를 무시하는 것이 합리적인 상황은?
