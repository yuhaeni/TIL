# 연속 메모리 할당

> 날짜: 2026-04-07

## 내용

### 스와핑(Swapping)

메모리가 부족할 때 현재 실행되지 않는 프로세스를 보조기억장치의 스왑 영역으로 내보내고(스왑 아웃), 필요할 때 다시 메모리로 올리는(스왑 인) 기법이다.

스왑 인될 때 물리 주소가 이전과 달라져도 프로세스가 정상 실행되는 이유:
- 프로세스는 논리 주소를 사용하고, 물리 주소 변환은 OS가 담당
- 스왑 인 시 OS가 변환 테이블만 업데이트하면 되므로, 프로세스는 자신이 어디 적재됐는지 몰라도 문제없이 동작

**스와핑 과도 발생 시 문제:**

디스크는 메모리보다 수백~수천 배 느리므로, 스와핑이 과도하게 발생하면 디스크 I/O가 폭증하여 CPU는 놀고 디스크만 바쁜 I/O 병목 상태가 된다.

진단 방법: `free`, `top` 명령어로 Swap used 값 확인. Swap used가 지속적으로 증가한다면 메모리가 꽉 차서 스왑 인할 공간이 없다는 의미다.

> **면접 예상 질문:** 스와핑이 과도하게 발생하면 시스템 성능에 어떤 영향을 미치고, 원인을 어떻게 진단할 수 있는가?

### 메모리 배치 전략

프로세스를 메모리의 빈 공간에 적재할 때 어느 공간을 선택하느냐에 따라 세 가지 전략이 있다.

| 전략 | 방식 | 특징 |
|---|---|---|
| 최초 적합(First Fit) | 첫 번째로 발견한 충분한 공간에 배치 | 탐색 빠름 |
| 최적 적합(Best Fit) | 가장 작은 충분한 공간에 배치 | 작은 단편 발생 |
| 최악 적합(Worst Fit) | 가장 큰 공간에 배치 | 남은 공간이 큼 |

**최적 적합이 항상 효율적이지 않은 이유:**

가장 작은 공간에 배치하면 남은 조각이 아주 작아진다. 이 작은 조각들은 다른 프로세스를 적재할 수 없을 만큼 작아 외부 단편화를 오히려 악화시킨다. 또한 모든 빈 공간을 탐색해야 하므로 속도도 느리다.

**최악 적합이 최적 적합보다 나을 수 있는 이유:**

가장 큰 공간에 배치하면 남는 조각도 크기 때문에 다른 프로세스가 들어올 여지를 남긴다. 최적 적합이 쓸 수 없는 작은 조각을 만드는 것과 반대로, 외부 단편화가 적게 발생할 수 있다.

> **면접 예상 질문:** 최적 적합(Best Fit)이 항상 가장 효율적인 선택은 아닌 이유는? 최악 적합(Worst Fit)이 최적 적합보다 나을 수 있는 상황은?

### 외부 단편화와 메모리 압축

프로세스들이 메모리에 올라오고 내려가는 과정에서 빈 공간이 띄엄띄엄 생기면, 총 빈 공간은 충분해도 연속되지 않아 프로세스를 적재하지 못하는 **외부 단편화(External Fragmentation)** 문제가 발생한다.

**메모리 압축(Compaction):** 흩어진 빈 공간을 하나로 모으기 위해 프로세스들을 재배치하는 기법.

단점:
- 압축하는 동안 시스템을 멈춰야 함
- 프로세스를 옮기는 오버헤드가 큼

따라서 트래픽이 적은 새벽 시간대나 유지보수 윈도우처럼 시스템 부하가 낮을 때 수행하는 것이 적합하다.

> **면접 예상 질문:** 외부 단편화를 해결하는 메모리 압축의 단점과, 압축을 수행하기 적합한 상황은?

- 최초/최적/최악 적합 전략의 특징과 트레이드오프 비교
- 외부 단편화와 메모리 압축의 단점 및 활용 시점
