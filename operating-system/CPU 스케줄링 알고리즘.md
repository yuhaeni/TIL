# CPU 스케줄링 알고리즘

> 날짜: 2026-03-30

## 내용

### FCFS (선입 선처리) 스케줄링

준비 큐에 먼저 들어온 프로세스부터 순서대로 처리하는 비선점형 스케줄링이다.

- **호위 효과(Convoy Effect):** CPU를 오래 사용하는 프로세스가 먼저 실행되면, 뒤의 짧은 프로세스들이 오래 기다리게 되는 현상
- 백엔드 실무 비유: 하나의 스레드 풀에서 10초짜리 리포트 생성이 스레드를 점유하면, 뒤의 1ms짜리 가벼운 요청들이 무한 대기
- **실무 해결법:** 무거운 작업은 별도 스레드 풀이나 비동기 큐로 분리

> **면접 예상 질문:** FCFS에서 발생하는 호위 효과란 무엇이고, 실제 백엔드 서비스에서 어떤 상황과 비슷한가?

---

### 라운드 로빈 스케줄링

각 프로세스에 **타임 슬라이스(정해진 시간)**만큼 CPU를 할당하고, 시간이 지나면 다음 프로세스로 넘기는 선점형 스케줄링이다.

- **타임 슬라이스가 너무 크면:** 사실상 FCFS와 같아져서 호위 효과 발생
- **타임 슬라이스가 너무 작으면:** 문맥 교환이 자주 발생하여 오버헤드 증가

> **면접 예상 질문:** 라운드 로빈에서 타임 슬라이스가 너무 크거나 작으면 각각 어떤 문제가 생기는가?

---

### SJF (최단 작업 우선) 스케줄링

CPU 사용 시간이 가장 짧은 프로세스를 먼저 실행하는 방식이다. 이론적으로 평균 대기 시간을 최소화한다.

- **핵심 한계:** 프로세스가 CPU를 정확히 몇 ms 사용할지 미리 예측할 수 없음
- CPU 집중/입출력 집중 경향은 알 수 있어도, 정확한 실행 시간 예측은 불가능 → 실제 운영체제에서는 잘 사용하지 않음

### SRT (최소 잔여 시간 우선) 스케줄링

SJF와 라운드 로빈의 아이디어를 합친 스케줄링이다.

- SJF처럼 작업 시간이 짧은 프로세스를 우선하되, 라운드 로빈처럼 타임 슬라이스 단위로 실행
- 타임 슬라이스가 끝나고 다음 프로세스를 고를 때, **남은 작업 시간(잔여 시간)**이 가장 짧은 프로세스를 선택
- 포인트: "작업 시간"이 아니라 "**잔여** 시간" 기준

> **면접 예상 질문:** SJF가 이론적으로 최적인데 실제로는 왜 잘 안 쓰이는가? SRT는 SJF와 어떤 점이 다른가?

### 우선순위 스케줄링과 에이징

프로세스마다 우선순위를 부여하고, 우선순위가 높은 프로세스부터 실행하는 방식이다.

- **기아(Starvation):** 우선순위가 낮은 프로세스가 계속 밀려나 무기한 대기하는 현상
- **에이징(Aging):** 오래 대기한 프로세스의 우선순위를 점차 높여서 기아를 방지하는 기법
  - 설계 공식: `최종 우선순위 = 기본 우선순위 + (대기 시간 × 가중치)`
  - 예: 대기 시간 1초마다 우선순위를 1씩 증가

> **면접 예상 질문:** 우선순위 스케줄링에서 기아 현상을 해결하는 에이징 기법을 어떻게 설계할 수 있는가?

### 다단계 피드백 큐 (Multilevel Feedback Queue)

우선순위별로 여러 개의 큐를 두고, 프로세스가 큐 사이를 이동하는 스케줄링이다. 현대 운영체제에서 가장 많이 채택된 알고리즘이다.

**프로세스 자동 분류 원리:**

- **입출력 집중 프로세스:** CPU를 짧게 쓰므로 타임 슬라이스 안에 실행이 끝남 → 높은 우선순위 큐에 머무름
- **CPU 집중 프로세스:** 타임 슬라이스 안에 못 끝남 → 점점 낮은 우선순위 큐로 내려감

**다른 알고리즘의 한계를 모두 해결:**

| 한계 | 해결 방식 |
|------|-----------|
| SJF: 실행 시간 예측 불가 | 타임 슬라이스 안에 못 끝나면 자동으로 낮은 큐로 이동 → 예측 필요 없음 |
| 우선순위: 기아 현상 | 에이징 적용으로 무기한 대기 방지 |
| FCFS: 호위 효과 | 타임 슬라이스로 CPU 독점 방지 |

**백엔드 실무 적용:**

- 작업 큐 시스템에서 다단계 피드백 큐 아이디어를 적용할 수 있음
- 처음엔 모든 작업을 높은 우선순위 큐에 넣고, 타임아웃 내에 못 끝나면 낮은 우선순위 큐로 이동
- 빠른 작업(이메일 발송 등)은 높은 큐에서 바로 처리, 무거운 작업(리포트 생성 등)은 낮은 큐로 내려감
- 실무 예시: Sidekiq, Celery 같은 작업 큐 시스템에서 큐를 분리하는 패턴

> **면접 예상 질문:** 다단계 피드백 큐에서 CPU 집중/입출력 집중 프로세스 중 어느 쪽이 높은 우선순위 큐에 오래 머무는가? 현대 운영체제가 이 알고리즘을 가장 많이 채택한 이유는?

### 문맥 교환 (Context Switch)

현재 실행 중인 프로세스의 문맥을 저장하고, 다음 프로세스의 문맥을 복원하는 과정이다.

**저장/복원 대상:**

- **프로그램 카운터(PC):** 다음에 실행할 명령어의 메모리 주소
- **레지스터 값들:** 실행 중 계산하던 중간 값과 연산 결과
- **스택 포인터:** 현재 스택 위치

이 정보들은 **PCB(프로세스 제어 블록)**에 저장된다.

> **면접 예상 질문:** 문맥 교환이 일어날 때 구체적으로 어떤 것들이 저장되고 복원되는가?
