# 페이징을 통한 가상 메모리 관리

> 날짜: 2026-04-08

## 내용

### TLB와 캐시 히트율

페이지 테이블을 메모리에 두면 메모리 접근이 두 배로 늘어나는 문제가 생긴다. 이를 해결하기 위해 페이지 테이블의 일부를 캐시해두는 **TLB(Translation Lookaside Buffer)** 를 사용한다.

- **TLB 히트**: 논리 주소 → TLB에서 프레임 번호 조회 → 물리 주소 접근 (메모리 접근 1회)
- **TLB 미스**: 논리 주소 → 페이지 테이블 접근 → 프레임 접근 (메모리 접근 2회)

TLB는 참조 지역성에 근거해 페이지를 저장하므로, 메모리를 랜덤하게 점프하면서 접근하면 TLB 히트율이 구조적으로 낮아진다. 논리 주소를 순차적으로 접근하는 코드가 랜덤 접근 코드보다 TLB 히트율이 높으며, 배열을 순차 접근으로 짜는 것이 성능에 유리한 이유 중 하나다.

> **면접 예상 질문:** TLB 히트율이 낮아지는 상황은 어떤 경우이고, 실무에서 어떤 성능 문제로 이어질 수 있는가?

### 페이지 교체 전략 — 참조 비트와 수정 비트

메모리가 꽉 찬 상황에서 페이지 폴트가 발생하면, 기존 페이지를 내보내고 새 페이지를 적재해야 한다. 이때 어떤 페이지를 내보낼지 결정하는 기준이 **참조 비트**와 **수정 비트**이다.

- **참조 비트(Reference Bit)**: CPU가 해당 페이지에 접근하면 1로 설정. 0이면 최근에 접근된 적 없는 페이지
- **수정 비트(Dirty Bit)**: 페이지가 수정되면 1로 설정. 0이면 메모리 값과 보조기억장치 값이 일치

**효율적인 교체 순서:**

1. 참조 비트가 0인 페이지를 먼저 찾는다 (덜 중요한 페이지)
2. 그 중에서 수정 비트가 0인 페이지를 우선 내보낸다
   - 수정 비트 0 → 보조기억장치에 별도 기록 없이 바로 내보낼 수 있음
   - 수정 비트 1 → 변경된 값을 보조기억장치에 기록하는 추가 작업 필요 → 비효율적

> **면접 예상 질문:** 메모리가 꽉 찬 상황에서 페이지를 교체할 때, 참조 비트와 수정 비트 중 어느 쪽을 먼저 고려해서 내보낼 페이지를 고르는 게 효율적인가?

### 페이지 크기 트레이드오프

| | 페이지 크기 크게 | 페이지 크기 작게 |
|---|---|---|
| 내부 단편화 | 많이 발생 | 적게 발생 |
| 페이지 테이블 크기 | 작아짐 | 커짐 |
| TLB 히트율 | 높아짐 | 낮아짐 |
| 유리한 상황 | 대용량 순차 접근 | 작은 프로세스가 많은 경우 |

- **크게 설정**: 같은 페이지 안을 순차적으로 접근하므로 TLB에 올린 페이지를 반복 활용 가능 → TLB 히트율 증가
- **작게 설정**: 내부 단편화가 적어 작은 프로세스들이 낭비 없이 적재될 가능성이 높아짐

> **면접 예상 질문:** 페이지 크기를 크게 설정하면 유리한 상황과 작게 설정하면 유리한 상황은 각각 언제인가?

### 보호 비트와 코드 인젝션

페이지 테이블의 각 엔트리에는 해당 페이지에 허용된 연산을 나타내는 **보호 비트(Protection Bit)** 가 있다. `r`(읽기), `w`(쓰기), `x`(실행) 세 비트로 구성된다.

코드 영역에 쓰기(w) 권한을 막는 이유: 코드 영역에는 프로그램의 명령어가 저장되어 있는데, 쓰기 권한이 열려 있으면 공격자가 명령어를 바꿔써서 프로그램이 의도와 다르게 동작하게 만들 수 있다. 이것이 **코드 인젝션(Code Injection)** 공격의 원리이다. 따라서 코드 영역은 읽기(r)와 실행(x)만 허용하고 쓰기(w)는 차단한다.

> **면접 예상 질문:** 실무에서 코드 영역에 실행(x) 권한은 주되 쓰기(w) 권한을 막는 이유는?

### 페이지 폴트 빈발과 스래싱

**페이지 폴트(Page Fault):** 유효 비트가 0인 페이지에 접근할 때 발생. OS가 보조기억장치에서 해당 페이지를 메모리로 가져온다.

페이지 폴트가 자주 발생하면 CPU는 빠른데 보조기억장치에 자주 접근하게 되어 CPU 효율성이 급격히 떨어진다. 이처럼 페이지 교체가 너무 자주 일어나 실제 작업보다 페이지 스와핑에 더 많은 시간을 쓰는 상태를 **스래싱(Thrashing)** 이라고 한다.

근본 원인: 프로세스 수 대비 메모리 크기가 작아서, 메모리에 올릴 수 있는 페이지 수보다 필요한 페이지가 훨씬 많은 상태.

> **면접 예상 질문:** 페이지 폴트가 자주 발생하면 성능에 어떤 영향이 있고, 근본적인 원인은 무엇인가?

## 학습 정리

- TLB의 역할과 히트율이 낮아지는 원인 이해 (랜덤 접근 vs 순차 접근)
- 참조 비트와 수정 비트를 활용한 효율적인 페이지 교체 순서 파악
- 페이지 크기에 따른 내부 단편화 / TLB 히트율 트레이드오프 비교
- 코드 영역의 쓰기 권한 차단과 코드 인젝션 공격의 연관성
- 페이지 폴트 빈발이 스래싱으로 이어지는 과정
