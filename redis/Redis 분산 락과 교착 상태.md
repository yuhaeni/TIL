# Redis 분산 락과 교착 상태

> 날짜: 2026-04-19

## 내용

### 경쟁 조건(Race Condition)과 동기화

두 요청이 거의 동시에 같은 Redis 키(또는 공유 자원)에 접근하면 데이터 정합성 문제가 발생한다.

```
요청 A: 값 읽기 (10) → 처리 → 저장 (11)
요청 B: 값 읽기 (10) → 처리 → 저장 (11)
→ A의 변경 사항이 B에 의해 덮어써짐! (사실상 +1만 반영)
```

이처럼 여러 주체가 같은 자원에 동시에 접근해서 결과가 실행 순서에 따라 달라지는 현상을 **경쟁 조건(Race Condition)** 이라고 한다.

해결책은 **동기화(Synchronization)** 다. 한 요청이 처리되는 동안 다른 요청은 **대기**하게 만든다.

### 분산 락(Distributed Lock)과 Redis

단일 JVM이라면 `synchronized`, `ReentrantLock`으로 해결 가능하지만, **여러 서버가 같은 자원에 접근하는 분산 환경**에서는 불가능하다.

→ 공통으로 접근 가능한 저장소에 락 정보를 두어야 한다. **Redis**가 대표적인 선택지.

```
서버 1 ─┐
서버 2 ─┼─→ [Redis] ← 락 획득 시도
서버 3 ─┘         key: "order:123", value: "uuid"
```

Redis가 분산 락에 적합한 이유:
- **단일 스레드**로 명령을 순차 처리 → 원자성 보장
- `SETNX`(Set if Not Exists) 명령으로 "락 획득" 연산 구현 가능
- 인메모리라 응답이 빠름

> **면접 예상 질문:** 분산 환경에서 동기화를 어떻게 구현하는가? Redis가 분산 락에 적합한 이유는?

### 교착 상태(Deadlock)와 TTL

Redis로 락을 걸었는데 **락을 쥔 서버가 갑자기 죽으면** 어떻게 될까?

- 락을 해제할 주체가 사라짐 → 락이 영원히 풀리지 않음
- 서버 다운뿐 아니라 **예외 발생, 네트워크 끊김** 등도 같은 상황
- 다른 모든 요청이 해당 자원에 영원히 접근 불가

→ 이 상태가 바로 **교착 상태(Deadlock)**.

**해결책: TTL(Time To Live) 적용**

락에 만료 시간을 걸어두어, 서버가 죽어도 일정 시간이 지나면 자동으로 해제되게 한다.

```
SET lock:order:123 "uuid" NX EX 30  # 30초 TTL
```

> **면접 예상 질문:** 분산 락에서 교착 상태는 왜 발생하는가? TTL이 교착 상태를 어떻게 해결하는가?

### TTL 트레이드오프

TTL을 설정하는 것 자체가 또 다른 딜레마를 만든다.

| 설정 | 문제 |
|---|---|
| TTL이 너무 **짧음** | 작업이 끝나기 전 락 조기 해제 → 다른 스레드 접근 → **Race Condition 재발** |
| TTL이 너무 **긺** | 서버 다운 시 장시간 락 유지 → 대기 시간 증가 → **처리량(throughput) 저하** |

작업 시간은 네트워크 상태, DB 부하, 외부 API 응답 등에 따라 가변적이라 **고정 TTL로는 최적값을 찾기 어렵다.**

### Redisson 워치독(Watchdog) 메커니즘

**Redisson** 라이브러리는 TTL 딜레마를 **워치독(Watchdog)** 으로 해결한다.

```
락 획득 → 워치독 스레드 기동
   ↓
작업 처리 중... (예: 20초 소요)
   ↓
워치독이 주기적으로 TTL 연장 (기본: 10초마다, 30초로 재설정)
   ↓
작업 완료 → 락 해제 + 워치독 종료
```

**핵심 이점:**
- 처리 중에는 락이 풀리지 않음 → **락 조기 해제 문제 해결**
- 서버가 죽으면 워치독도 함께 죽음 → TTL이 자연스럽게 만료 → **교착 상태 회피**

**두 가지 문제를 한 번에 해결**하는 것이 워치독의 진짜 가치다.

```java
RLock lock = redissonClient.getLock("order:123");
lock.lock(); // 기본 30초 TTL + 워치독 자동 연장
try {
    // 비즈니스 로직
} finally {
    lock.unlock();
}
```

**주의:** `lock(long leaseTime, ...)`처럼 leaseTime을 명시적으로 지정하면 워치독이 동작하지 않는다. 워치독은 leaseTime을 지정하지 않았을 때만 활성화된다.

### 전체 흐름 정리

```
Race Condition
   ↓ 해결 필요
동기화(Synchronization)
   ↓ 분산 환경
분산 락(Distributed Lock) — Redis
   ↓ 서버 다운 시
교착 상태(Deadlock)
   ↓ 해결책
TTL 적용
   ↓ TTL 딜레마
짧으면: 락 조기 해제 / 길면: 대기 시간 증가
   ↓ 실무 해결
Redisson 워치독(Watchdog) → TTL 자동 연장
```

## 학습 정리

- **Race Condition**: 여러 주체가 동시에 공유 자원에 접근해 결과가 순서에 따라 달라지는 현상 → **동기화**로 해결
- 분산 환경에서는 공통 저장소(Redis)에 락 정보를 두는 **분산 락** 필요
- Redis 분산 락은 락 보유자가 죽으면 **교착 상태** 발생 → **TTL**로 자동 해제
- TTL 트레이드오프: 짧으면 조기 해제, 길면 처리량 저하
- **Redisson 워치독**은 작업 중에는 TTL 연장, 서버 다운 시 자동 만료되어 두 문제를 동시에 해결
