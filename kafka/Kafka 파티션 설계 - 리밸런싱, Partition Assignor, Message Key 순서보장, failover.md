# Kafka 파티션 설계 — 리밸런싱, Partition Assignor, Message Key 순서보장, failover

> 날짜: 2026-07-21

## 내용

파티션이 1개인 토픽을 여러 개로 새로 구성하려 할 때, "몇 개로 나눌까"부터 "누가 나눠주는가", "순서는 어떻게 지키는가"까지 이어지는 설계 흐름을 정리한다.

### 파티션 개수 산정 — 처리량 공식과 컨슈머 수의 배수

파티션 개수를 정하는 출발점은 **처리량**이다.

```
최소 파티션 수 ≈ (목표 처리량) ÷ (컨슈머 1개의 처리량)
```

들어오는 속도가 소비 속도보다 빠르면 **랙(lag)** 이 쌓이므로, 컨슈머 1개가 못 따라가면 파티션을 늘려 병렬 소비해야 한다.

여기에 **컨슈머 수의 배수**라는 기준을 더한다.

| 파티션 : 컨슈머 | 분배 결과 | 문제 |
|---|---|---|
| 10 : 3 | 4 / 3 / 3 | 한 컨슈머만 과부하 (skew) |
| 12 : 3 | 4 / 4 / 4 | 균등 분배 ✅ |

파티션 수를 컨슈머 수의 배수로 잡으면 부하가 골고루 나뉘어 **데이터 쏠림(skew)** 을 막을 수 있다. 또한 파티션은 **늘리기만 되고 줄이기 어렵기** 때문에, 미래 확장을 고려해 약간 여유를 두되 **과다하게 잡지는 않는다**(브로커 부담·리밸런싱 비용 ↑).

> **면접 예상 질문:** 파티션 개수는 무엇을 기준으로 정하는가? 파티션 수를 컨슈머 수의 배수로 잡으면 좋은 이유는?

### 리밸런싱(Rebalancing) — 파티션 재분배와 그 비용

컨슈머 그룹에 컨슈머가 **새로 들어오거나 죽으면**, 파티션을 컨슈머들에게 다시 나눠줘야 한다. 이 재분배 과정이 **리밸런싱(Rebalancing)** 이다.

- **Group Coordinator**: 컨슈머 그룹을 관리하는 **브로커**. 그룹 멤버십과 리밸런싱을 조율한다.
- **Group Leader**: 컨슈머 그룹 내에서 실제 **분배 계산을 실행하는 컨슈머 하나**.

**리밸런싱 비용:** (예전 eager 방식에선) 리밸런싱이 일어나는 잠깐 동안 그룹 전체가 소비를 멈춘다(stop-the-world). **파티션이 많을수록** 재분배 계산량과 멈춤 시간이 커진다 → 파티션을 무작정 크게 잡으면 안 되는 이유 중 하나.

> **면접 예상 질문:** 리밸런싱은 언제 발생하는가? Group Coordinator와 Group Leader의 역할 차이는? 파티션이 많으면 리밸런싱에 어떤 영향이 있는가?

### Partition Assignor — 파티션 할당 전략

실제로 "어느 컨슈머가 어느 파티션을 맡을지" 규칙을 실행하는 부품이 **Partition Assignor**다. (`partition.assignment.strategy` 설정)

| 전략 | 특징 |
|---|---|
| `RangeAssignor` | 토픽별로 파티션을 범위로 나눠 할당 (토픽 수 적으면 skew 발생 가능) |
| `RoundRobinAssignor` | 전체 파티션을 라운드로빈으로 골고루 분배 |
| `StickyAssignor` | 균등 분배 + 리밸런싱 시 기존 할당 최대한 유지 |
| `CooperativeStickyAssignor` | Sticky + **점진적(incremental) 리밸런싱** — 전체 멈춤 없이 일부만 재조정 |

핵심 흐름: **Coordinator**가 리밸런싱을 트리거 → **Group Leader**가 지정된 **Assignor**로 분배안 계산 → 각 컨슈머에 통보.

> **면접 예상 질문:** Partition Assignor의 종류와 차이는? CooperativeStickyAssignor가 기존 방식보다 나은 점은?

### Message Key와 Partitioner — 순서 보장

Kafka는 **같은 파티션 안에서만** 메시지 순서(오프셋 순서)를 보장한다. 따라서 같은 주문의 `생성 → 취소` 이벤트가 서로 다른 파티션으로 흩어지면 순서가 깨진다.

이를 막으려면 **같은 엔티티의 이벤트는 항상 같은 파티션으로** 보내야 하고, 이때 사용하는 것이 **Message Key(Record Key)** 다.

```
파티션 번호 = hash(Key) % 파티션 개수
```

- 파티션을 결정하는 부품은 **Partitioner** (기본: `DefaultPartitioner`)
- **Key 있음** → `hash(Key) % 파티션수` 로 동일 Key는 동일 파티션
- **Key 없음** → 라운드로빈으로 골고루 분산

주문 이벤트를 보낼 때 Key에 **주문 ID**를 넣으면, 같은 주문의 이벤트들이 같은 파티션으로 모여 **순서가 보장**된다.

> **면접 예상 질문:** Kafka에서 메시지 순서는 어느 범위까지 보장되는가? 특정 엔티티의 이벤트 순서를 지키려면 어떻게 하는가? Message Key가 없으면 파티션은 어떻게 정해지는가?

### 컨슈머 > 파티션 — idle 컨슈머와 failover

파티션이 3개인데 컨슈머를 5개 띄우면, 파티션 3개는 컨슈머 3개가 하나씩 맡고 **나머지 2개는 파티션을 못 받아 idle(대기)** 상태가 된다.

```
컨슈머 그룹 내 유효 병렬성의 상한 = 파티션 개수
→ 파티션보다 컨슈머를 많이 띄워도 소용없다
```

하지만 idle 컨슈머가 완전히 쓸모없는 건 아니다. 일하던 컨슈머 하나가 **죽으면 리밸런싱**이 일어나고, 대기하던 컨슈머가 그 **파티션을 통째로 넘겨받아** 마지막 커밋된 **오프셋 다음부터** 처리를 이어간다.

즉, 여분 컨슈머는 평소엔 놀지만 **장애 시 즉시 투입되는 대기 인력(failover, 고가용성)** 역할을 한다. (오프셋이 "어디까지 처리했는지"를 기록해두기에 이어받기가 가능하다.)

> **면접 예상 질문:** 파티션보다 컨슈머가 많으면 어떻게 되는가? 남는 컨슈머는 아무 쓸모가 없는가? 죽은 컨슈머의 파티션은 어떻게 복구되는가?

## 학습 정리

- 파티션 개수 = **(목표 처리량 ÷ 컨슈머 1개 처리량)** 을 최소로, **컨슈머 수의 배수**로 잡아 균등 분배(skew 방지)
- 파티션은 늘리기만 되고 줄이기 어려움 → 여유는 두되 과다 금지(브로커 부담·리밸런싱 비용 ↑)
- 컨슈머 합류/탈퇴 시 **리밸런싱** 발생 — **Group Coordinator**(브로커)가 조율, **Group Leader**(컨슈머)가 계산
- **Partition Assignor**(Range/RoundRobin/Sticky/CooperativeSticky)가 실제 할당 규칙, Cooperative는 점진적 리밸런싱
- 순서는 **같은 파티션 안에서만** 보장 → **Message Key**(예: 주문 ID)로 `hash(Key) % 파티션수` → 동일 Key 동일 파티션
- 유효 병렬성 상한 = 파티션 수, 여분 컨슈머는 **failover 대기 인력**으로 오프셋 이후부터 이어받음
