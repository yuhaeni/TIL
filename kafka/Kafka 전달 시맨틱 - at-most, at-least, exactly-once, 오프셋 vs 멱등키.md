# Kafka 전달 시맨틱 — at-most-once, at-least-once, exactly-once, 오프셋 vs 멱등키

> 날짜: 2026-07-20

## 내용

Kafka에서 자주 나오는 용어(시맨틱, 오프셋, 멱등키, 브로커, 클러스터)를 정리한 기록. 특히 **시맨틱(semantic)** 은 "의미"라는 뜻이고, Kafka 문맥에서는 **메시지가 몇 번 전달되는지에 대한 보장(delivery guarantee)** 을 가리킨다.

### 전달 시맨틱(Delivery Semantics) — 유실/중복 2축으로 이해하기

메시지 전달을 택배 배송 정책에 비유하면 "최대 한 번만 보낸다 / 최소 한 번은 반드시 도착시킨다 / 정확히 딱 한 번만 도착시킨다" 중 하나를 고르는 것과 같다. Kafka 용어로는 아래 3가지다.

핵심은 **두 축**으로 보는 것: 유실(안 올 수 있나?) / 중복(여러 번 올 수 있나?)

| 시맨틱 | 유실 | 중복 | 실제 도착 횟수 |
|---|---|---|---|
| **at-most-once** | O | X | **0 또는 1번** |
| **at-least-once** | X | O | 1번 이상 |
| **exactly-once** | X | X | 딱 1번 |

- **at-most-once**: 재전송을 안 하므로 유실 가능 → **0번(아예 못 받음)** 도 있을 수 있다. 대신 중복은 없다.
- **at-least-once**: 유실은 막지만(재전송), 그 대가로 **중복**이 생길 수 있다.
- **exactly-once**: 유실도 중복도 없다. 대신 뒤에 나오듯 구현 비용이 **가장 크다.**

> 헷갈렸던 점: "정확히 한 번(exactly-once)"과 "최대 한 번(at-most-once)"이 같은 말 아닌가? → **아니다.** 결정적 차이는 **유실 가능 여부**. at-most-once는 0번이 될 수 있고(유실 O), exactly-once는 절대 유실되지 않는다.

> **면접 예상 질문:** Kafka의 3가지 전달 시맨틱을 유실/중복 관점에서 비교하고, at-most-once와 exactly-once의 결정적 차이를 설명하라.

---

### 시맨틱은 토픽 단위 옵션이 아니다 — 프로듀서 + 컨슈머 설정의 조합

흔한 오해: 토픽에 "시맨틱=exactly-once" 같은 옵션을 지정하면 끝이다? → **아니다.**

시맨틱은 **프로듀서 설정과 컨슈머 설정이 합쳐져 결정되는 결과**다. 그래서 데이터 흐름(파이프라인)마다 다르게 가져갈 수 있다.

- 프로듀서가 ack를 못 받았을 때 **재전송을 하도록** 두면 → 유실은 막지만 중복 발생 → at-least-once
- 프로듀서가 **재전송을 아예 안 하도록** 두면 → 유실 가능 → at-most-once
- 재전송 + 중복 제거 장치(멱등 프로듀서 + 트랜잭션)까지 얹으면 → exactly-once

> ACK(acknowledgment)는 "잘 받았어요"라는 확인 신호로, TCP의 ACK와 결이 같은 개념. Kafka에서는 컨슈머가 "어디까지 처리했는지" 표시하는 **오프셋 커밋**과도 연결된다.

> **면접 예상 질문:** 전달 시맨틱을 토픽 단위 설정 하나로 고정할 수 있는가? 시맨틱이 어떤 설정들의 조합으로 결정되는지 설명하라.

---

### 멱등성(Idempotency) — 왜 exactly-once 대신 "at-least-once + 컨슈머 멱등성"인가

**멱등성(idempotency)** = "같은 메시지가 여러 번 와도 결과는 한 번 처리한 것과 같게" 만드는 성질.

예를 들어 "결제 완료 알림은 딱 한 번만 가야 하니 exactly-once"라는 직관은 목표 자체는 맞다. 하지만 실무에는 트레이드오프가 있다.

- **exactly-once의 비용**: Kafka에서 트랜잭션 같은 무거운 장치가 필요해 **성능 비용이 크고 설정도 복잡**하다.
- **현업의 현실적 선택**: 프로듀서는 **at-least-once(중복 허용)** 로 두고, 대신 **컨슈머 쪽에서 중복을 걸러낸다.**
  - 장점: exactly-once보다 성능 비용이 작고, Kafka 트랜잭션 없이도 되어 상대적으로 단순하다.

즉, 프로듀서 단에서 무거운 exactly-once를 쓰는 대신, **컨슈머를 멱등하게** 만들면 같은 효과를 더 낮은 비용으로 얻을 수 있다.

> **면접 예상 질문:** 결제/정산처럼 유실은 안 되고 중복은 허용 불가인 요구사항에서, exactly-once와 "at-least-once + 컨슈머 멱등성" 중 무엇을 택하고 왜인지 설명하라.

---

### 오프셋 vs 멱등키 — 중복 판단은 오프셋으로 할 수 없다

컨슈머가 "이 메시지 아까 처리했나?"를 판단하려면 각 메시지를 유일하게 구별할 표식이 필요하다. **오프셋으로는 안 된다.**

- **오프셋(offset)**: 브로커가 파티션 로그에 메시지를 **저장하는 순간 순서대로 매기는 위치 번호**. (연못에 돌을 하나씩 던지면 1번, 2번, 3번 자리가 생기는 것과 같다.)
- 오프셋은 **프로듀서가 매기는 게 아니다.** 프로듀서는 메시지를 던지기만 하고, 번호는 브로커가 붙인다. → "프로듀서 입장의 오프셋" 같은 건 없다.
- 그래서 프로듀서가 **재전송**하면 브로커에 **두 번 저장** → **서로 다른 오프셋 2개**가 생기고, 컨슈머는 이를 별개 메시지 2개로 받는다. **내용이 같아도 오프셋은 다르다.** → 오프셋으로는 중복 판별 불가.

대신 **메시지 내용 자체에 심어두는 유일한 표식**을 쓴다:

- **멱등키(idempotency key)** 또는 **고유 메시지 ID(unique message ID / event ID)**
- 예: 결제 알림마다 `payment-id: 12345` 를 심어두면, 컨슈머가 "12345는 아까 처리했네?" 하고 걸러낸다.

처리한 ID를 **어디에 기록**하느냐 (앞서 멱등성 학습과 연결):

| 방법 | 방식 | 특징 |
|---|---|---|
| **DB 테이블 + Unique 제약** | 이미 있으면 중복 → INSERT 실패로 감지 (Inbox 패턴) | 영구 보장, 가장 안전 |
| **Redis + TTL** | `SETNX`로 "처리함" 표시 후 일정 시간 뒤 자동 삭제 | 빠름, 단기 방어선 |

> **면접 예상 질문:** 컨슈머 중복 제거를 오프셋으로 하면 안 되는 이유는? 멱등키를 무엇으로 잡고 처리 이력을 어디에 저장할지 트레이드오프와 함께 설명하라.

### Kafka 구성요소 — 프로듀서 / 브로커 / 컨슈머, 그리고 클러스터

시맨틱을 이해하려면 오프셋을 "누가 언제" 매기는지 알아야 하고, 그러려면 구성요소 구분이 필요하다.

- **프로듀서(Producer)**: 메시지를 보내는 쪽 (택배 보내는 사람)
- **브로커(Broker)**: 메시지를 실제로 저장하는 **Kafka 서버 그 자체** (택배 보관 창고). 오프셋은 여기서 매겨진다.
- **컨슈머(Consumer)**: 메시지를 꺼내 읽는 쪽 (택배 받는 사람)

**클러스터(Cluster)**: 브로커 여러 대를 묶은 것 (창고 하나 = 브로커, 창고 여러 개를 묶은 물류망 전체 = 클러스터).

- 왜 여러 대로 묶나: 브로커 하나가 장애나도 다른 브로커에 **복제본(replica)** 이 있으면 데이터를 잃지 않는다.
- 또한 파티션을 여러 브로커에 나눠 담아 **부하를 분산**한다. → 파티션 설계와 직접 연결되는 개념.

> **면접 예상 질문:** Kafka에서 오프셋은 누가 매기는가? 브로커와 클러스터의 관계, 그리고 복제본(replica)이 시맨틱(유실 방지)에 어떤 역할을 하는지 설명하라.

## 학습 정리

- **시맨틱(semantic) = 메시지 전달 보장 방식**. Kafka엔 at-most-once / at-least-once / exactly-once 3가지.
- 시맨틱은 **유실/중복 2축**으로 보면 명확: at-most-once(0~1번, 유실O·중복X), at-least-once(1번+, 유실X·중복O), exactly-once(딱 1번).
- exactly-once와 at-most-once의 결정적 차이는 **유실 가능 여부**.
- 시맨틱은 토픽 단위 옵션이 아니라 **프로듀서 설정과 컨슈머 설정이 합쳐져** 결정된다.
- 결제/정산 등 현업에선 exactly-once의 비용 때문에 **at-least-once + 컨슈머 멱등성**을 더 흔히 쓴다.
- 중복 판단은 **오프셋으로 불가** (재전송 = 새 오프셋). **멱등키(고유 ID)** 로 판별하고, 처리 이력은 **DB Unique / Redis TTL** 에 기록.
- 오프셋은 **브로커**가 파티션 저장 시점에 매긴다. 브로커 여러 대 묶음 = **클러스터**, replica로 유실 방지 + 부하 분산.
