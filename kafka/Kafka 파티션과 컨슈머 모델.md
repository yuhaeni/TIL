하나의 파티션은 하나의 컨슈머(스레드)만 처리해야 한다.

여러 스레드가 같은 파티션을 동시에 읽으면:
- 메시지 처리 **순서 보장** 불가
- **중복 처리** 발생 가능
- **데이터 정합성** 깨짐

**파티션 수 = 최대 병렬 처리 수**이므로, 컨슈머가 파티션보다 많으면 남는 컨슈머는 idle(대기) 상태가 되어 리소스 낭비다.

| 비유 | Kafka |
|---|---|
| 큰 강 | 토픽 |
| 수로 | 파티션 |
| 수로에서 물 퍼가는 사람 | 컨슈머 |

처리량을 늘리고 싶을 때는 **파티션을 먼저 늘리고**, 파티션 수에 맞게 컨슈머를 늘려야 한다.

> **면접 예상 질문:** 같은 파티션을 여러 스레드가 동시에 처리하면 안 되는 이유는? 처리량을 늘릴 때 파티션과 컨슈머 중 어느 쪽을 먼저 고려해야 하는가?

### 오프셋(Offset)과 장애 복구

오프셋은 컨슈머가 파티션에서 어디까지 읽었는지 기록하는 값이다.

**장애 발생 시 흐름:**

1. 컨슈머가 메시지를 처리하다가 장애 발생
2. 재시작 후 마지막 **커밋된 오프셋**부터 다시 읽음
3. 메시지를 처리했지만 오프셋 커밋을 못 한 경우 → 같은 메시지를 다시 읽게 됨 (중복 처리 발생)

이것이 **at-least-once** 전달 보장 방식의 특성이다. 중복 처리를 허용하되, 처리 결과가 항상 동일하도록 **멱등성**을 보장해야 한다.

> **면접 예상 질문:** Kafka 컨슈머가 메시지 처리 중 장애가 발생하면 어떻게 되는가? 오프셋의 역할은?

### 멱등성(Idempotency)과 중복 처리 방지

**멱등성**: 같은 연산을 여러 번 수행해도 결과가 동일한 성질. (수학의 1×1×1 = 1처럼)

Kafka at-least-once 환경에서 중복 처리 방지 전략:

- **Redis TTL 활용**: 처리 완료된 메시지 ID를 Redis에 TTL과 함께 캐싱
- 재처리 시 Redis에서 처리 여부 확인 → 이미 처리했으면 스킵
- 영구적으로 보관할 필요 없는 일시적 중복 방지 데이터이므로 Redis TTL이 적합

```
메시지 수신
    ↓
Redis에서 처리 여부 확인
    ↓
처리된 경우 → 스킵 (멱등성 보장)
처리 안 된 경우 → 처리 후 Redis에 기록
```

> **면접 예상 질문:** Kafka 재처리 시 중복 처리를 어떻게 막을 수 있는가? 멱등성이란 무엇인가?

### 동시성(Concurrency) vs 병렬성(Parallelism)

"파티션 수 = 최대 병렬 처리 수"라고 말할 때, 정확히는 **동시성과 병렬성을 구분**해야 한다.

| 구분 | 동시성 (Concurrency) | 병렬성 (Parallelism) |
|---|---|---|
| 정의 | 여러 작업을 **번갈아가며** 처리 | 여러 작업을 **물리적으로 동시에** 처리 |
| 자원 | 단일 코어로도 가능 (스레드 컨텍스트 스위칭) | 멀티 코어/멀티 머신 필요 |
| 비유 | 한 명의 요리사가 여러 요리를 번갈아 만듦 | 여러 명의 요리사가 동시에 다른 요리를 만듦 |

**Kafka 컨슈머 관점:**
```
[싱글 코어 + 컨슈머 N개]
  → 동시성만 가능 (스레드 스위칭으로 번갈아 처리)
  → 파티션을 늘려도 진정한 병렬 처리는 X

[멀티 코어 / 멀티 서버 + 컨슈머 N개]
  → 진정한 병렬성 가능
  → 파티션 수만큼 동시 처리량 증가 ✅
```

**핵심:** 파티션을 늘려도 **컨슈머가 실행되는 환경(코어 수, 서버 수)이 받쳐주지 않으면** 병렬 처리 효과가 제한된다.

### 워크로드별 파티션 수 튜닝 — CPU Bound vs I/O Bound

파티션 수는 단순히 "많을수록 좋다"가 아니라 **워크로드 특성**에 맞춰야 한다.

| 워크로드 | 특징 | 적정 파티션 수 |
|---|---|---|
| **CPU Bound** | 연산 위주 (압축, 직렬화, 계산) | **≈ 코어 수** |
| **I/O Bound** | 외부 호출 위주 (DB, API, 네트워크) | 코어 수보다 적어도 OK / 더 많아도 가능 |

**CPU Bound:** 코어가 쉬지 않고 연산하므로 **코어 수보다 많은 파티션은 컨텍스트 스위칭 오버헤드**만 늘어난다.

**I/O Bound:** I/O 대기 동안 CPU가 놀기 때문에 **코어보다 많은 스레드/파티션이 유리** — Brian Goetz 공식이 이를 정량화한다.

**Brian Goetz 공식 (Java Concurrency in Practice):**
```
N_threads = N_cpu × U_cpu × (1 + W/C)

  N_cpu  = 코어 수
  U_cpu  = 목표 CPU 활용률 (0 ~ 1)
  W/C    = (대기 시간) / (계산 시간) 비율
```

**예시 — I/O 비율이 높을 때:**
- 코어 4개, U=1, W/C=3 (대기 75% / 계산 25%) → `4 × 1 × 4 = 16` 스레드
- 즉, 코어 4개여도 **파티션 16개까지 효과**가 있을 수 있음

**파티션 과다의 단점:**
- 브로커 메모리/파일 핸들 증가
- 리밸런싱 시간 증가
- 컨슈머 컨텍스트 스위칭 오버헤드

**튜닝 원칙:**
> "워크로드를 측정 → CPU vs I/O 비율 산정 → 공식으로 후보값 도출 → 부하 테스트로 검증."

> **면접 예상 질문:** CPU Bound와 I/O Bound 워크로드에서 파티션 수를 어떻게 다르게 잡는가? Brian Goetz 공식은 어떻게 적용하는가?

### Java 스레드 상태와 CPU 점유 — I/O 대기 시 CPU는 비어있다

I/O Bound에서 "스레드를 코어 수보다 많이 만들어도 된다"는 말이 헷갈리는 이유는 **"대기 중인 스레드도 CPU를 잡고 있는 것 아닌가?"** 라는 직관 때문이다. 정답은 **아니다.**

Java 스레드는 상태에 따라 CPU 점유 여부가 다르다.

| 상태 | CPU 점유 | 발생 시점 |
|---|---|---|
| `RUNNABLE` | ✅ 점유 | 실제 코드 실행 중 |
| `BLOCKED` | ❌ 비점유 | `synchronized` 락 대기 |
| `WAITING` | ❌ 비점유 | `Object.wait()`, `Thread.join()`, `LockSupport.park()` |
| `TIMED_WAITING` | ❌ 비점유 | `Thread.sleep()`, `Future.get(timeout)` |

**I/O 호출 시 실제로 일어나는 일:**

```
1. 스레드: socket.read() / DB 쿼리 / API 호출 등 I/O 시스템 콜 호출
2. OS 커널: I/O 요청을 네트워크/디스크 컨트롤러에 위임
3. 스레드 상태: RUNNABLE → BLOCKED/WAITING (스케줄러가 CPU에서 내려놓음)
4. CPU 코어: 비어있는 동안 다른 RUNNABLE 스레드 실행
5. I/O 완료 인터럽트 → 커널이 스레드를 RUNNABLE로 전환 → 스케줄러가 다시 CPU에 올림
```

**핵심:** I/O 대기 중인 스레드는 OS 스케줄러가 **CPU 큐에서 제외**하기 때문에, 코어가 비어있는 동안 다른 스레드가 실행될 수 있다. 그래서 I/O Bound 작업은 스레드 수를 코어 수보다 많이 잡아도 의미가 있다.

### 스레드 증가의 3가지 부작용 — 이론 최적값을 그대로 쓰면 안 되는 이유

Brian Goetz 공식으로 산출한 이론값(예: 40 스레드)을 그대로 적용하면 안 된다. 스레드는 **공짜 자원이 아니다**.

#### 1. 메모리 부담

| 자원 | 크기 (기본값) |
|---|---|
| 스레드 스택 (`-Xss`) | 1MB |
| TLAB (Thread Local Allocation Buffer) | 수십 KB |
| OS 레벨 스레드 메타데이터 | 소량 |

```
스레드 40개 × 1MB = 40MB (스택만)
컨슈머 파드 5개 운영 시 = 200MB
극단적으로 스레드 500개 → OOM: unable to create native thread
```

#### 2. 외부 API 입장의 부작용

40 스레드가 동시에 외부 API를 호출하면 **DDoS처럼 보인다**.

- HTTP 429 Too Many Requests
- IP 블랙리스트 / CAPTCHA
- 상대 서버 다운 → 본인이 가해자가 됨
- 스크래핑 환경에서는 봇 탐지(CloudFlare, Akamai)에 차단

→ Rate limiting, exponential backoff, 세션/쿠키 관리가 필수.

#### 3. DB 커넥션 풀 고갈 — 병목의 이동

```
[의도]
스레드 40개 → 외부 API 동시 호출 → 처리량 ↑

[실제]
스레드 40개 → API 응답 수신 ✅
            → DB 저장 시도
            → HikariCP 커넥션 풀 20개 다 점유됨 🚫
            → 나머지 20 스레드는 커넥션 대기 (BLOCKED)
            → HikariCP 타임아웃(기본 30초) → SQLException 발생
```

이 현상을 **"병목의 이동(bottleneck shift)"** 이라 한다. 한 자원의 병목을 풀면 다음 자원에서 막힌다.

> **면접 예상 질문:** I/O Bound 컨슈머에서 스레드를 이론값(예: Brian Goetz 공식 결과)대로 늘렸을 때 발생할 수 있는 부작용 3가지를 설명하라.

### HikariCP 커넥션의 실체 — TCP 소켓 + DB 프로세스 한 묶음

"HikariCP 커넥션을 늘리면 DB 부하가 왜 올라가는가?"라는 질문은 **커넥션의 실체**를 알아야 답할 수 있다.

#### TCP 소켓의 자원 구조

```
소켓 1개 생성 시 OS가 할당하는 자원:
├─ 파일 디스크립터 (FD)
├─ 송신 버퍼 (Send Buffer, 64~256KB)
├─ 수신 버퍼 (Receive Buffer, 64~256KB)
├─ 연결 상태 (TCP State Machine)
└─ 포트 번호
```

TCP는 연결 지향 프로토콜로 **3-way handshake**(SYN → SYN/ACK → ACK)로 연결을 수립한다. 매 쿼리마다 새로 맺으면 수 ms씩 낭비되므로, HikariCP가 미리 소켓을 열어두고 재사용한다.

#### PostgreSQL의 특수성 — 커넥션 1개 = OS 프로세스 1개

```
앱이 커넥션 1개 맺음
   ↓
PostgreSQL이 fork() 호출 → 백엔드 프로세스 1개 생성
   ↓
이 프로세스가 그 커넥션 전담
```

| 커넥션당 PostgreSQL이 잡는 자원 | 크기 |
|---|---|
| 프로세스 메모리 (`shared_buffers` 제외) | 5~10MB |
| `work_mem` (정렬/해시) | 4MB (쿼리당 여러 번 할당 가능) |
| `temp_buffers` | 8MB |
| OS 컨텍스트 스위칭 대상 | — |

**MySQL 비교:**

| | PostgreSQL | MySQL (InnoDB) |
|---|---|---|
| 커넥션 단위 | **프로세스** | 스레드 |
| 무게 | 무거움 (10MB+) | 가벼움 (1MB+) |
| 권장 `max_connections` | 100~200 | 1000+ |

PostgreSQL이 무거운 이유로 운영 환경에서는 **PgBouncer** 같은 외부 풀러를 추가로 둔다.

#### HikariCP 커넥션 = 3가지 자원 묶음

```
HikariCP 커넥션 1개:
├─ 앱 쪽: TCP 소켓 (송수신 버퍼 ~500KB)
├─ 네트워크: TCP 연결 상태 (라우터/방화벽도 추적)
└─ DB 쪽: PostgreSQL 백엔드 프로세스 (10MB)
```

→ HikariCP `maximumPoolSize`를 늘리면 **앱뿐 아니라 DB 서버에도 같은 수만큼 프로세스가 늘어난다**. 이게 DB CPU/메모리 경합으로 이어지고, 결국 **DB를 공유하는 다른 서비스의 응답 시간**까지 영향을 준다.

> **면접 예상 질문:** HikariCP 커넥션을 늘리면 DB 부하가 올라가는 이유는? PostgreSQL과 MySQL의 커넥션 처리 차이는?

### 시나리오 분석 — I/O Bound 컨슈머와 DB 커넥션 풀 충돌

**상황:** 코어 4개 서버, Brian Goetz 공식상 이론 최적 = 40 스레드, HikariCP 커넥션 풀 = 20개. 어떻게 조정할 것인가?

| 선택 | 액션 | 장점 | 단점 |
|---|---|---|---|
| **A** | 스레드를 20개로 줄여 풀에 맞춤 | 메모리/API/DB 안정, 즉시 적용 가능 | 이론 최적값 대비 처리량 ↓, 코어가 덜 활용됨 |
| **B** | 커넥션 풀을 40개로 늘림 | 스레드 풀 활용 ↑ | DB 프로세스 ↑ → 다른 서비스 응답 시간 영향, `max_connections` 한계 |
| **C** | API 호출 풀과 DB 저장 풀을 분리 | 자원별 독립 튜닝, 한쪽 장애가 다른 쪽 전파 X | 구현 복잡도 ↑, 중간 큐/버퍼 관리 필요, 메모리 ↑ |

**5년차 답변에 필요한 요소 (자가 진단 5영역):**

1. **흐름의 연결성** — 왜 그 선택인지 인과 설명
2. **비즈니스 가치** — 기술 언어를 사용자/매출 관점으로 변환
3. **팀 관점** — DB 공유 팀과 사전 합의, ADR 문서화, Runbook 작성
4. **측정과 검증** — 추측이 아닌 지표 기반 (다음 섹션)
5. **Trade-off** — 선택한 안의 단점도 명시

**기술 언어 → 비즈니스 언어 변환:**

| 기술 표현 | 비즈니스 표현 |
|---|---|
| DB 부하 감소 | 결제/조회 API 응답 시간 안정 |
| 커넥션 풀 고갈 방지 | 사용자 요청 실패율 ↓, 이탈률 ↓ |
| 쿼리 응답 시간 안정 | SLA(99% 1초 이내) 준수 |
| DB CPU 90% → 50% | 피크 시간대 신규 서비스 수용 가능 |

> **면접 예상 질문:** I/O Bound 컨슈머의 이론 최적 스레드 수가 DB 커넥션 풀 크기를 초과할 때, 어떤 선택지가 있으며 각각의 트레이드오프는?

### 모니터링 지표 — 측정 기반 튜닝

스레드 수가 적절한지 판단하려면 **측정**이 필요하다. PromQL 기준 핵심 지표:

| 지표 | 의미 | 위험 신호 |
|---|---|---|
| `kafka_consumer_lag` | 미처리 메시지 수 | 지속 증가 = 스레드 부족 (처리 속도 < 유입 속도) |
| `process_cpu_usage` | JVM CPU 사용률 | I/O Bound인데 < 30% = 스레드 더 늘려도 OK / > 90% = 컨텍스트 스위칭 과다 |
| `hikaricp_connections_active` | 사용 중 커넥션 | 풀 크기에 근접 = 포화 임박 |
| `hikaricp_connections_pending` | 커넥션 대기 스레드 | > 0이면 병목 발생 |
| `hikaricp_connections_timeout_total` | 커넥션 획득 타임아웃 | 0보다 크면 처리 실패 발생 |
| `rate(kafka_consumer_records_consumed_total[1m])` | 초당 처리량 | 스레드 늘렸는데 안 늘면 다른 곳에 병목 |
| `jvm_memory_used_bytes{area="heap"}` | 힙 메모리 사용 | 90% 근접 시 GC 압박 |
| `rate(jvm_gc_pause_seconds_sum[1m])` | GC pause 누적 | 스레드 ↑ 후 함께 증가하면 메모리 압박 |

**튜닝 사이클:**

```
1. 초기값(예: 코어 수)으로 시작 → 측정
2. Kafka lag 증가 시 스레드 증가
3. CPU/커넥션 풀/GC 지표 동시 모니터링
4. 어느 지표가 먼저 한계에 닿는지 확인 → 그게 진짜 병목
5. 병목 자원을 늘리거나, 풀 분리(C안)로 전환
```

→ 면접에서 **"측정 → 조정 → 재측정"** 패턴으로 답하면 실무 5년차 답변이 된다.

> **면접 예상 질문:** 컨슈머 스레드 수가 적정한지 어떻게 검증하는가? 어떤 지표를 보는가?

## 학습 정리

- 파티션 1개 = 컨슈머 1개 (순서 보장 + 데이터 정합성)
- 파티션 수 = 최대 병렬 처리 수 → 처리량 증가 시 파티션 먼저 늘리기
- at-least-once 방식에서 오프셋 커밋 실패 시 중복 처리 발생 → 멱등성으로 해결
- Redis TTL을 이용한 중복 처리 방지가 일시적 데이터에 적합
- 환경별 토픽 자동 매핑으로 수동 변경 실수 방지, 단 런타임 변경 불가 트레이드오프 존재
- **동시성 ≠ 병렬성** — 멀티코어/멀티서버 환경이 받쳐줘야 파티션 증가가 진정한 병렬 처리로 이어짐
- **CPU Bound는 ≈ 코어 수**, **I/O Bound는 Brian Goetz 공식(`N_cpu × U × (1 + W/C)`)**으로 산정
- 파티션 과다는 브로커 자원·리밸런싱·컨텍스트 스위칭 오버헤드를 키움 → 측정 + 부하 테스트 필수
- I/O 대기 중인 Java 스레드는 `BLOCKED/WAITING` 상태로 **CPU를 점유하지 않음** → 그래서 I/O Bound는 스레드를 코어보다 많이 잡아도 의미가 있음
- 스레드 증가는 **메모리(스택 1MB)·외부 API 차단·DB 커넥션 풀 고갈** 3가지 부작용을 동반 → 이론 최적값을 그대로 쓰면 안 됨
- HikariCP 커넥션 1개 = **TCP 소켓 + PostgreSQL 백엔드 프로세스(10MB)** 한 묶음 → 풀 크기 증가는 DB 부하 직결
- 시나리오별 선택지(A: 스레드 ↓, B: 풀 ↑, C: 풀 분리)는 **5영역 자가 진단**(흐름/비즈니스/팀/측정/Trade-off)으로 답변 보강
- 튜닝은 **측정 → 조정 → 재측정** 사이클: `kafka_consumer_lag`·CPU 사용률·HikariCP `active/pending`·GC pause를 동시에 추적
