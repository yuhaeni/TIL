# Kafka 재시도 설계 — backoff, head-of-line blocking, non-blocking 재발행, @RetryableTopic

> 날짜: 2026-07-19

## 내용

### backoff — 재시도 사이의 대기 시간

**backoff**는 Kafka 전용 용어가 아니라 **재시도(retry) 전반에서 쓰는 일반 용어**다. "실패했을 때 곧바로 다시 시도하지 말고, **잠깐 기다렸다가** 재시도하라"는 뜻이다.

- 문이 잠겼을 때 손잡이를 미친듯이 흔드는 대신 "3초 뒤 다시, 그래도 안 되면 6초, 12초..." 이렇게 **점점 텀을 늘려가며** 두드리는 것.
- 매번 대기 시간을 2배씩 늘리는 방식을 **exponential backoff(지수 백오프)**라 한다.

**왜 backoff가 필요한가?**
- 곧바로 재시도하면, 아직 장애가 풀리지 않은 외부 시스템(DB, API 등)을 계속 두들겨서 **부하만 키운다**.
- 잠깐 기다렸다 재시도하면 그 사이 장애가 복구될 여지를 준다.

### head-of-line blocking — sleep으로 backoff를 걸면 안 되는 이유

Kafka는 **하나의 파티션 안에서는 메시지 순서가 보장**되며, 컨슈머는 그 파티션을 **맨 앞부터 offset 순서대로 하나씩** 처리한다.

이때 backoff를 `Thread.sleep()`으로 구현하면 문제가 생긴다.

**은행 창구 한 줄** 비유:
- 컨슈머 스레드 하나가 담당하는 것은 파티션 하나(= 창구 한 줄).
- 맨 앞 손님을 처리하다가 직원이 `Thread.sleep(10초)`로 자버리면 → **그 줄 뒤에 서 있는 손님 전부가 10초 동안 정지**한다.
- 앞 메시지가 안 끝났으니 뒤 메시지도 처리될 수 없다.

이렇게 **줄 맨 앞(head-of-line) 하나가 막혀서 그 뒤 전체가 멈추는 현상**을 head-of-line blocking이라 한다.

**중요 — 헷갈렸던 포인트:** 정지되는 것은 *다른 파티션*이 아니라 **같은 파티션 안에서 뒤에 줄 선 메시지들**이다. 다른 파티션은 각자 다른 컨슈머 스레드가 담당하므로 영향받지 않는다.

```
[Thread.sleep 방식]
파티션-0: [msg1(sleep 10초...)] [msg2 대기] [msg3 대기] [msg4 대기]  ← 줄 전체 인질
```

> **면접 예상 질문:** Kafka 컨슈머에서 backoff를 Thread.sleep으로 구현하면 어떤 문제가 발생하는가? head-of-line blocking이란?

### non-blocking 재발행 — 딜레이 후 재시도 토픽으로 옮기기

head-of-line blocking을 피하려면, 실패한 메시지에서 스레드를 **붙잡지 말고 바로 풀어줘야** 한다.

**해결책:** 실패한 메시지를 `Thread.sleep`으로 붙들고 있는 대신 → **딜레이 후 별도의 재시도 토픽에 재발행(republish)**하고, 원래 파티션은 다음 메시지로 계속 진행한다.

```
[non-blocking 재발행 방식]
파티션-0: [msg1 실패 → 재시도 토픽으로 이동] [msg2 처리] [msg3 처리] ...  ← 줄은 계속 전진
retry 토픽: [msg1] ← 딜레이 후 나중에 재처리
```

- `Thread.sleep`(blocking) → 그 파티션 줄 전체가 대기 (뒤 메시지들 인질)
- 딜레이 후 재시도 토픽 재발행(non-blocking) → 문제 메시지만 옆으로 빼고 원래 줄은 계속 전진

**트레이드오프 — 순서 보장 포기:**
실패한 메시지를 뒤로 빼서 나중에 처리하므로, 원래 "들어온 순서대로 처리"라는 약속이 **깨진다**. 따라서 non-blocking 재시도는 **순서가 중요하지 않은 로직**에 적합하다. (환영 이메일 발송 등은 OK, 순서가 결과에 영향을 주는 도메인은 주의)

> **면접 예상 질문:** backoff를 Thread.sleep이 아니라 재시도 토픽 재발행으로 해야 하는 이유는? 이 방식이 포기하는 것은 무엇인가?

### @RetryableTopic vs tiered 토픽 — 자동 vs 수동

이 둘은 대립 관계가 아니라 **"어노테이션 자동 생성 vs 손수 구현"**의 관계다.

**tiered(계층형) 토픽 패턴** — 재시도 단계별로 토픽을 여러 개 두는 설계:
```
orders → orders-retry-0 → orders-retry-1 → ... → orders-dlt(Dead Letter Topic)
```
- 실패하면 다음 단계 토픽으로 넘기고, 각 단계마다 backoff 딜레이를 다르게 준다.
- 최종적으로 계속 실패하면 **DLT(죽은 편지함)**로 보내 격리한다.
- 이 구조를 **직접 구현**하는 것이 tiered 토픽 패턴.

**@RetryableTopic** — Spring Kafka가 위 tiered 구조를 **어노테이션 하나로 자동 생성**해주는 기능:
```java
@RetryableTopic(attempts = "3", backoff = @Backoff(delay = 1000, multiplier = 2.0))
@KafkaListener(topics = "orders")
public void consume(...) { ... }
```
- retry 토픽들과 DLT를 알아서 만들어준다.
- 시도 횟수·최초 실패 시각 같은 메타 정보를 헤더로 캐리하는 것도 지원.

**핵심:** 속(재시도 토픽 구조)은 똑같고, **직접 짜느냐 / 어노테이션에 맡기느냐**의 차이다. 그래서 설계 문서에서 "`@RetryableTopic`/tiered 토픽 중 구현 때 정한다"고 열어둔 것.

> **면접 예상 질문:** @RetryableTopic과 수동 tiered 토픽 구성의 차이는? DLT(Dead Letter Topic)는 어떤 역할인가?

### 파티션 · 컨슈머 · 컨슈머 그룹 — 담당(할당) vs 실제 동시 처리

재시도 설계를 이해하려면 파티션-컨슈머 매핑을 정확히 알아야 한다.

| Kafka 용어 | 마트 비유 | 정의 |
|---|---|---|
| 토픽(Topic) | 마트 전체 | 메시지가 쌓이는 통 |
| 파티션(Partition) | 계산대 | 토픽을 쪼갠 조각. 동시 처리의 단위 |
| 컨슈머(Consumer) | 계산원 | 메시지를 꺼내 처리하는 일꾼(스레드) |
| 컨슈머 그룹(Group) | 계산원 팀 | 컨슈머들을 묶은 팀. 팀이 토픽 하나를 나눠 처리 |

**매핑 규칙 — 한쪽 방향만 1:1 (비대칭):**
- ✅ 한 파티션 → 한 컨슈머 (파티션은 주인이 **반드시 하나**. 두 컨슈머가 같은 파티션을 못 읽음)
- ✅ 한 컨슈머 → 여러 파티션 가능 (일꾼은 여러 계산대 겸임 가능)
- ❌ 두 컨슈머 → 한 파티션 (절대 안 됨)

**헷갈렸던 핵심 포인트 — "담당(할당)"과 "실제 동시 처리"는 다르다:**
- 컨슈머 1명이 파티션 3개를 **담당(할당)**할 수는 있다. (소유권 3개)
- 하지만 스레드가 하나면 3개를 **진짜 동시에** 처리하는 게 아니라 **번갈아가며** 처리한다. (계산원 한 명이 계산대 3개를 왔다갔다)

**그래서 실질 병렬성:**
```
실질 병렬 처리량 = min(파티션 수, 컨슈머(스레드) 수)
```
- 파티션 5 + 컨슈머 5 → 각자 하나씩 → 진짜 5배 병렬 ✅
- 파티션 5 + 컨슈머 2 → 겸임 → 2배에 그침 (일꾼 부족)
- 파티션 2 + 컨슈머 5 → 3명은 idle → 2배가 한계

**파티션을 여러 개로 쪼개는 진짜 이유:** 순서 보장이 아니라, **여러 컨슈머가 동시에(병렬로) 처리할 수 있는 자리(상한)를 늘려 처리량을 확장**하기 위해서다. 단, 그 자리를 실제로 채우려면 컨슈머 스레드도 같이 늘려야 한다.

> **면접 예상 질문:** 컨슈머가 파티션을 "담당"하는 것과 "동시에 처리"하는 것의 차이는? 실질 병렬 처리량을 결정하는 공식은?

### 파티션 할당과 리밸런싱

컨슈머가 파티션을 배정받는 것은 랜덤이 아니라 **정해진 규칙**에 따른다.

- **파티션 할당(Partition Assignment)**: 컨슈머에게 파티션을 배정하는 것. 각 컨슈머가 받은 파티션 목록이 그 컨슈머의 **assignment**.
- **리밸런싱(Rebalancing)**: 컨슈머가 그룹에 들어오거나 나갈 때 할당을 **다시 나누는** 과정.
- **어사이너(Assignor)**: 어떻게 나눌지 정하는 전략. `RangeAssignor`, `RoundRobinAssignor`, `CooperativeStickyAssignor` 등.

**실무 팁 — 파티션 수 설정:**
- 기본은 "파티션 수 ≥ 컨슈머 수" (남는 컨슈머가 idle이 되지 않도록).
- 파티션 수는 **늘리기는 쉬워도 줄이기가 어렵다.** 트래픽이 커져 컨슈머를 더 붙이고 싶을 때 파티션이 부족하면 못 늘린다.
- 그래서 미래 확장을 고려해 **파티션을 다소 여유 있게** 잡는 게 흔한 전략. (단 너무 많으면 브로커 자원·리밸런싱 오버헤드가 커지므로 무한정 늘리진 않음)

> **면접 예상 질문:** 리밸런싱은 언제 발생하는가? 파티션 수를 컨슈머 수보다 여유 있게 잡는 이유는?

## 학습 정리

- **backoff** = 재시도 사이에 두는 대기 시간(Kafka 전용 아님). 곧바로 재시도하면 장애난 외부 시스템에 부하만 커짐 → 점점 텀을 늘리는 exponential backoff.
- backoff를 `Thread.sleep`으로 걸면 **head-of-line blocking** 발생 — 같은 파티션 뒤 메시지 전체가 정지(다른 파티션은 영향 없음).
- 해결책은 **non-blocking 재발행** — 실패 메시지를 딜레이 후 재시도 토픽으로 옮기고 원래 줄은 계속 전진. 대신 **순서 보장을 포기**하는 트레이드오프.
- **@RetryableTopic**은 tiered(계층형) 재시도 토픽 + DLT 구조를 어노테이션으로 자동 생성. tiered 토픽 수동 구현과 속은 같고 자동화 여부만 다름.
- 파티션-컨슈머는 **비대칭 관계** — 파티션은 주인이 반드시 하나, 컨슈머는 여러 파티션 겸임 가능.
- **"담당"과 "동시 처리"는 다르다** — 실질 병렬 처리량 = `min(파티션 수, 컨슈머 수)`. 파티션을 쪼개는 진짜 이유는 순서 보장이 아니라 **병렬 처리량 확장**.
