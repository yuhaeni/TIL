# PostgreSQL vs MySQL InnoDB — MVCC, UNDO LOG, WAL, VACUUM

> 날짜: 2026-06-10

## 내용

> "PostgreSQL에는 REDO LOG, UNDO LOG가 없다"는 말을 자주 듣는데, 정확히는 **UNDO LOG는 없고(MVCC로 대체), REDO에 해당하는 WAL은 있다**가 맞는 표현이다. 두 DB는 모두 MVCC와 WAL을 쓰지만, **옛날 행 버전을 어디에 저장하느냐**가 다르다.

### REDO LOG / UNDO LOG — InnoDB가 변경을 처리하는 두 로그

InnoDB는 트랜잭션이 데이터를 바꿀 때 두 종류의 로그를 남긴다.

- **REDO LOG**: 변경사항을 실제 데이터 파일에 반영하기 **전에 먼저 기록**한다. DB가 갑자기 꺼졌다 켜졌을 때, 아직 디스크에 반영 안 된 변경을 이 로그를 보고 **복구(redo)** 한다.
- **UNDO LOG**: 쿼리 실행 **전의 옛날 데이터를 백업**한다. `ROLLBACK` 시 원래 값으로 되돌리거나, 다른 트랜잭션이 "옛날 버전"을 읽어야 할 때(MVCC) 사용한다.

InnoDB는 `UPDATE` 시 **원본 자리에 새 값을 덮어쓴다(in-place update)**. 원본이 사라지므로, 옛날 버전은 UNDO LOG라는 **별도 창고**에 따로 보관해야 한다.

> **면접 예상 질문:** REDO LOG와 UNDO LOG는 각각 어떤 장애/상황에 대비하는 로그인가요? 둘의 역할 차이를 설명해주세요.

---

### MVCC — 옛날 버전을 어디 두느냐가 두 DB의 핵심 차이

MVCC(Multi-Version Concurrency Control)는 **여러 버전의 행을 유지해 동시성을 제어**하는 기법으로, InnoDB와 PostgreSQL **둘 다 사용**한다. "MVCC를 쓴다"는 건 공통점이고, 차이는 **옛날 버전을 어디에 저장하느냐**다.

- **InnoDB**: 원본을 덮어쓰고, 옛날 버전은 **UNDO LOG(별도 창고)** 에 보관 → 그래서 UNDO LOG가 필요
- **PostgreSQL**: 원본을 덮어쓰지 않고, 수정할 때마다 **새 버전의 행(tuple)을 테이블 안에 추가**해서 옛날 버전과 공존시킴 → 그래서 **UNDO LOG가 없음**

PostgreSQL은 옛날 버전이 테이블 안에 그대로 살아있으므로 별도 창고가 필요 없다.

> **면접 예상 질문:** PostgreSQL과 MySQL InnoDB 모두 MVCC를 쓰는데, 옛날 버전 데이터를 저장하는 방식은 어떻게 다른가요?

---

### dead tuple & VACUUM — PostgreSQL의 청소부 (JVM의 GC 같은 존재)

PostgreSQL은 수정할 때마다 옛날 버전을 테이블에 남기므로, 아무도 더 이상 보지 않는 **죽은 행(dead tuple)** 이 계속 쌓인다. 이를 청소하는 것이 **VACUUM**(자동으로 도는 **autovacuum**)이다.

- JVM의 GC가 참조 없는 객체를 청소하듯, VACUUM은 **어떤 트랜잭션도 더 이상 볼 필요 없는 죽은 행**의 공간을 재사용 가능하게 만든다.
- **주의**: 일반 VACUUM은 공간을 OS에 돌려주지 않고 **"재사용 가능 표시"만** 한다.

> **면접 예상 질문:** PostgreSQL에서 dead tuple은 왜 생기나요? autovacuum이 이를 어떻게 처리하며, 일반 VACUUM과 VACUUM FULL의 차이는 무엇인가요?

---

### Table Bloat — 청소가 안 되면 왜 조회가 느려지는가

VACUUM이 `UPDATE`/`DELETE` 속도를 못 따라잡으면 죽은 행이 계속 쌓여 테이블이 실제 데이터보다 훨씬 커진다. 이를 **bloat(부풀어 오름)** 라 한다. (예: 살아있는 데이터 1GB인데 테이블이 10GB → 9GB만큼 bloat)

**bloat가 조회를 느리게 만드는 이유** (면접용 구체화):

PostgreSQL은 데이터를 **페이지(블록, 8KB) 단위**로 읽는다.

- 원래 살아있는 행 100개가 페이지 **1개**에 꽉 차 있었다고 하자.
- bloat로 한 페이지에 살아있는 행이 10개씩만 남으면, 같은 100개를 읽는 데 페이지 **10개**를 읽어야 한다.
- 즉 디스크 **I/O가 10배**로 늘어난다. **디스크는 메모리보다 수천 배 느리므로, I/O 횟수가 곧 응답 시간**을 좌우한다.
- 게다가 메모리 캐시(**shared_buffers**)에도 죽은 행이 든 페이지까지 올라가서 **캐시 적중률(hit ratio)** 까지 떨어진다.

**관련 용어**
- **bloat**: 죽은 행이 안 치워져 테이블이 살아있는 데이터보다 커진 상태
- **shared_buffers**: PostgreSQL이 디스크 페이지를 올려두는 메모리 캐시 영역

> **면접 예상 질문:** 테이블 bloat가 발생하면 조회 성능이 왜 떨어지나요? 페이지/I/O 관점에서 구체적으로 설명해주세요.

### VACUUM FULL — 디스크 공간까지 회수하는 대청소 (강한 락 트레이드오프)

이미 bloat된 테이블의 **디스크 공간을 OS에 진짜로 반환**하려면 `VACUUM FULL`이 필요하다.

- 테이블을 **통째로 새 파일에 다시 써서** 죽은 행을 완전히 제거하고 공간을 OS에 반환한다. (10GB → 1GB)
- **트레이드오프**: 작업 중 해당 테이블에 **`ACCESS EXCLUSIVE LOCK`(가장 강한 락)** 을 걸어 **읽기/쓰기 전부 차단**된다. 운영 중 함부로 돌리면 장애가 된다.
- 실무에선 무중단 처리를 위해 `pg_repack` 같은 확장을 쓰기도 한다.

> **면접 예상 질문:** VACUUM FULL은 일반 VACUUM과 무엇이 다르며, 운영 환경에서 사용할 때 어떤 점을 주의해야 하나요?

### WAL — PostgreSQL의 "REDO LOG"는 이름이 WAL일 뿐 (기법은 공통)

PostgreSQL에 REDO LOG라는 **이름**은 없지만, **WAL(Write-Ahead Log)** 이 똑같은 역할을 한다.

- **WAL은 PostgreSQL 전용이 아니라 "기법(개념)" 의 이름**이다. "변경을 데이터에 반영하기 전에 로그에 먼저 쓴다"는 원칙 자체를 WAL이라 부른다.
- 따라서 **InnoDB의 REDO LOG도 WAL의 한 구현체**다. MySQL도 당연히 이 원리를 쓴다.

| | WAL 기법을 구현한 것 |
|---|---|
| MySQL InnoDB | **REDO LOG** |
| PostgreSQL | 아예 이름을 **WAL** 이라 부름 |

**왜 두 번 쓰나? (WAL 한 번, 나중에 테이블 한 번)**
- 변경을 테이블 곳곳(흩어진 페이지)에 바로 쓰면 **랜덤 I/O**라 느리다.
- WAL은 로그 파일 **맨 끝에 차곡차곡 이어 쓰기(순차 I/O)** 라 빠르다.
- 그래서 "빠른 WAL에 먼저 안전하게 적고 커밋 완료 → 실제 테이블 반영은 나중에" 전략을 쓴다. 장애 시 WAL을 보고 복구한다.

> **면접 예상 질문:** WAL은 왜 변경을 테이블에 바로 쓰지 않고 로그에 먼저 기록하나요? MySQL의 REDO LOG와 PostgreSQL의 WAL은 같은 개념인가요?

### ROLLBACK 속도 vs 테이블 청소 부담 — 저장 방식의 트레이드오프

옛날 버전 저장 방식의 차이는 `ROLLBACK` 성능과 평소 테이블 상태의 트레이드오프로 이어진다.

- **InnoDB**: 원본을 이미 덮어썼으므로, `ROLLBACK` 시 UNDO 창고에서 옛날 값을 꺼내 **원본 자리에 다시 복사**해야 한다 → **롤백이 느림**. 대신 평소 테이블은 **깔끔**(별도 VACUUM 불필요).
- **PostgreSQL**: 옛날 버전이 테이블에 그대로 살아있으므로, `ROLLBACK` 시 새 버전만 **"무효"로 표시**하면 끝 → **롤백이 빠름**. 대신 `UPDATE`/`DELETE`마다 **죽은 행이 무조건 쌓여** VACUUM 청소 부담이 남는다.

```
[공통점]
- 둘 다 MVCC 사용 (동시성 제어)
- 둘 다 WAL(=REDO) 사용 (장애 복구 + 순차 I/O 성능)

[차이점] — 옛날 버전을 어디에 두느냐
┌──────────────┬─────────────────────┬──────────────────────┐
│              │ InnoDB              │ PostgreSQL           │
├──────────────┼─────────────────────┼──────────────────────┤
│ 옛날 버전 저장 │ UNDO LOG (별도 창고)  │ 테이블 안에 새 버전 쌓음 │
│ UNDO LOG     │ 있음                 │ 없음                  │
│ REDO 역할    │ REDO LOG            │ WAL                  │
│ ROLLBACK     │ 느림 (복사 필요)      │ 빠름 (무효 표시만)     │
│ 평소 테이블   │ 깔끔                 │ bloat 위험 → VACUUM 필요│
└──────────────┴─────────────────────┴──────────────────────┘
```

> **면접 예상 질문:** PostgreSQL과 InnoDB 중 ROLLBACK이 더 빠른 쪽은 어디이며 왜 그런가요? 그 대가로 어떤 트레이드오프가 생기나요?

## 학습 정리

- "PostgreSQL엔 REDO/UNDO가 없다"는 정확히는 **UNDO만 없고(MVCC로 대체), REDO 역할은 WAL이 한다**는 뜻이다.
- 두 DB 모두 **MVCC + WAL을 쓴다.** 진짜 차이는 **옛날 행 버전을 어디 두느냐**: InnoDB는 UNDO LOG(별도 창고), PostgreSQL은 테이블 안에 새 버전을 쌓는다.
- PostgreSQL은 옛날 버전이 테이블에 쌓이므로 **dead tuple → bloat** 문제가 생기고, **VACUUM(autovacuum)** 청소부가 필요하다. (JVM의 GC와 유사)
- bloat가 조회를 느리게 하는 이유: 한 페이지에 살아있는 행이 줄어 같은 데이터를 읽는 데 **페이지 수 = 디스크 I/O가 배수로 증가**하고, shared_buffers 캐시 효율도 떨어진다.
- 트레이드오프: PostgreSQL은 **ROLLBACK이 빠른 대신** 테이블 청소(VACUUM) 부담, InnoDB는 **ROLLBACK이 느린 대신** 평소 테이블이 깔끔. 공간 회수는 `VACUUM FULL`(강한 락 주의).
