# PostgreSQL Advisory Lock — check-then-act race, xact vs session, hashtext 충돌, Redis 분산 락 비교

> 날짜: 2026-07-16

## 내용

어떤 리소스 단위로 여러 테이블을 `DELETE → INSERT` 로 통째 교체할 때, 같은 리소스에 대한 동시 처리가 데이터를 깨뜨리는 것을 막아야 한다. 이때 `pg_advisory_xact_lock` 으로 리소스 단위 직렬화를 거는 방식과, 그 선택을 뒷받침하는 DB 락 개념들을 정리한다.

```java
public void lock(String key) {
    jdbcTemplate.queryForObject("SELECT pg_advisory_xact_lock(hashtext(?))", Object.class, key);
}

// 하나의 트랜잭션 안에서
lock(key);                                     // 리소스 단위 직렬화
LocalDateTime stored = findLatestCollectedAt(key);
if (stored != null && incoming != null && incoming.isBefore(stored)) {
    return; // 묵은 payload 로 최신 데이터 덮어쓰기 방지
}
removeData(key);                               // DELETE
addData(key, ...);                             // INSERT
```

### check-then-act race — 확인과 행동 사이의 틈

**check-then-act**: 먼저 확인(check)하고 → 그 확인 결과를 믿고 행동(act)하는 패턴. 여기선 "저장된 게 나보다 최신인가?" 확인하고 → 아니면 갈아엎는다.

문제는 **확인과 행동 사이에 시간 틈**이 있다는 것. 냉장고 열어 "우유 없네!" 확인하고 사러 나간 사이, 룸메이트도 똑같이 확인하고 사러 나가는 상황과 같다.

같은 키로 요청 A, B 가 동시에 들어오면:

1. A 가 `findLatestCollectedAt` 로 "저장된 게 오래됐네, 갈아엎어도 돼!" 판단
2. **A 가 아직 쓰기 전에** B 가 최신 데이터를 `INSERT`
3. A 는 과거의 판단("갈아엎어도 됨")을 그대로 믿고 `remove → add` 실행 → **B 가 넣은 최신 데이터를 지우고 묵은 데이터로 덮어씀**

→ 확인한 순간의 진실이 행동하는 순간엔 이미 거짓이 됐다. 이게 race condition의 핵심. 락으로 "읽고-판단-쓰기"를 하나의 트랜잭션으로 직렬화해 이 틈을 없앤다.

> **면접 예상 질문:** check-then-act race가 발생하는 원리를 설명하고, 락 없이 두 요청이 인터리빙될 때 데이터가 어떻게 깨지는지 예시로 설명해보세요.

### 행 락(row lock) vs advisory 락 — 무엇에 거느냐

DB 락은 **"무엇에 거느냐"** 로 나눌 수 있다.

| | 대상 | 예시 |
|---|---|---|
| **행 락 (row lock)** | 실제 존재하는 특정 데이터 행 | `SELECT ... FOR UPDATE` (`WHERE id=5` 그 행) |
| **advisory 락 (권고 락)** | 내가 정한 임의의 정수 키 (DB는 의미를 모름) | `pg_advisory_xact_lock(42)` |

- 행 락 = "5번 사물함을 잠근다"
- advisory 락 = "특정 이름이 적힌 팻말을 손에 쥔다" (팻말은 사물함이 아니라 그냥 이름표, 즉 **약속/convention**)

**advisory 락을 쓰는 이유**: 로직이 기존 행을 통째로 `DELETE` 후 `INSERT` 한다. 락을 거는 순간 "지켜야 할 그 행"이 사라져 버리고, 새로 넣을 행은 아직 존재하지도 않는다. 없는 행엔 자물쇠를 못 채운다. 그래서 **"행"이 아니라 "리소스라는 개념"에 락**을 걸어야 하고, 그게 임의의 키에 거는 advisory 락이다.

`SELECT ... FOR UPDATE` 의 `FOR UPDATE` 는 UPDATE문 실행 중이 아니라 **"나 이 행 이따 수정할 거니까(FOR UPDATE) 조회하는 지금부터 미리 락을 잡아둔다"** 는 의도 선언. 행 락에는 강도(세기) 옵션도 있다:

- `FOR UPDATE` / `FOR NO KEY UPDATE` — 배타(exclusive) 계열. "이 행 바꾸거나 지울 거야"
- `FOR SHARE` / `FOR KEY SHARE` — 공유(share) 계열. "읽는 동안 남이 못 바꾸게만" (주로 FK 무결성용)
- `FOR DELETE` 는 없다 — DB 입장에선 UPDATE든 DELETE든 "그 행을 배타 점유"라는 점이 같아서 `FOR UPDATE` 로 커버된다.

> **면접 예상 질문:** DELETE 후 INSERT로 데이터를 통째로 교체하는 로직에서 행 락(`SELECT ... FOR UPDATE`)을 쓰기 곤란한 이유는 무엇인가요? advisory 락이 더 적합한 이유와 함께 설명해보세요.

### xact 락 vs 세션 락 — 락 수명과 커넥션 풀 누수

advisory 락에는 형제가 둘 있다.

| 함수 | 수명 | 해제 시점 |
|---|---|---|
| `pg_advisory_xact_lock` | **트랜잭션** | 커밋/롤백 시 **자동 해제** |
| `pg_advisory_lock` | **세션(커넥션)** | `pg_advisory_unlock` 직접 호출 or 커넥션 종료 시까지 |

세션 락(`pg_advisory_lock`)은 트랜잭션이 끝나도 안 풀린다. `unlock` 한 줄을 깜빡하면 그 키는 영영 잠긴 채 남는다.

**커넥션 풀에서 특히 위험한 이유**: 요즘 백엔드는 HikariCP 등 커넥션 풀에서 커넥션을 빌려 쓰고 반납한다. 세션 락을 잡은 채 unlock을 깜빡한 커넥션이 풀에 **반납**되면, 그 커넥션은 **여전히 잠긴 채** 반납된다. 다음 요청은 **자기가 걸지도 않은 유령 락을 쥔 커넥션**을 물려받는다.

- 이 유령 락이 안 풀린 채 굴러다니면 → 같은 키를 잠그려는 다른 요청이 무한 대기
- 여러 커넥션으로 번지면 → 풀의 커넥션이 죄다 "대기" 상태로 **고갈(pool exhaustion)** → DB도 앱도 안 죽었는데 **서비스 먹통**

그래서 `xact` 버전이 안전한 선택. 커밋이든 롤백이든 트랜잭션만 끝나면 DB가 알아서 풀어주니 깜빡할 여지가 없다.

> **면접 예상 질문:** `pg_advisory_lock`(세션 락)을 커넥션 풀 환경에서 사용하다 unlock을 누락하면 어떤 장애로 이어지나요? `pg_advisory_xact_lock`이 이 문제를 어떻게 원천 차단하는지 설명해보세요.

### hashtext 충돌 — 과잉 잠금은 안전, 과소 잠금은 위험

advisory 락은 키로 **정수**만 받는다. 그래서 문자열 키를 `hashtext()` 로 정수(int4, 32비트)로 변환해서 넣는다. 무한한 문자열을 유한한 정수에 매핑하니 **해시 충돌(collision)** — 서로 다른 두 키가 같은 정수를 받는 일 — 이 발생할 수 있다.

**이 충돌은 정합성 버그가 아니라 성능(throughput) 손해다.** 핵심 원리:

> **과잉 잠금(over-locking)은 안전하고, 과소 잠금(under-locking)은 위험하다.**

| 상황 | 락 방향 | 결과 |
|---|---|---|
| 다른 키인데 같은 번호 (해시 충돌) | 과잉 잠금 | 괜히 기다림 → **성능 손해**, 데이터는 안전 (false contention) |
| 같은 키인데 다른 번호 | 과소 잠금 | race 발생 → **정합성 버그** |

- 충돌한 두 키는 서로 다른 데이터를 다루는데도 같은 락 번호를 두고 다툰다(순서대로 처리). 하지만 기다린 뒤엔 각자 자기 데이터를 정확히 처리 → **데이터는 안 깨진다.** 이게 **false contention(가짜 경합)**.
- 정합성이 깨지려면 "같은 키인데 다른 번호"(과소 잠금)여야 하는데, **해시 함수는 결정론적(같은 입력 → 항상 같은 출력)** 이라 같은 키는 무조건 같은 번호를 받는다. 즉 **과소 잠금은 해시 때문에는 절대 안 생긴다.** 생길 수 있는 건 안전한 방향의 과잉 잠금뿐.

**실무 임팩트는 미미**: 32비트(약 43억 값)라 충돌 확률 자체가 낮고, 충돌해도 두 키가 정확히 같은 순간에 처리돼야만 실제 대기가 생긴다. "낮은 확률 × 낮은 확률"이라 감수할 만한 트레이드오프. (정 신경 쓰이면 64비트 bigint 키로 충돌 확률을 더 낮출 수 있다.)

**과잉 잠금의 스펙트럼**: 과잉이 극단으로 가면(세션 락 누수처럼 안 풀리는 락) 데이터는 멀쩡한데 서비스가 멈춘다. 즉 "과잉은 안전"은 **정합성 관점에서만** 참이고, 극단적 과잉은 **가용성(availability) 사고**가 된다.

- 과소 잠금 → 데이터가 깨진다 (**정합성** 사고)
- 적당한 과잉 → 데이터 안전, 살짝 느림 (해시 충돌 — 감수)
- 극단적 과잉 (안 풀리는 락) → 데이터 안전, 근데 **서비스 먹통** (**가용성** 사고)

> **면접 예상 질문:** advisory 락 키를 `hashtext`로 만들 때 해시 충돌이 나면 정합성 버그인가요 성능 문제인가요? "과잉 잠금은 안전, 과소 잠금은 위험" 원리로 설명하고, 왜 해시로는 과소 잠금이 생기지 않는지 말해보세요.

### DB advisory 락 vs Redis 분산 락 — 자원의 위치가 갈림길

같은 동시성 제어인데 Redis 분산 락이 아니라 DB advisory 락을 고를 수 있다.

**Redis 분산 락 동작**: Redis에는 `Lock` 전용 자료형이 없다. 분산 락은 평범한 명령어를 조합한 **기법/패턴**이고, Redisson이 이를 `lock()`/`unlock()` API로 감싼 것.

```
# 획득: key가 없을 때만(NX) 만들고, 30초 뒤 자동 만료(PX)
SET lock:<리소스키>  <고유토큰>  NX  PX 30000

# 해제: 저장된 값이 내 토큰과 같을 때만 DEL (Lua 스크립트로 원자 실행)
```

- `NX` = Not eXists → 동시에 쏴도 딱 하나만 성공 = 상호 배제(mutual exclusion)
- `PX 30000` = 30초 TTL → 락 잡은 서버가 죽어도 영영 안 잠기게 (`EX`는 초, `PX`는 밀리초 단위)
- **고유토큰(UUID 등)** 확인 후 해제 → 내 락이 TTL로 이미 풀리고 남이 새 락을 잡았는데 내가 뒤늦게 DEL 해서 **남의 락을 지우는 사고** 방지
- **Watchdog** = 작업이 TTL보다 길어지면 백그라운드로 TTL을 계속 연장

**비교표**:

| | DB advisory 락 | Redis 분산 락 |
|---|---|---|
| 락 수명 | **트랜잭션에 묶임** — 커밋/롤백 시 자동 해제 | **TTL(PX)로 시간 추측** — 짧으면 조기 만료, 길면 장애 시 오래 잠김 |
| 롤백 시 | 락도 자동으로 같이 풀림 (원자적) | Redis는 롤백을 **모름** → 수동 관리 |
| 락 vs 데이터 위치 | **같은 집(PostgreSQL)** | **다른 집** → 둘의 생사 불일치 위험 |
| 추가 인프라 | 없음 | Redis 클러스터 별도 운영 |
| 부담 | DB 부하·커넥션 물고 있음 | DB 부하를 Redis로 분산 |

**TTL 조기 만료 시나리오**: DB 작업이 TTL(30초)보다 오래 걸리면, 아직 처리 중인데 `lock:...` key가 먼저 만료돼 사라진다. 그 순간 다른 요청이 같은 리소스에 `SET ... NX` 로 락 획득에 성공 → **두 요청이 동시에 락을 쥔 상태** → 상호 배제가 깨져 막으려던 race가 그대로 터진다.

**핵심 판단 기준 — "지켜야 할 자원이 어디 있느냐"**:

- 자원이 **단일 DB 한 곳**에 있고 이미 **트랜잭션 안**에서 돈다 → DB advisory 락이 깔끔. 앱 서버가 여러 대여도 **모두 같은 PostgreSQL 하나에 붙으므로** DB가 공용 심판이 되어 문제없다. (그래서 "다중 서버"만으로는 Redis를 써야 할 이유가 안 된다!)
- 자원이 **DB에 없거나**(외부 API 호출 제한 등), 서비스마다 **서로 다른 DB**를 써서 **공통으로 바라보는 심판 DB가 없다** → 모두가 바라보는 중립 지대 = Redis 같은 외부 분산 락이 필요.

> **면접 예상 질문:** 단일 DB 트랜잭션 안에서 도는 로직에 Redis 분산 락 대신 DB advisory 락을 쓰는 이유 2~3가지를 말해보세요. 반대로 Redis 분산 락이 반드시 필요한 상황은 언제인가요?

## 학습 정리

- **check-then-act race**: 확인과 행동 사이의 틈에서 다른 요청이 끼어들어 데이터가 깨진다. 락으로 "읽기-판단-쓰기"를 하나의 트랜잭션으로 직렬화해 막는다.
- **DELETE→INSERT 로 통째 교체하는 로직은 행 락을 못 쓴다** — 잠글 행이 사라졌다 새로 생기므로, "리소스라는 개념"에 거는 advisory 락을 쓴다.
- **`xact` 락 vs 세션 락**: `xact`는 트랜잭션 종료 시 자동 해제. 세션 락(`pg_advisory_lock`)은 unlock 누락 시 커넥션 풀 누수 → pool exhaustion → 서비스 먹통.
- **hashtext 충돌 = 과잉 잠금** → false contention(성능 손해)이지 정합성 버그가 아니다. 해시는 결정론적이라 과소 잠금(정합성 사고)은 안 생긴다. 단, 극단적 과잉(안 풀리는 락)은 가용성 사고.
- **DB advisory 락 vs Redis 분산 락의 갈림길은 "서버 수"가 아니라 "자원의 위치"** — 단일 DB 안이면 트랜잭션에 묶이는 advisory 락, 여러 DB·서비스에 걸치면 공통 심판인 Redis.
